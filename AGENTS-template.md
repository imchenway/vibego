## Claude Code 风格表达规则

默认采用接近 Claude Code 的工程同事式表达：清楚、克制、先结论、少噪音，让用户能快速判断“发生了什么、为什么、下一步做什么”。

- 先给结论，再给原因、修法和验证方式；不要先铺排大段过程。
- Bug/故障类回复固定收敛为“现象 -> 影响 -> 根因 -> 修法 -> 验证”，除非用户要求完整排查日志。
- 少用术语、少贴过程日志、少堆文件清单；必要术语必须用人话解释其影响。
- 面向用户决策输出：需要用户确认时，明确推荐项、优缺点、风险和回滚方式。
- 不要为了显得严谨而输出空泛免责声明；只保留与当前决策或验证直接相关的信息。
- 不要宣称“已完成/已修复/已验证”，除非已经执行对应验证命令并看到成功结果；验证失败或未跑必须直说。
- 长回复必须结构化，但每段尽量短；Telegram/移动端场景优先使用清单、短段落和明确标题。

## HTML 图形沟通默认协议

默认协作原则：非琐碎任务默认优先使用 `vibe-diagram` skill，用单文件 HTML 图承载核心沟通；AGENTS 负责判断何时触发；`vibe-diagram` 负责具体制图约束、自动路由、视觉质量、交互和附件交付。

必须触发的场景：

- 功能迭代或开发设计：必须先用 HTML 图展示现状、目标方案、前后差异、影响面、测试矩阵、风险与回滚，再进入实现或让用户确认。
- 缺陷排查：必须用 HTML 图展示现象、影响、证据链、可疑节点、已确认根因、高亮根因节点、修法、验证与回滚；未验证根因只能标注为假设。
- 系统/功能理解：必须用 HTML 图展示组件、依赖、中间件、DB、MQ/队列语义、关键调用链或业务规则。
- 需求评审、方案选择、设计定稿：必须用 HTML 图展示问题、目标、选项、推荐方案、AC、待决策项和风险。
- 最终交付：如果本轮产出或修改了 HTML 图，必须作为 `.html` 文件附件发送；禁止只发 Markdown 链接或路径。

可跳过的场景：

- 用户明确要求不要画图、只要一句话、或任务是琐碎格式转换时，可不生成 HTML。
- 无法写文件或平台不支持附件时，必须说明限制，并输出完整 HTML 代码块作为 fallback。

边界：AGENTS 不重复定义具体画图样式；所有制图样式、路由、可访问性、弹窗、移动端和质量红线以 `vibe-diagram` skill 为准。

## HTML 图交付后的文本压缩规则

  或 docs。

- 若同时完成代码修复，聊天只列受影响文件、验证结果和剩余待执行动作，避免把 docs/HTML 内容复制到聊天。
- 不要通过大幅删除 AGENTS 约束来降噪；应保持“AGENTS 定触发与交付边界、skill 定图形细则、HTML/docs 承载细节、聊天只做短确认”的分层。

## HTML-only 交付信封模式

当用户明确要求“禁止文本对话”“全部内容都用 HTML 文件沟通”“只发 HTML 附件/文件”等 HTML-only 口径时，文本回复只允许作为交付信封，所有实质内容必须进入项目内单文件 HTML。

生效后必须遵守：

- 所有分析、设计、排障、方案、决策、验收、总结、代码逻辑说明、证据链、风险、回滚、测试矩阵，都必须写入 HTML 文件。
- 文本回复只允许作为交付信封：Codex 默认只输出可点击 `file://` 链接与绝对路径兜底；Telegram 来源只输出项目内 `.html/.htm` 路径以触发附件发送。
- 不得在聊天文本里展开分析、方案、证据链、测试矩阵或验收总结；这些内容即使很短，也应进入 HTML。
- 只有阻塞性澄清问题可以临时使用文本，且最多 1 个问题，不得顺带展开分析。
- 若当前环境无法写入 HTML 文件，才允许输出完整 HTML 代码块作为 fallback，并明确说明无法写文件。

边界：HTML-only 不是“模型完全不输出文本”，而是把文本通道降级为路径/附件通知；实质沟通内容必须以 HTML 文件为准。

# 前端页面设计约束
前端实现禁止把需求清单直接翻译成页面模块。
所有页面必须按用户操作逻辑递进设计，而不是把所有功能一次性展示出来。

## 前端质量技能自动调用矩阵

所有前端页面、组件、布局、样式开发，必须先使用 `frontend-skill` 与 `impeccable`：

- 编码前必须输出：visual thesis、content plan、interaction thesis；
- 禁止默认卡片堆叠、后台面板式平铺、模板化 SaaS 首页；
- 页面必须先确定唯一核心目标与用户主路径，再写组件。

当用户提到“漂亮、高级、现代、专业、重设计、颠覆、沉浸式、官网、落地页、像真实产品、记忆点、Awwwards 风格”等目标时，必须额外
使用 `premium-frontend-ui`：

- 先确定一个明确视觉方向，而不是混合多个风格；
- 必须强化 typography、空间、层级、首屏视觉锚点；
- 高级效果不得牺牲可读性、性能和移动端体验。

当任务涉及滚动叙事、视差、hero 入场、hover 微交互、页面转场、Framer Motion、GSAP、ScrollTrigger、动画体验时，必须使用
`gsap-framer-scroll-animation`：

- 优先使用项目已有动画方案或 CSS；
- 如需新增 Framer Motion、GSAP、Lenis、Three/WebGL 等依赖，必须先说明收益、成本、性能风险并征得确认；
- 所有连续动效必须支持 `prefers-reduced-motion` 降级。

所有前端实现必须使用 `accessibility` 做基础可访问性约束：

- 交互元素必须支持键盘访问与清晰 focus 状态；
- 文本对比度、点击区域、表单错误提示、图标按钮 label、弹窗 focus trap 必须覆盖；
- 不允许只靠颜色表达状态；
- 动效必须有 reduced-motion 降级。

前端交付前，如能启动页面，必须使用 `visual-qa` 做截图级自检：

- 检查 spacing、alignment、typography、color、responsive、overflow、空状态、错误态、加载态；
- 发现页面像“功能清单 / 卡片集合 / 后台面板”时，必须回到信息架构重构，而不是局部美化；
- 如果无法截图或无法运行页面，必须明确说明未覆盖原因。

## 核心原则
- 一个页面只做一件核心事情
- 高频操作放浅层
- 低频功能收纳到二级页面、弹窗、抽屉、Tab 或更多菜单
- 列表页只负责选择对象，详情页才展示完整信息
- 设置、配置、高级选项默认收纳
- 首页不是功能大全，只保留主路径和核心入口

## 明确禁止
- 禁止卡片堆叠
- 禁止把所有功能垂直平铺在一个页面
- 禁止每个页面都加大标题和说明文案
- 禁止出现“管理你的 xxx”“在这里查看 xxx”这类废话
- 禁止把设置项、历史记录、作品详情、会员充值等全部塞进首页
- 禁止为了覆盖需求牺牲页面主次关系

## 标题规则
如果左侧菜单、顶部导航、Tab 或面包屑已经说明当前页面是什么，内容区不要再重复显示同名大标题。
页面应直接进入核心内容。
只有业务对象名、关键状态页、落地页、空状态页，才允许使用页面级标题。

## 实现前必须先判断
编码前必须回答：
1. 这个页面的唯一核心目标是什么？
2. 用户当前最需要做什么？
3. 哪些内容必须首层展示？
4. 哪些内容应该收纳到下一层？
5. 是否存在卡片堆叠、标题废话、功能平铺？
   如果发现页面像功能清单、后台面板、卡片集合，必须先重构信息架构，再写代码。

## Impeccable 自动调用规则
用户不需要手动指定 `impeccable` 的子命令,根据任务类型自动选择对应命令/工作流,必要时只向你确认设计方向。
当任务涉及以下任一内容时，agent 必须自动启用 `impeccable` skill：
- 新前端项目从 0-1 创建
- 前端页面、组件、布局、样式开发
- 官网、落地页、管理后台、数据看板、移动端 H5
- UI 美化、视觉重设计、交互优化、动效设计
- 表单、表格、弹窗、抽屉、空状态、错误态、加载态
- Figma 还原、设计系统、组件库、主题 token
- 用户提到“漂亮、高级、现代、专业、重设计、颠覆、交互好、像真实产品”

### 自动选择规则
agent 必须根据任务意图自行选择 impeccable 工作流：
- 从 0-1 新建前端项目：使用 `impeccable shape` 先确定产品视觉与交互方向，确认后进入 `impeccable craft`。
- 已有项目整体重设计：使用 `impeccable shape`，若用户要求颠覆性改版，则进入 departure mode，把当前 UI 作为 anti-reference。
- 已有页面变漂亮：使用 `impeccable polish`。
- UI 太普通、缺少记忆点：使用 `impeccable bolder`。
- UI 太花、太吵、信息过载：使用 `impeccable quieter` 或 `distill`。
- 布局、间距、视觉层级混乱：使用 `impeccable layout`。
- 文案、按钮、错误提示、标签不清楚：使用 `impeccable clarify`。
- 空状态、错误态、加载态、边界场景不足：使用 `impeccable harden`。
- 移动端、平板、宽屏适配：使用 `impeccable adapt`。
- 需要高级动效、页面转场、惊艳交互：先使用 `impeccable overdrive` 提供 2-3 个方向，说明性能、兼容性和复杂度风险，待用户确认后再实现。
- 只做检查不改代码：使用 `impeccable critique` 或 `impeccable audit`。

### 执行约束
- 如果用户只说“做得漂亮一点 / 高级一点 / 重新设计”，agent 应自动判断应使用哪个 impeccable 工作流。
- 如果任务范围不清，agent 应先用自然语言确认目标。
- 任何前端实现不得只满足“功能能用”，还必须满足产品级视觉、交互状态、响应式和可访问性要求。
- 如涉及新增依赖、改构建、改 CI，必须先单独说明风险并请求用户确认。

## 验收标准

合格页面应该让用户按操作路径逐步完成任务，而不是把所有功能摆出来。
宁可少展示，也不要平铺；宁可多一层递进，也不要单页堆完。
