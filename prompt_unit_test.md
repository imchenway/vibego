# 单元测试补齐到 100% 提示词

你将以【严格证据模式（Ultra Strict Evidence Mode）】执行任务：在不降低工程标准的前提下，为当前项目补齐/新增单元测试，使覆盖率达到
100%，并且单元测试覆盖的场景必须尽可能多而全（覆盖分支/异常/边界/状态/交互/不变量），同时保证测试稳定、可重复、可审计。

========================

0) 规约与安全边界（必须先执行）
   ========================
   0.1 先读取并严格遵守（按优先级）：

- $HOME/.config/vibego/AGENTS.md（如存在）
- 当前目录及子目录下所有 AGENTS.md（如存在；冲突以离目标文件最近的为准）
- 若冲突无法裁决 => 失败关闭：停止并输出冲突点与需人工裁决的问题

0.2 若存在以下文件，必须优先读取并以其为事实源：

- ./AGENTS.evidence.json（如存在）
- 或离目标文件最近目录下的 AGENTS.evidence.json（如存在）
  后续“命令/事实/护栏”优先从 evidence.json 读取；缺失则只能以仓库证据补齐，禁止凭经验推断。

0.3 写入范围与禁止项：

- 允许修改：测试代码（tests 目录、src/test、__tests__、*.spec.*, *.test.* 等）
- 允许新增：测试辅助代码（test utils/fixtures/mocks），前提是位于测试范围内
- 允许极少量生产代码改动：仅当为“可测性”而必须，并且必须遵循：
  (a) 先写 characterization tests 锁定现有行为
  (b) 变更必须是行为保持的最小重构（如把时间/随机数/外部依赖抽象为可注入接口）
  (c) 在 PLAN 中明确说明必要性与风险，并等待明确允许后才可实施
- 禁止：改动业务语义、改公共 API 合约、改生产默认配置、改覆盖率门槛、改 CI 流水线逻辑
- 禁止：新增或扩大覆盖率 exclusions（除非仓库已有明确证据且只是“遵循现有排除”，不得为了达标新增排除）
- 禁止：为了测试而引入大依赖/新测试框架/高侵入 mock 工具（除非仓库已有并有证据）
- 禁止：git commit/push

0.4 严格证据规则（违反即失败）：

- “如何跑测试/如何算覆盖率/门禁阈值/排除项/命令”必须来自仓库证据（CI/脚本/README/配置文件）。
- 找不到证据 => 只能写 TODO，禁止“通常应该是…/大概率…/我猜…”
- 所有命令必须标注状态：
  ✅ Verified（本次实际执行成功）
  ⚠️ Unverified（有证据但未执行，说明原因）
  🔴 Failed（尝试执行失败，记录失败摘要，不修代码）
  ❌ Unknown（无证据，写 TODO 需要补的证据来源）

========================

1) 目标定义（必须可验证）
   ========================
   1.1 覆盖率目标：

- 100%（以项目现有覆盖率工具输出为准）
- 优先同时达成：line + branch = 100%
- 若工具仅提供 line：line=100% 仍需尽力做到 branch 级别等价覆盖（用测试策略覆盖所有分支与异常路径）

1.2 覆盖范围：

- 默认覆盖全部生产代码（现有 exclusions 必须遵守；不得新增/扩大 exclusions）
- 若仓库存在“生成代码/第三方/自动生成 DTO”等排除项：必须引用证据并严格遵循；不得自行扩展

1.3 “场景尽可能多而全”的具体标准（强制）：
对每个被测单元（类/函数/模块/组件/服务）至少覆盖以下维度中“适用的所有项”：
A. 正常路径：典型输入、典型状态、典型依赖响应
B. 边界值：最小/最大、空集合/空串、单元素/大集合、极端数值、溢出/精度（按语言与业务适用）
C. 反向与校验：非法输入、缺参、越界、类型不符、格式错误（应抛错/返回错误码/返回空等）
D. 异常分支：依赖抛异常、超时、重试失败、解析失败、IO 失败、序列化失败（如适用）
E. 状态迁移：对象/状态机的每个合法迁移 + 非法迁移（如适用）
F. 幂等与重复：重复调用、重复提交、重复事件（如适用）
G. 交互断言：对依赖的调用次数/参数/顺序（只在必要时；避免过度 mock）
H. 不变量/性质：输出满足不变量（例如排序/去重/范围限制/单调性等）
I. 并发/时序（如适用）：多线程安全、竞态窗口（用可控注入/虚拟时钟/同步原语让测试确定性）
J. 安全/权限（如适用）：鉴权缺失、权限不足、越权访问路径（在单元范围内用 mock 验证）

“适用”判定必须基于仓库证据与代码结构，不得凭空增加业务假设。

========================

2) 建立现状基线（Baseline）
   ========================
   2.1 侦查（只读）并确定：

- 语言/框架/测试框架/覆盖率工具
- 官方测试与覆盖率命令（以 AGENTS.evidence.json > CI/脚本 > README > 构建配置 为准）
- 是否存在 coverage 阈值、branch 要求、排除项
  以上每条都必须记录 evidence（path+anchor，可选 snippet）。

2.2 运行覆盖率（优先最小副作用）：

- 只运行“有证据的”测试+覆盖率命令
- 记录命令与结果状态（✅/🔴）与关键输出摘要
- 若无法运行（缺依赖/权限/环境）：失败关闭，输出 TODO（需要的证据或环境条件）

2.3 产出“覆盖率缺口清单”（必须可定位）：

- 按文件/类/函数列出：未覆盖行/未覆盖分支/未覆盖异常路径
- 为每个缺口标注类型：分支/异常/边界/状态/交互/不可达/平台差异
- 若报告工具能输出具体行号/分支信息，必须引用；否则用代码定位（rg/grep + 文件路径）

========================

3) PLAN（补齐策略与测试矩阵）
   ========================
   3.1 为每个缺口建立测试矩阵（Test Matrix）：

- 被测单元：文件/类/函数（路径）
- 场景集合：按 1.3 的 A~J 列出适用项
- 断言点：至少 1 个“有意义断言”（行为/返回/状态/交互/不变量）
- 依赖策略：真实对象/轻量 stub/mock（遵循仓库现有工具与惯例）
- 预计覆盖：覆盖哪些分支/异常/边界
  每条矩阵项必须能对应到代码中的具体分支/异常点（引用路径/锚点）。

3.2 约束与风险：

- 不允许为覆盖率写“无意义断言”或“只跑到行”
- 不允许把集成测试当单元测试来凑覆盖率
- 不允许引入新依赖来“方便测试”，除非仓库已有且有证据
- 如遇不可达分支/死代码/平台特性：必须列为 TODO 并给出证据，禁止擅自改 exclusions 或删代码

3.3 执行顺序（增量、可回滚）：

- 按模块/目录分批推进：每批补完就跑测试+覆盖率
- 每批都要更新“缺口清单”直到清零

========================

4) IMPLEMENT（实现补齐：增量推进）
   ========================
   4.1 测试实现规范（强制）：

- 每个测试必须包含至少 1 个有意义断言
- 尽量使用参数化测试覆盖分支组合（若框架支持且仓库已采用）
- 避免测试实现细节（private/内部结构），优先测试对外行为
- Mock 约束：
    - 禁止 mock 被测对象本身
    - 对时间/随机数/UUID：优先可控注入/固定种子；若仓库已有虚拟时钟/clock 注入模式，必须遵循（证据）
    - 对网络/DB/MQ：单元测试默认 mock；若仓库已有 testcontainers 且明确将其用于单元层（少见），必须有证据，否则归类为
      integration，不计入“单元补齐”策略

4.2 若必须做“可测性重构”（默认不做）：

- 先写 characterization tests 锁定行为
- 再做最小重构（注入依赖/抽象接口），不得改变对外语义
- 任何生产代码变更必须在 DELIVER 中明确列出并解释“为何必要、如何保证行为不变”

========================

5) SELF-TEST（验收：稳定性 + 100%）
   ========================
   5.1 运行完整测试与覆盖率命令（必须来自证据）：

- 输出结果摘要
- 覆盖率必须 100%（按工具输出）
- 若工具支持 branch，branch 也必须 100%

5.2 稳定性要求（强制）：

- 覆盖率达标后，必须连续运行两次完整测试+覆盖率命令，结果一致才算通过
- 若出现抖动/不稳定：必须定位原因（时间/随机/并发/顺序依赖），通过可控注入/固定种子/隔离状态解决；禁止“重跑碰运气”

5.3 若仍无法达到 100%：

- 严格失败关闭：
    - 列出阻塞原因与证据（不可达分支、平台差异、生成代码等）
    - 给出最小解决路径选项：
      A) 继续补测试（首选）
      B) 仅遵循仓库现有 exclusions（必须有证据）
      C) 如需新增 exclusions 或调整门禁 => 必须停止并请求人工批准（给出证据、影响范围、最小排除建议）

========================

6) DELIVER（交付：可审计）
   ========================
   必须输出：

- 新增/修改的测试文件列表（按模块分组）
- 若有生产代码改动：列出文件与变更理由（证明行为保持）
- 执行过的命令清单与状态（✅/⚠️/🔴/❌），包含关键输出摘要
- 覆盖率最终结果关键输出（摘取关键行）
- 测试矩阵覆盖情况总结（哪些场景已覆盖：A~J）
- 测试矩阵必须以表格输出，并为每个场景标注：覆盖的分支/异常点（文件路径+锚点）
- TODO/已知问题（若有，每条写缺少的证据来源或需人工裁决项）

现在开始执行。
