## 交付验收图规则

交付验收图必须默认回答“原始要求是否逐条满足、证据是否足够、还剩什么动作”。每一条用户需求或 AC 都必须拥有独立 R# 泳道；没有证据只能标 `待确认/warn/blocked`，不能并入总通过。验证命令、测试结果、截图核对和打包门禁必须贴在对应 R# 泳道内；全量门禁只能兜底，不能替代逐条证据。风险、回滚和待执行动作必须从对应 R# 引出。交付验收主画布不得再用矩形卡片节点承载每一步。

## 交付验收必答信息

交付验收图必须在首屏或紧邻主轨道的固定信息区直接外显，不得只在最终聊天收尾字段里说明：改了什么、影响哪些功能入口、用户需要执行什么脚本、需要重启什么服务、如何验证、剩余未覆盖点。如果无需脚本或无需重启，也必须明确写“无需执行脚本”或“无需重启服务”。交付说明栏写 `改了什么`、`影响功能入口`、`验证方式`、`待用户执行脚本 / 重启服务`、`未覆盖点`。

交付说明栏不能做成 5 张等权重卡片；必须收敛成一条横向或纵向连通的信息带。旧图问题只能作为旁注或反例标记。主图第一视觉必须是当前面板的关系结构。改了什么、影响入口、如何验证、脚本 / 重启、未覆盖点默认纵向排列；除非用户明确要求横向总控线，否则不要把这五项横向铺满首屏。

## 交付验收图专用骨架

交付验收图必须长成需求到证据的验收轨道。推荐主图语法：`验收账本：原始要求 → 交付变更 → 验证证据 → 验收判定 → 待执行 / 回滚`。证据泳道连回 R#；风险动作板按 `通过 / 待观察 / 需要用户动作 / 回滚` 分列；交付时间线按反馈、实现、验证、同步、验收组织。

## 反直觉版式红线

- 不得把“验证闭环”“代码影响点”“保留与回滚”拆成彼此等权重的底部卡片区。
- 如果读者需要在需求卡、代码卡、验证卡之间来回跳读才能判断某条需求是否通过，必须重画。
- 交付验收样例优先使用克制的产品式验收账本或路线牌，避免重装饰 SVG 图纸感。
- pass / warn / fail / blocked 必须同时使用文字标签和形状或图标；不能只靠颜色。
