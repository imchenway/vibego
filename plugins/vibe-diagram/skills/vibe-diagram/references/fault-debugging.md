## 故障排查图专用骨架

故障排查图必须先画当前现状链路，再画根因和修法。用户需要先理解“当前代码 / 系统 / 功能设计怎样运行并走到问题点”，再判断证据、根因和修复动作是否可信。

固定主骨架：

`现象 / 影响 → 当前现状链路 → 证据裁决 → 根因或最高可疑点 → 修法 → 验证 / 回滚`

骨架执行规则：

1. **现象 / 影响**：写用户可见现象、影响范围、触发条件和当前验证状态；不要只写错误结论。
2. **当前现状链路**：现状链路必须展示当前代码、接口、状态、配置或运行路径如何一步步走到问题点；这是故障排查图的主画布，不是旁注。
3. **证据裁决**：证据必须贴在现状链路节点上，用 E1/E2/E3 标明来源、观察结果和指向判断；假设裁决只能作为证据到根因的旁路，不得把假设列表画成主图。
4. **根因或最高可疑点**：根因节点必须落在现状链路上；证据不足时只能标最高可疑点 H#，不能写成已确认根因 R#。
5. **修法**：修法节点必须明确切断、替换、补偿或兜底哪一段故障链路；修法不能漂在底部方案区。
6. **验证 / 回滚闭环**：验证 / 回滚闭环必须贴近修法节点，直接标出验证命令、结果状态、回滚动作和剩余风险。

按故障性质选择形态：

- 小故障用单图因果链：现象、当前链路、证据、根因、修法、验证顺着一条主线推进。
- 链路故障用代码时序排障图：参与者列 + 时间自上而下，证据贴在调用、返回、异常或兜底行上。
- 影响两处以上用 before/after 流程化对照：左/上画当前故障逻辑，右/下画修复后逻辑，中轴标保留、删除、新增、风险和回滚。

## 故障排查图规则

故障排查图默认回答“用户怎样最快看懂故障来龙去脉”，不是把根因分析文字搬进 HTML。

- 主结构固定为：故障故事线 + 证据阶梯 + 假设裁决 + 行动闭环。
- 故障故事线只保留 5 到 7 个节点：现象、影响、触发点、关键证据、根因或最高可疑点、修法、验证状态；节点只写短句和状态标签。
- 故障故事线必须画成一条明确因果路径，根因或最高可疑点是视觉焦点；证据、假设、行动必须锚定到故障故事线节点，禁止各自另起一排等权重卡片。
- 故障故事线不是“竖向故事线 + 圆角卡片列表”：主路径必须依靠因果线、流程符号、阶段轨道、泳道或时序轴表达关系；一列同形卡片即使带左侧竖线、步骤图标和箭头标签，也必须判为卡片堆叠。
- 证据阶梯按 E1/E2/E3 编号展示：来源、观察结果、指向的判断必须优先写在对应故障故事线节点内，跨节点汇总才进入证据阶梯；日志、截图、JSONL、SQL、命令输出等原始长内容才允许进入点击详情。
- 假设裁决必须分为“成立 / 排除 / 待验证”三类；每个假设必须绑定证据编号或写“待补证据”。
- 根因节点必须同时显示证据编号、置信度、验证状态；未验证只能标为最高可疑点，不能写成已确认根因。
- 主图不写长段根因分析；根因推理用“因为 E1+E2，所以 R1 成立”这类一跳因果短句写在根因节点内，完整原始材料才放可见证据表或弹窗补充。
- 行动闭环必须展示：已做修法、验证命令/结果、回滚动作、剩余风险；验证未执行或失败时，不得写“已修复”。
- 视觉优先级：故障故事线 > 根因或最高可疑点 > 验证状态 > 证据编号 > 详细日志；避免把证据表、假设树和修法清单画成同等重量的卡片堆叠。

## 根因、修法与前后对照规则

根因和修复方案不是说明文字，必须成为主图中的视觉焦点。

1. 缺陷排查和开发设计默认使用单图高亮根因与修法：根因节点标 `根因 R#`，计划修法节点标 `修法 F#`，验证节点标 `验证 V#`
   ；不能只靠颜色表达。
2. 证据必须贴近或连到根因节点，测试与回滚必须贴近或连到修法节点，避免证据区、方案区、风险区各自漂浮。
3. 当修复改变两处以上关键节点，或改变调用链、数据流、状态流转、接口契约、权限边界、用户主路径之一时，必须改用前后对照图。
4. 前后对照图必须明确：左/上为当前逻辑或故障逻辑，右/下为修复后逻辑或目标逻辑，中间或旁路标出保留、删除、新增、风险和回滚点。
5. 若前后差异很小，禁止为了形式感画双图；用单图高亮根因与修法更清晰。
6. 若前后双图在桌面宽度内仍会拥挤，必须改为上下两图或纵向阶段对照，不得缩小字体或继续横向加卡片。
7. before 固定在左侧或上方，after 固定在右侧或下方；这是前后对照图的阅读约定，不能随节点编号或美观需要调换。
8. 禁止用左右交替排布表达前后差异；左右交替只能用于非对照型时间线，且不得与 before / after 语义混用。
9. 纵向卷轴中也必须保持左侧为当前/故障逻辑，右侧为修复后/目标逻辑；移动端折叠时保持上方为 before、下方为 after。
10. 如果节点不属于前后对照，只能放在中轴、旁注或详情中，避免把中性说明误读为 before 或 after。

## 流程图语法门禁

前后对照只是容器，不是图形语法本身。功能迭代、开发设计和故障修复必须优先画流程图或流程化对照图。

1. 主画布必须包含开始/结束事件、活动节点、决策菱形、带标签箭头；缺少这些流程图基本元素时，不得把结果称为流程图。
2. before/after 每一侧内部也必须是流程图：当前逻辑要展示旧流程如何走到问题点，修复后逻辑要展示新流程如何绕过或消除问题点。
3. 禁止把 before/after 列画成普通说明卡片列表；每一列至少要有 3 个按方向连接的流程节点，且必须有一个明确输入和一个明确输出。
4. 根因节点和修法节点必须落在流程路径上，不能浮在旁边当说明卡片；根因节点用 `根因 R#`，修法节点用 `修法 F#`。
5. 辅助证据优先写入流程节点内部，或作为流程节点的证据锚点 / 紧邻旁注；原始长材料可放点击详情，但不得把证据列表、假设列表、方案列表画成与流程同等重量的主内容。
6. 对开发迭代图，必须画出用户/系统当前主流程、变更点、目标主流程、验证闭环；对故障修复图，必须画出排查流程、判断分支、根因定位、修复动作、验证/回滚闭环。
7. 若流程跨角色或跨端，使用泳道或阶段轨道；若流程跨状态，使用状态流转；若流程跨接口/数据，使用时序或数据流。不要用卡片网格替代这些语法。
