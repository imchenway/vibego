## 技术设计与需求决策图规则

技术设计图默认回答“如何落地实现、有哪些契约、如何验证与回滚”；需求 / 决策沟通图默认回答“为什么做、怎么选、怎样验收”。

- 技术设计图必须覆盖：受影响目录、接口/API 契约、数据库/字段/索引、模块职责、关键流程、测试矩阵、风险与回滚。
- 需求 / 决策沟通图必须覆盖：用户场景、业务目标、核心规则、方案对比、推荐项、验收标准、未决问题。
- 方案对比只保留 2 到 4 个选项；每个选项写优点、缺点、适用边界和回滚成本。
- 推荐方案要视觉突出，但必须保留反例和风险；不要把推荐方案画成唯一事实。
- 开发设计定稿要标出 QPS/TPS、权限、幂等、重试、日志、指标、告警和灰度发布。
- 如果用户要看页面长相，应切页面设计稿；如果用户要看运行组件，应切系统架构图。

## 标题与生图类型规则

顶部标题必须以触发的生图类型开头。标题格式：`生图类型：主题结论`。生图类型取自自动路由结果或用户明确点名的图型，而不是取
skill 名称；不要把 skill 名称写成页面主标题。

常用标题前缀：

- `系统架构：`、`业务架构：`、`业务流程：`、`代码时序：`
- `状态/数据模型：`、`故障排查：`、`页面设计稿：`
- `技术设计：`、`需求决策：`、`交付验收：`

示例：故障排查图应写 `故障排查：Zeus 每次启动弹出钥匙串密码`；技术设计图应写 `技术设计：社会商品名称校验规则收敛`。不要写
`vibe-diagram：...` 或 `vibe-diagram 不是换皮卡片...`。

## 输出前自检

交付前必须检查并修正：

- 是否只生成一个单文件 HTML？
- 是否直接发送 HTML 文件或完整 HTML 代码块？
- 是否没有外部依赖？
- 是否已按自动路由规则选择唯一主图型？
- 顶部标题是否以本次触发的生图类型开头，而不是以 skill 名称开头？
- 所选图型的必选元素是否齐全？
- 是否只有一张主图？
- 是否没有常驻“如何阅读此图”侧栏？
- 是否节点文字不溢出、无背景线穿字、无横向溢出？
- 节点是否承载足够信息，而不是只有两行标题再把关键证据/风险/回滚拆到底部卡片？
- 关键细节是否直接呈现在主图或紧邻主图的可见区域，而不是藏在点击详情里？
- 若提供点击弹窗，弹窗是否只做补充且空白/Esc 可关闭，不会自动滚动页面？
- 若提供点击弹窗，弹窗正文是否没有显示字面量 `\n`、`\t`、未解码 HTML 实体或 JSON 转义痕迹？
- 系统架构图是否明确展示组件、中间件/运行时、DB、MQ/队列语义？
- Codex 默认场景是否提供可点击 `file://` HTML 链接和绝对路径兜底？
- Telegram 来源场景是否能看到 `.html` 文件卡片，而不是只有 Markdown 链接或路径文字？

最终回复要短，只说明 HTML 已生成/已发送；不要输出冗长过程日志。
