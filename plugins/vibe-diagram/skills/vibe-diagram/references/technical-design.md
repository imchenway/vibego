# vibe-diagram reference: technical-design

> 本文件由 TASK_20260630_022 拆分自 `vibe-diagram/SKILL.md`。只有路由命中对应图型后才读取；读取失败必须 fail-closed。


## 候选全集清单

- 生图类型：技术设计图
- 首选候选：模块 / 契约 / 数据 / 发布回滚拓扑
- 必生成备选候选：
  - API 契约泳道
  - 数据流 + 一致性边界
  - 发布切换轨道
- 校准期要求：当前图型命中后，HTML 内必须按候选 A/B/C/D…纵向展开首选候选与全部备选候选；每个候选都必须是真图。信息不足时也必须生成该备选候选，用“待确认节点”标明缺口。

## 技术设计与需求决策图规则

技术设计图默认回答“如何落地实现、有哪些契约、如何验证与回滚”；需求 / 决策沟通图默认回答“为什么做、怎么选、怎样验收”。

- 技术设计图必须覆盖：受影响目录、接口/API 契约、数据库/字段/索引、模块职责、关键流程、测试矩阵、风险与回滚。
- 需求 / 决策沟通图必须覆盖：用户场景、业务目标、核心规则、方案对比、推荐项、验收标准、未决问题。
- 方案对比只保留 2 到 4 个选项；每个选项写优点、缺点、适用边界和回滚成本。
- 推荐方案要视觉突出，但必须保留反例和风险；不要把推荐方案画成唯一事实。
- 开发设计定稿要标出 QPS/TPS、权限、幂等、重试、日志、指标、告警和灰度发布。
- 如果用户要看页面长相，应切页面设计稿；如果用户要看运行组件，应切系统架构图。

## 各图型形态与布局契约

每个图型必须先确定“它应该长成什么样”，再写 HTML。图型名称不是标题装饰；如果主画布没有对应形态，就不能用该生图类型命名。

- **系统架构图必须长成北向南分层拓扑**：外部入口在上，控制面、工作面、运行与中间件、状态与证据面逐层向下；每层使用 2-5
  个同层节点，层间短箭头表达主流向，跨层长依赖改成旁注或编号引用。
- **业务架构图 / 领域地图必须长成能力层 + 对象关系 + 规则约束**
  ：参与方在上或左，能力层承接业务目标，领域对象用拥有、产生、约束、审批、结算等关系线连接，规则矩阵贴近对象而不是单独堆文字。
- **业务流程图必须长成 BPMN-light 流程**：开始/结束事件、活动节点、决策菱形、带标签箭头齐全；主路径 5-9
  步，异常分支放下方或旁路并短回流，不能画成说明表格。
- **代码时序图必须长成参与者列 + 时间自上而下**
  ：参与者列固定在顶部，生命线向下延伸；每一步调用、返回、抛出、异步回调都占独立消息行，消息箭头横向或微斜连接参与者生命线，源码锚点作为行内证据或旁注。
- **状态 / 数据模型图必须长成状态机、ER-lite 或生命周期**：状态节点只承载状态名、进入条件、退出条件，动作写在线上；实体节点展示关键字段、约束、索引和逻辑关系，不把字段清单塞成普通卡片。
- **故障排查图必须长成因果链、流程化对照或排障时序图**：先按问题性质选择“因果链 / 流程化 before-after /
  代码时序”之一；根因、证据、修法和验证必须落在主路径上，不能把证据、假设、方案拆成等权重文字区。
- **页面设计稿必须长成页面线框 / artboard**：每个候选或单稿都要像真实页面，有视口边界、首屏层级、主操作、收纳区、状态变体和响应式标记；不能把需求点翻译成卡片清单。
- **技术设计图必须长成模块 / 契约 / 数据 / 发布回滚的落地设计图**：模块边界、API
  契约、数据结构、并发一致性、观测点、测试和回滚必须通过拓扑、泳道或矩阵绑定到同一实现主线。
- **需求 / 决策沟通图必须长成决策树或方案矩阵与主路径绑定**：问题、目标、选项、推荐、验收和风险必须围绕一个决策主轴展开；方案矩阵只做辅助，不得替代决策树或主路径。

如果某类图无法按上述形态画出主谓宾关系，必须换图型；不要保留错误图型再靠标题、图例或长说明补救。

## 布局、箭头与防重叠算法门禁

生成 HTML 前必须像排版算法一样先规划，再落 DOM。禁止一边写节点一边补箭头。

1. **画布先分配主轴和泳道，再放节点，最后连线**。宽度用于承载泳道、参与者列、before/after
   或局部对照；高度用于承载时间、阶段、因果递进和证据展开。复杂内容宁可纵向滚动，也不要横向挤压成缩略图。
2. **节点先排版后连线**。先确定层级、泳道、节点尺寸、节点间距和旁注位置；同层节点必须等高或按内容自适应后统一留白，节点间距不得小于
   16px，主路径阶段间距不得小于 28px。
3. **箭头只能连接节点边缘锚点**。北向南主线使用下边缘到上边缘锚点；左右对照使用右边缘到左边缘锚点；状态机优先连接状态边缘中点；代码时序图消息箭头必须连接参与者生命线中心或消息端点。
4. **箭头标签不压正文**。箭头标签放在线段中点或折线拐点旁，并使用轻背景遮罩；禁止箭头穿过节点正文、标题、图例、交互按钮或弹窗触发区。
5. **连线层必须低于节点层**。SVG / canvas / pseudo-element 线条放在节点下层，节点正文、按钮和标签在上层；若线必须跨越复杂区域，改用编号步骤、阶段连接或局部小图，不得让线条穿字。
6. **节点正文必须使用 HTML/CSS 可换行容器**。必须通过 max-width、min-height、height:auto、line-height、overflow-wrap:anywhere
   和合理
   padding 保证文本可读；不得用固定高度裁切文字，不得用 `white-space: nowrap` 承载正文，禁止用 line-clamp、max-height 或
   overflow:hidden 裁掉节点正文。
7. **长节点优先自动增高或拆成子节点**。节点正文可以超过 3 行，关键信息先留在节点里；只有完整日志、长 SQL、长
   JSON、源码片段、跨节点汇总矩阵才放相邻证据表或弹窗补充，不能把单个节点的关键判断外移到底部卡片。
8. **必须用桌面宽度和 390px 宽度分别检查**。桌面检查箭头锚点、重叠、主路径焦点和页面宽度利用；390px
   检查是否降级为可读纵向流、节点文字不溢出、无横向依赖。
9. **如果任一节点重叠、线穿字、文字溢出，必须重排**。优先处理顺序：缩短文案 → 增大节点 / 调整 min-height → 改为旁注 →
   拆阶段 / 改泳道 → 换更合适图型；禁止用缩小字号、截图缩放或隐藏溢出蒙混过关。

## 图形语法硬约束

卡片堆积不是图。卡片不是全局禁用，但必须限用。所有图型都必须显式表达关系，而不是把 Markdown 大纲换成一组圆角卡片。

1. 生成 HTML 前，必须先选定一种视觉语法：分层拓扑、阶段轨道、泳道流程、时序轴、状态机、ER-lite、因果链、决策树、矩阵热区或页面线框之一。
2. 允许使用卡片承载摘要、节点、泳道单元或分组边界；卡片必须服务箭头、泳道、坐标、层级、时序或状态转换，不能通过堆卡片来冒充一图胜千言。
3. 禁止把卡片作为唯一图形语法；如果整张图的主要信息只能靠阅读卡片文字理解，而不是靠空间、连线、方向、阶段或状态理解，必须重画。
4. 主画布必须占据首屏主要面积，读者第一眼看到的是“关系结构”，不是标题下面的卡片列表。
5. 辅助信息不得与主图同等视觉重量；证据、风险、测试、回滚、方案说明优先写入对应主路径节点内部或紧贴节点的子行 /
   角标；底部证据/矩阵只承载跨多个节点的汇总或原始长材料索引，弹窗只可补充原始长材料。
6. 禁止把“主链路卡片 + 证据卡片 + 假设卡片 + 方案卡片 + 表格”并列铺满画面；这属于卡片堆叠，不属于一图胜千言。
7. 如果去掉箭头、坐标轴、泳道、分层、包含关系或状态转换线后，剩下的仍然是一组同等权重文字卡片，必须重画。
8. 每种图都必须有一个视觉主谓宾：谁作用于谁、什么导致什么、数据从哪到哪、状态如何转移、哪个选项支配哪个结果；无法表达时应换图型，而不是继续加卡片。
9. 故障排查图主路径不得由一列同形圆角卡片承担；左侧竖线、步骤图标、箭头标签只能作为辅助连接，不能把“竖向故事线 +
   圆角卡片列表”包装成流程图。
10. 如果隐藏节点正文后只剩一列卡片和弱连接线，或读者必须逐张读卡片才能理解因果，必须重画为流程图、因果链、泳道、时序轴或状态转换图。

## 关键细节外显规则

点击弹窗可以保留，但只能做渐进增强，不能把图变成“先点卡片再读详情”的信息迷宫。

1. 节点优先承载关键信息；关键细节必须直接呈现在对应节点内部，至少也要紧邻节点可见，包括：结论、根因、修法、验证状态、方案取舍、验收标准、规则口径、接口/DB
   契约、测试矩阵、风险回滚和待确认项。
2. 点击详情只能用于补充、放大或复制主图已可见的信息，例如原始日志片段、完整命令输出、长 SQL、长 JSON、截图说明、源码摘录或证据原文。
3. 证据结论必须静态可见，但原始证据默认进入对应节点的点击详情；节点内静态展示证据编号、结论、可信度或状态即可，避免把文件路径、行号和日志片段挤进主图。
4. 不得把验收标准、规则口径、接口/DB 契约、测试矩阵、风险回滚、根因证据结论或方案优缺点仅放入弹窗或底部独立卡片；这些内容必须至少以摘要、编号、矩阵或旁注形式静态可见，且优先写在对应节点内。
   换言之，不得把验收标准、规则口径、接口/DB 契约、测试矩阵、风险回滚、根因证据或方案优缺点仅放入弹窗；可进入弹窗的是已经在节点可见的原始证据展开。
5. 弹窗不得承载唯一信息源；关闭所有弹窗后，读者仍必须能从页面静态内容理解主结论、主路径、关键规则、风险与回滚。
6. 如果关闭 JavaScript 或不点击任何节点仍读不懂主结论，必须重画；不要用“可点击详情”掩盖主图信息不足。
7. 不要为了保持两行节点而把信息拆到底部证据卡片；节点文字可以分行、分段、列出输入/输出/证据/风险，节点高度无须统一压短。

## 证据详情与底部证据区规则

证据要支撑读图，而不是抢走主图。默认策略是“节点内摘要，点击看原文”，不要把所有证据平铺到底部。

1. **原始证据默认进入对应节点的点击详情**：点击详情可以承载文件路径、行号、日志片段、SQL、JSON、命令输出和截图说明；节点内只保留
   E#、证据结论、可信度、状态和少量关键口径。
2. **节点内静态展示证据编号、结论、可信度或状态即可**：例如 `E2：handler 缺失，置信高`、`V1：测试未跑`、`待确认：RDS 运行状态`
   ；不要把长路径和多行日志塞进主画布。
3. **不要默认在底部铺完整证据卡片**：底部证据区只用于跨节点冲突裁决、全局证据索引或测试矩阵；如果证据只服务单个节点，必须归入该节点详情。
4. **底部证据区必须轻量**：只保留 E# → 节点名 → 结论 → 详情入口的索引；不能把底部做成第二张同等重量的证据卡片墙。
5. **截图也要分层**：截图里的用户可见现象和判断写入节点；截图原图、裁剪说明和坐标标注进入点击详情。

## 节点信息承载规则

用户读图的最低成本路径应该是“沿主路径读节点”，而不是“读两行节点，再跳到底部证据卡片，再回到节点”。因此：

1. **节点优先承载关键信息**：每个主路径节点至少写清动作/状态、关键判断、输入输出或证据结论之一；故障、迭代、技术设计图还要在相关节点内写清证据、风险、测试、回滚默认写入对应主路径节点。
2. **不要为了保持两行节点而把信息拆到底部证据卡片**：两行节点只适合高层概览；如果用户需要理解来龙去脉，节点必须承载足够上下文。
3. **节点可以限制宽度，但高度必须随内容自动增长**：宽度用 `max-width` 控制可读行长，正文用自动换行；高度使用 `height:auto`
   或仅设置 `min-height`，让节点随内容增高。
4. **优先增高节点和自动换行，而不是把关键细节挪到图外底部卡片**：只有原始日志、完整
   SQL/JSON、长源码片段、跨多个节点的汇总矩阵，才进入节点点击详情、底部轻量索引或弹窗。
5. **底部证据/矩阵只承载跨多个节点的汇总或原始长材料索引**：底部区域不能替代节点正文，不能成为理解单个节点的必读入口；单节点证据优先进入该节点点击详情。
6. **禁止用 line-clamp、max-height 或 overflow:hidden 裁掉节点正文**；如果节点过长，应拆成相邻子节点、阶段组或泳道段，而不是隐藏文字。

## 累计用户约束不可回退

视觉风格可以调整，但不得覆盖制图硬规则。柔和卡片风格只是一种视觉外观，不是图形语法豁免；回到某种样式只代表外观回退，不代表前序制图要求回退。

前序用户约束必须同时满足：

1. 一图胜千言，而不是卡片胜千言；卡片只能作为节点、摘要、泳道单元或分组边界，不能替代关系结构。
2. HTML 宽度不友好、长度可承载推理；默认正常页面宽度、北向南主线、局部横向关系，复杂内容用纵向滚动展开。
3. 完全从左到右只适合很短流程；长流程必须改为北向南、阶段轨道、泳道或左上到右下的时序图。
4. 根因、修法、验证闭环必须是主路径焦点；不能把根因和计划修法藏在说明区或底部选项区。
5. 大改动使用 before / after 对照，小改动使用单图高亮；before 一律在左或上，after 一律在右或下。
6. 故障排查、开发迭代必须具备流程图语法：开始/结束事件、活动节点、决策菱形、带标签箭头、验证和回滚闭环。
7. 背景系统必须统一；浅色背景可以有高级白色系明度层次、低对比工程网格、点阵或坐标纸肌理，但不得局部割裂或抢主线。
8. 视觉可以柔和、圆角、浅蓝白，但主图仍必须靠箭头、泳道、坐标、层级、时序或状态转换来读懂。

## 视觉质量门禁

禁止交付原始工程草图感的 SVG。图可以克制，但不能粗糙；可以信息密集，但不能靠硬边框和说明文字堆出来。

1. 视觉质量必须服务流程阅读：第一眼能看出主路径、关键节点、分支、根因、修法和验证闭环。
2. 使用统一的线宽、字号、留白、层级和图例；同一张图内不得混用多套边框、色块、字体尺度和箭头样式。
3. 禁止粗暴边框、重阴影、满屏说明文字和低级默认样式；宁可减少节点，也不要用廉价装饰弥补结构不清。
4. 流程节点必须像图形符号而不是 UI 容器：事件、活动、决策、状态、实体、泳道和连接点要各有明确形态。
5. 主图应保留足够留白，文字短句化但信息不能空心化；关键解释优先进入节点正文或节点内分行，节点可以变高，弹窗只补充原始材料，不能替代可见细节。
6. 颜色只用于状态和路径强调，不用于装饰；根因、风险、修法、验证必须同时有文字标签，不能只靠颜色识别。
7. 背景只能使用固定白底工程网格、轴线或轻分层遮罩，不得抢主路径，也不得用大面积彩色容器制造“图已经完成”的错觉。
8. 浅色背景默认以白色为主色，但白色背景不能是扁平纯白；使用瓷白、珍珠白、雪白、雾白等白色系明度层次；允许使用极轻白底工程网格、点阵或坐标纸肌理，但网格必须低对比、低存在感；背景纹理必须全局统一，HTML
   body、SVG 主画布和弹窗遮罩以外的页面区域应共享同一背景系统，不要只在主画布局部铺网格或局部底纹；避免蓝色底、灰色底、米黄纸张感背景；背景只能提供质感和空间层次，不得抢主线。

## HTML 画布方向规则

HTML 的宽度不友好，但长度可以承载推理。宽度服务阅读，长度服务推理。

1. 默认采用正常页面宽度 + 北向南主线 + 局部横向关系；不要通过卡片限制宽度，也不要按横向海报思路把所有节点塞进一屏。
2. 禁止为了塞满首屏而横向卡片化；首屏只放结论、图例和主路径起点，后续用纵向滚动展开阶段、证据、分支、矩阵、风险和回滚。
3. 复杂图默认优先北向南：北向南分层架构、纵向故障故事线、纵向状态生命周期、纵向阶段轨道或纵向时间轴。
4. 需要表达调用先后时，或采用左上角向右下角的时序图或斜向流程：时间从左上推进到右下，避免纯水平拉长。
5. 完全从左到右只适合很短的流程图；超过 5 个主节点不得继续横向铺开，必须改为北向南、分段泳道或左上到右下的时序图。
6. 正常使用页面宽度，依靠浏览器缩放和响应式重排适配；不要为了制造视觉边界而额外包卡片。
7. 移动端不得依赖横向滚动理解主图；桌面可左右对照；移动端或内容复杂时自动改为上下对照，但语义仍保持“前 / 后”。
8. 移动端不能把整张 SVG 等比缩成缩略图；若桌面主图依赖宽画布、左右旁路或大面积 SVG，移动端必须改为纵向流程、分段 SVG
   或可读的阶段轨道，保持字号、节点和箭头可直接阅读。

## 通用视觉规则

1. 只保留一张主图，不要做“主链路卡片 + 分层卡片 + 关系卡片 + TODO 卡片”的拼盘。
2. 不要常驻“如何阅读此图”侧栏；主图必须自解释。
3. 节点放名称、短职责、类型标签和必要细节；输入输出、风险、证据结论优先写进节点内部，至少紧贴节点呈现，点击弹窗只放原始长材料或重复说明。
4. 不允许文字溢出、节点遮挡、线条穿正文、背景线穿过标题或节点。
5. 移动端不允许横向溢出；390px 宽度左右仍需可读。
6. 默认使用浅色工程蓝图风格：克制、清晰、低噪音。
7. 状态不能只靠颜色表达，必须有文字标签或图例。

## 图形质量硬红线

这些规则适用于所有图型，违反任一条都必须重排，不得交付：

1. 任何图都禁止文字被节点、连线、标签或背景层遮挡。
2. 禁止节点正文溢出节点边框；必要时优先增大节点、增加高度、换行或拆相邻子节点，不能把关键正文转移到底部卡片或弹窗。
3. 顶部标题必须克制：桌面建议 20-30px，移动端建议 18-24px；禁止用超大标题挤压主图。
4. 顶部描述最多一行，只写读图必要上下文；不要在顶部堆需求背景、解释语或总结。
5. 箭头必须短、直、少交叉；优先使用同一方向的内联短箭头、正交折线或阶段连接。
6. 连线不得穿过节点正文、标题、标签、图例或弹窗触发区。
7. 若主路径需要大量跨层长箭头，必须改为编号步骤、阶段轨道、泳道分段或辅助规则矩阵。
8. 发现遮挡或箭头混乱必须重排，不得交付；发现关键细节只能点击后才可读也必须重排；不要用“可点击详情”掩盖主图不可读。

## 点击交互规则

1. 点击交互是可选增强，不是信息主入口；不点击任何节点也必须能读懂 HTML 图。
2. 如果提供弹窗，点击节点后打开小弹窗，不要让页面自动滚到底部。
3. 弹窗居中或靠近视口中心显示。
4. 点击空白处关闭弹窗，Esc 关闭弹窗。
5. 弹窗必须有关闭按钮。
6. 关闭弹窗后，焦点回到触发节点。
7. 弹窗内容只允许补充节点名称、类型、职责、输入、输出、风险、证据或待确认点的扩展说明；这些关键项的摘要必须已在主图、旁注、矩阵或证据表中可见。
8. 点击详情里的换行必须使用真实换行、`&#10;` 或渲染前归一化；禁止让用户看到字面量 `\n`、`\t`、HTML 实体源码或 JSON 转义痕迹。

## CSS 与可访问性规则

必须满足或等价实现：

- `box-sizing: border-box`。
- `overflow-wrap: anywhere`。
- 节点文字允许换行，禁止节点正文使用 `white-space: nowrap`。
- 节点正文使用 `height:auto` 或仅设置 `min-height`；禁止 `line-clamp`、正文 `max-height`、`overflow:hidden` 裁切节点内容。
- SVG 节点文字规则：优先使用 HTML/CSS 节点承载可换行正文；若使用 SVG `<text>`，必须使用 `<tspan>` 分行、`foreignObject`
  或缩短为编号标签；禁止把长句直接放进单个 SVG `<text>` 节点；交付前必须用 390px 与桌面宽度检查节点正文不溢出。
- 正文字号不小于 12px，主节点标题建议 14px 以上。
- 交互元素必须有 `:focus-visible`。
- 支持 `prefers-reduced-motion` 降级。
- 使用 button/a/details 等语义元素，禁止 div 假按钮。

## 证据与不确定性

1. 事实型图必须从用户材料、仓库文件、函数名、配置、README、测试或运行证据中取证。
2. 不能确认的内容写“待确认”“未发现”或“推断”，不能画成确定事实。
3. 仓库架构图需要尽量标出文件路径或关键锚点。
