# HTML 图形表达协议

当用户要求画系统架构图、业务流程图、代码时序图、故障排查图、页面设计稿，或希望用图沟通需求、设计、代码逻辑、故障链路时，必须优先生成**单文件 HTML 图**。目标是“一图胜千言”：打开即能读懂，点击只做补充，截图也能阅读。

## AGENTS 配合协议

当 AGENTS 要求默认通过 HTML 图沟通时，本 skill 负责把任务阶段转换成可读的单文件 HTML 图。AGENTS 只判断是否触发；本 skill 决定图型、布局、质量红线、交互与附件交付。

| 任务阶段 | 默认图型 | 必须表达 |
| --- | --- | --- |
| 功能迭代 / 开发设计 | 前后差异对比图 或 技术设计图 | 现状、目标方案、前后差异、影响面、接口/DB/模块、测试矩阵、风险与回滚 |
| 缺陷排查 / 故障分析 | 故障排查图 | 现象、影响、证据链、可疑节点、已确认根因、高亮根因节点、修法、验证与回滚 |
| 系统/功能理解 | 系统架构图、业务架构图、代码时序图 或 状态/数据模型图 | 组件、依赖、中间件、DB、MQ/队列语义、关键调用链、业务规则或状态口径 |
| 设计定稿 / 方案确认 | 需求 / 决策沟通图 或 页面设计稿 | 问题、目标、选项、推荐项、AC、待决策项、前后体验变化 |
| 最终交付 / 验收说明 | 交付影响图 或 验证闭环图 | 改动文件、用户可见变化、验证命令、未覆盖风险、待执行脚本 |

- 每个 HTML 图都必须能脱离聊天记录独立阅读：标题、目标、主路径、关键结论、证据/待确认项必须在图内自解释。
- 功能迭代的前后差异对比图必须明确“当前 / 修改后 / 不变 / 风险 / 回滚”，避免只画目标方案。
- 缺陷排查的根因节点只有证据成立后才能标为“根因”；证据不足时只能高亮为“可疑节点”或“待验证”。
- 若 AGENTS 判定为琐碎任务或用户明确要求不要画图，可不生成 HTML，但不得把这个例外扩大到开发设计、缺陷排查、架构理解和方案定稿。

## 交付铁律

1. 交付目标按来源判断：**Codex 默认**，vibego 注入了 Telegram 来源上下文时按 Telegram 交付。
2. Codex 默认场景：最终回复必须给一个可点击的 Markdown `file://` 链接，例如 `[打开 HTML 图](file:///绝对路径/xxx.html)`，并保留绝对路径作为兜底；禁止只给相对路径或裸路径。
3. Telegram 来源场景：最终回复必须显式引用项目内 `.html/.htm` 文件绝对路径或相对路径，触发 worker 自动 `send_document`；也就是最终必须直接发送 `.html` 文件，必须作为文件附件发送，用户侧交付标准是 Telegram 中必须看到文件卡片，而不是只有文字路径。
4. Telegram 来源场景不要把 `file://` 作为 Telegram 主入口；如同时提供 Codex 兜底链接，必须放在附件说明之后，且文案保持简短。
5. 不需要 PNG；只有用户明确要求图片预览时，PNG 才能作为 fallback，且不能替代 HTML 文件。
6. 禁止只发送 Markdown 链接、截图预览、文件路径或“打开 HTML”文字；这些只能作为对应平台主交付之外的补充说明。
7. 如果当前平台无法发送附件且 `file://` 链接不可用，必须输出完整 HTML 代码块，让用户可直接复制保存，并明确说明“当前平台不支持附件发送”。
8. 不要输出 Mermaid、PlantUML、Graphviz、纯 Markdown 说明或图片提示词，除非用户明确要求。
9. 单 HTML 内必须包含 `<!doctype html>`、内联 CSS、少量原生 JS；禁止外部 CDN、npm 包、远程字体、远程图片和构建工具。
10. 主内容必须 static-first：核心节点、层级、关系、结论直接写在 HTML DOM 中；禁止首屏主内容依赖 JS 动态渲染。
11. JavaScript 只能做 progressive enhancement：节点详情、小弹窗、选中态、关闭弹窗、复制、打印。

## 自动路由规则

收到制图需求后，先判断用户真正要理解什么，再选择一种主图型。不要因为材料里同时出现“架构、流程、接口、状态”就混画；主图只回答一个核心问题，其他内容放弹窗、矩阵或证据区。

| 用户需求信号 | 自动路由到 | 主结构 | 必须避免 |
| --- | --- | --- | --- |
| 用户提到系统、部署、组件、服务、进程、worker、中间件、DB、MQ、运行时 | 系统架构图 | 北向南分层模块地图 | 蜘蛛网连线、跨层长箭头 |
| 用户提到业务能力、领域、对象、规则、价值链、角色协作、业务边界 | 业务架构图 / 领域地图 | 能力分层 + 领域对象关系 | 把业务架构误画成执行流程 |
| 用户提到流程、办理、审批、流转、主路径、异常分支、用户旅程 | 业务流程图 | BPMN-light 或阶段轨道 | 密集泳道表格、空白格子 |
| 用户提到调用链、函数、事件、回调、异步、重试、事务、内部复杂逻辑 | 代码时序图 | 参与者列 + 时间自上而下 | 把源码结构画成泛架构图 |
| 用户提到状态、状态流转、生命周期、实体关系、表结构、数据口径 | 状态 / 数据模型图 | 状态机、ER-lite 或数据流 | 把状态变化塞进流程节点小字 |
| 用户提到 Bug、故障、排查、根因、验证、回滚、证据链 | 故障排查图 | 现象 → 证据 → 假设 → 根因 → 修法 → 验证 | 未验证结论画成确定事实 |
| 用户提到页面、设计稿、交互、布局、移动端、空/错/加载态 | 页面设计稿 | 单主路径界面草图 | 功能清单式卡片堆叠 |
| 用户提到 API、数据库、模块、契约、部署、回滚、开发设计定稿 | 技术设计图 | 契约 + 模块 + 数据 + 发布回滚 | 只画概念图不落接口/表/风险 |
| 用户提到需求评审、方案选择、A/B/C/D、AC、决策定稿 | 需求 / 决策沟通图 | 问题 → 选项 → 推荐 → AC → 风险 | 把讨论纪要平铺成大段文字 |

路由冲突优先级：

1. 用户明确点名图型时，优先按点名图型生成。
2. 若同时出现架构和流程：想看组件依赖选系统/业务架构；想看动作先后选业务流程。
3. 若同时出现流程和时序：跨角色业务动作选业务流程；代码调用、回调、事务选代码时序。
4. 若同时出现设计和实现：用户要确认页面长相选页面设计稿；要确认接口、表、模块选技术设计图。
5. 若材料不完整：仍画最能回答主问题的图，未知项标注“待确认”，不要脑补。

## 先选一种主图型

不要混画。材料很复杂时，只选一种主图型做主图，其他信息放弹窗、规则表、证据区或 TODO 区。

- **系统架构图**：模块、进程、worker、服务、DB、MQ/队列语义、运行时、中间件、外部系统、部署边界。
- **业务架构图 / 领域地图**：业务能力、领域对象、业务规则、角色边界、价值链、上下游协作。
- **业务流程图**：角色、用户旅程、审批流、状态流转、异常分支、规则矩阵。
- **代码时序图**：请求链路、事件链路、异步任务、回调、重试、幂等、事务边界。
- **状态 / 数据模型图**：状态机、生命周期、实体关系、表结构、字段口径、数据流向。
- **故障排查图**：现象、影响、证据、假设、根因、修法、验证、风险、回滚。
- **页面设计稿**：页面主路径、布局层级、交互状态、空/错/加载态、响应式、可访问性。
- **技术设计图**：API 契约、数据库、模块职责、并发一致性、发布回滚、观测点。
- **需求 / 决策沟通图**：问题、目标、约束、方案对比、推荐方案、验收标准、待决策项。

## 图形语法硬约束

卡片堆积不是图。卡片不是全局禁用，但必须限用。所有图型都必须显式表达关系，而不是把 Markdown 大纲换成一组圆角卡片。

1. 生成 HTML 前，必须先选定一种视觉语法：分层拓扑、阶段轨道、泳道流程、时序轴、状态机、ER-lite、因果链、决策树、矩阵热区或页面线框之一。
2. 允许使用卡片承载摘要、节点、泳道单元或分组边界；卡片必须服务箭头、泳道、坐标、层级、时序或状态转换，不能通过堆卡片来冒充一图胜千言。
3. 禁止把卡片作为唯一图形语法；如果整张图的主要信息只能靠阅读卡片文字理解，而不是靠空间、连线、方向、阶段或状态理解，必须重画。
4. 主画布必须占据首屏主要面积，读者第一眼看到的是“关系结构”，不是标题下面的卡片列表。
5. 辅助信息不得与主图同等视觉重量；证据、风险、测试矩阵、方案说明只能作为贴近主图的标注、角标、旁路、缩略矩阵、可点击详情或下方轻量表格。
6. 禁止把“主链路卡片 + 证据卡片 + 假设卡片 + 方案卡片 + 表格”并列铺满画面；这属于卡片堆叠，不属于一图胜千言。
7. 如果去掉箭头、坐标轴、泳道、分层、包含关系或状态转换线后，剩下的仍然是一组同等权重文字卡片，必须重画。
8. 每种图都必须有一个视觉主谓宾：谁作用于谁、什么导致什么、数据从哪到哪、状态如何转移、哪个选项支配哪个结果；无法表达时应换图型，而不是继续加卡片。

## 累计用户约束不可回退

视觉风格可以调整，但不得覆盖制图硬规则。柔和卡片风格只是一种视觉外观，不是图形语法豁免；回到某种样式只代表外观回退，不代表前序制图要求回退。

前序用户约束必须同时满足：

1. 一图胜千言，而不是卡片胜千言；卡片只能作为节点、摘要、泳道单元或分组边界，不能替代关系结构。
2. HTML 宽度不友好、长度可承载推理；默认正常页面宽度、北向南主线、局部横向关系，复杂内容用纵向滚动展开。
3. 完全从左到右只适合很短流程；长流程必须改为北向南、阶段轨道、泳道或左上到右下的时序图。
4. 根因、修法、验证闭环必须是主路径焦点；不能把根因和计划修法藏在说明区或底部选项区。
5. 大改动使用 before / after 对照，小改动使用单图高亮；before 一律在左或上，after 一律在右或下。
6. 故障排查、开发迭代必须具备流程图语法：开始/结束事件、活动节点、决策菱形、带标签箭头、验证和回滚闭环。
7. 背景系统必须统一；浅色背景可以有高级白色系明度层次、低对比工程网格、点阵或坐标纸肌理，但不得局部割裂或抢主线。
8. 视觉可以柔和、圆角、浅蓝白，但主图仍必须靠箭头、泳道、坐标、层级、时序或状态转换来读懂。

## 视觉质量门禁

禁止交付原始工程草图感的 SVG。图可以克制，但不能粗糙；可以信息密集，但不能靠硬边框和说明文字堆出来。

1. 视觉质量必须服务流程阅读：第一眼能看出主路径、关键节点、分支、根因、修法和验证闭环。
2. 使用统一的线宽、字号、留白、层级和图例；同一张图内不得混用多套边框、色块、字体尺度和箭头样式。
3. 禁止粗暴边框、重阴影、满屏说明文字和低级默认样式；宁可减少节点，也不要用廉价装饰弥补结构不清。
4. 流程节点必须像图形符号而不是 UI 容器：事件、活动、决策、状态、实体、泳道和连接点要各有明确形态。
5. 主图应保留足够留白，文字短句化；详细解释进入弹窗、旁注或下方轻量规则区，不能塞进节点。
6. 颜色只用于状态和路径强调，不用于装饰；根因、风险、修法、验证必须同时有文字标签，不能只靠颜色识别。
7. 背景可以使用极淡工程纸、轴线或分层底纹，但不得抢主路径，也不得用大面积容器制造“图已经完成”的错觉。
8. 浅色背景默认以白色为主色，但白色背景不能是扁平纯白；使用瓷白、珍珠白、雪白、雾白等白色系明度层次；允许使用极轻白底工程网格、点阵或坐标纸肌理，但网格必须低对比、低存在感；背景纹理必须全局统一，HTML body、SVG 主画布和弹窗遮罩以外的页面区域应共享同一背景系统，不要只在主画布局部铺网格或局部底纹；避免蓝色底、灰色底、米黄纸张感背景；背景只能提供质感和空间层次，不得抢主线。

## HTML 画布方向规则

HTML 的宽度不友好，但长度可以承载推理。宽度服务阅读，长度服务推理。

1. 默认采用正常页面宽度 + 北向南主线 + 局部横向关系；不要通过卡片限制宽度，也不要按横向海报思路把所有节点塞进一屏。
2. 禁止为了塞满首屏而横向卡片化；首屏只放结论、图例和主路径起点，后续用纵向滚动展开阶段、证据、分支、矩阵、风险和回滚。
3. 复杂图默认优先北向南：北向南分层架构、纵向故障故事线、纵向状态生命周期、纵向阶段轨道或纵向时间轴。
4. 需要表达调用先后时，或采用左上角向右下角的时序图或斜向流程：时间从左上推进到右下，避免纯水平拉长。
5. 完全从左到右只适合很短的流程图；超过 5 个主节点不得继续横向铺开，必须改为北向南、分段泳道或左上到右下的时序图。
6. 正常使用页面宽度，依靠浏览器缩放和响应式重排适配；不要为了制造视觉边界而额外包卡片。
7. 移动端不得依赖横向滚动理解主图；桌面可左右对照；移动端或内容复杂时自动改为上下对照，但语义仍保持“前 / 后”。
8. 移动端不能把整张 SVG 等比缩成缩略图；若桌面主图依赖宽画布、左右旁路或大面积 SVG，移动端必须改为纵向流程、分段 SVG 或可读的阶段轨道，保持字号、节点和箭头可直接阅读。

## 根因、修法与前后对照规则

根因和修复方案不是说明文字，必须成为主图中的视觉焦点。

1. 缺陷排查和开发设计默认使用单图高亮根因与修法：根因节点标 `根因 R#`，计划修法节点标 `修法 F#`，验证节点标 `验证 V#`；不能只靠颜色表达。
2. 证据必须贴近或连到根因节点，测试与回滚必须贴近或连到修法节点，避免证据区、方案区、风险区各自漂浮。
3. 当修复改变两处以上关键节点，或改变调用链、数据流、状态流转、接口契约、权限边界、用户主路径之一时，必须改用前后对照图。
4. 前后对照图必须明确：左/上为当前逻辑或故障逻辑，右/下为修复后逻辑或目标逻辑，中间或旁路标出保留、删除、新增、风险和回滚点。
5. 若前后差异很小，禁止为了形式感画双图；用单图高亮根因与修法更清晰。
6. 若前后双图在桌面宽度内仍会拥挤，必须改为上下两图或纵向阶段对照，不得缩小字体或继续横向加卡片。
7. before 固定在左侧或上方，after 固定在右侧或下方；这是前后对照图的阅读约定，不能随节点编号或美观需要调换。
8. 禁止用左右交替排布表达前后差异；左右交替只能用于非对照型时间线，且不得与 before / after 语义混用。
9. 纵向卷轴中也必须保持左侧为当前/故障逻辑，右侧为修复后/目标逻辑；移动端折叠时保持上方为 before、下方为 after。
10. 如果节点不属于前后对照，只能放在中轴、旁注或详情中，避免把中性说明误读为 before 或 after。

## 流程图语法门禁

前后对照只是容器，不是图形语法本身。功能迭代、开发设计和故障修复必须优先画流程图或流程化对照图。

1. 主画布必须包含开始/结束事件、活动节点、决策菱形、带标签箭头；缺少这些流程图基本元素时，不得把结果称为流程图。
2. before/after 每一侧内部也必须是流程图：当前逻辑要展示旧流程如何走到问题点，修复后逻辑要展示新流程如何绕过或消除问题点。
3. 禁止把 before/after 列画成普通说明卡片列表；每一列至少要有 3 个按方向连接的流程节点，且必须有一个明确输入和一个明确输出。
4. 根因节点和修法节点必须落在流程路径上，不能浮在旁边当说明卡片；根因节点用 `根因 R#`，修法节点用 `修法 F#`。
5. 辅助证据只能作为流程节点的证据锚点、旁注或点击详情；不得把证据列表、假设列表、方案列表画成与流程同等重量的主内容。
6. 对开发迭代图，必须画出用户/系统当前主流程、变更点、目标主流程、验证闭环；对故障修复图，必须画出排查流程、判断分支、根因定位、修复动作、验证/回滚闭环。
7. 若流程跨角色或跨端，使用泳道或阶段轨道；若流程跨状态，使用状态流转；若流程跨接口/数据，使用时序或数据流。不要用卡片网格替代这些语法。

## 通用视觉规则

1. 只保留一张主图，不要做“主链路卡片 + 分层卡片 + 关系卡片 + TODO 卡片”的拼盘。
2. 不要常驻“如何阅读此图”侧栏；主图必须自解释。
3. 节点只放名称、短职责、类型标签；长解释、证据、输入输出、风险进入点击弹窗。
4. 不允许文字溢出、节点遮挡、线条穿正文、背景线穿过标题或节点。
5. 移动端不允许横向溢出；390px 宽度左右仍需可读。
6. 默认使用浅色工程蓝图风格：克制、清晰、低噪音。
7. 状态不能只靠颜色表达，必须有文字标签或图例。

## 图形质量硬红线

这些规则适用于所有图型，违反任一条都必须重排，不得交付：

1. 任何图都禁止文字被节点、连线、标签或背景层遮挡。
2. 禁止节点正文溢出节点边框；必要时增大节点、缩短文案、换行或改为弹窗详情。
3. 顶部标题必须克制：桌面建议 20-30px，移动端建议 18-24px；禁止用超大标题挤压主图。
4. 顶部描述最多一行，只写读图必要上下文；不要在顶部堆需求背景、解释语或总结。
5. 箭头必须短、直、少交叉；优先使用同一方向的内联短箭头、正交折线或阶段连接。
6. 连线不得穿过节点正文、标题、标签、图例或弹窗触发区。
7. 若主路径需要大量跨层长箭头，必须改为编号步骤、阶段轨道、泳道分段或辅助规则矩阵。
8. 发现遮挡或箭头混乱必须重排，不得交付；不要用“可点击详情”掩盖主图不可读。

## 系统架构图规则

系统架构图默认使用北向南单图结构：

`外部入口 → 控制面 → 工作面 → 运行与中间件 → 状态与证据面`

必须清晰展示：

1. 组件名称：服务、进程、worker、controller、service、repository、脚本。
2. 外部入口：用户、客户端、API、Bot、Webhook、CLI。
3. 控制面：编排器、配置管理、状态管理、调度器。
4. 工作面：业务处理模块、任务服务、命令服务、会话服务。
5. 运行与中间件：runtime、queue、tmux、worker pool、scheduler、MQ/broker、模型 CLI、第三方 SDK。
6. 数据与证据：DB、SQLite、Postgres、Redis、JSONL、日志、pointer、文件系统、对象存储。
7. 如果没有独立 MQ，也要明确画出“无独立 MQ / 由某模块承担队列语义”。
8. 层间只用短箭头和短标签表达主流向；禁止蜘蛛网连线和长距离跨层箭头。
9. 背景不得出现穿过文字、标题或节点的装饰线。

## 业务架构图规则

业务架构图 / 领域地图默认表达“业务由哪些能力、对象和规则构成”，不是表达“操作先后顺序”。

- 推荐结构：参与方 / 业务能力层 / 领域对象层 / 规则与约束层 / 外部上下游。
- 每个能力节点只写能力名、目标、归属角色；详细规则进入弹窗或规则矩阵。
- 领域对象要标出核心关系：拥有、依赖、产生、约束、审批、结算、归档等。
- 业务规则必须显式标注口径、阈值、优先级、状态边界；不能只写“按规则处理”。
- 上下游系统、人工角色、第三方服务要画出边界，不要混在同一节点里。
- 如果出现时间顺序、审批先后、异常回退，应切到业务流程图，不要在业务架构图里画长流程。

## 业务流程图规则

默认采用 **BPMN-light 横向主流程**，而不是密集泳道表格。

- 使用事件圆点表示开始/结束，圆角矩形表示活动，决策菱形表示判断，细连接线表示流向。
- 泳道只表达责任归属，使用弱背景带或左侧角色标签；禁止把业务流程图画成密集表格，不要铺满行列网格和空白格子。
- 主路径只保留 5 到 9 个关键步骤，超过则拆阶段或折叠为阶段组。
- 每个节点只写编号、动词短句、结果标签；说明、输入输出、风险、证据进点击弹窗。
- 决策分支必须从菱形网关分出，正向主路径保持水平，异常/退回分支放下方或旁路，回流用短标签“返回 Sx”。
- 多条件、多角色、多状态规则必须用矩阵表作为辅助区，不画蜘蛛网。
- 禁止出现大量“等待、无需操作、空白占位”节点；没有业务动作的格子不画。
- 视觉优先级：主路径连接线 > 关键节点 > 分支 > 责任归属 > 规则矩阵；网格线不得抢主路径焦点。

## 代码时序图规则

- 使用“参与者列 + 时间自上而下”。
- 参与者建议 3 到 6 个，超过则合并为逻辑边界。
- 每行最多一个主调用；返回、异常、重试、异步回调分行表达。
- 每个步骤绑定源码证据：文件路径、函数名、调用点、输入、输出、异常；无证据写“待确认”。
- 事务边界用半透明区块标注 begin/commit/rollback。

## 状态 / 数据模型图规则

状态 / 数据模型图默认回答“对象有哪些状态、状态如何变化、数据如何约束流程”。

- 状态机：状态节点只写状态名、进入条件、退出条件；动作写在线上，不写进状态标题。
- 数据模型：使用 ER-lite，实体、关键字段、唯一约束、索引、外键/逻辑关联必须分清。
- 生命周期：按创建、处理中、完成、失败、归档或回滚等阶段组织，不要把所有字段塞进节点。
- 字段口径必须标注来源、单位、枚举值、是否可空、幂等键或版本号。
- 并发、一致性、补偿、重试、撤销、软删等数据风险必须作为辅助标记。
- 如果重点是“谁先做什么”，应切业务流程图；如果重点是“哪个函数调用哪个函数”，应切代码时序图。

## 故障排查图规则

故障排查图默认回答“用户怎样最快看懂故障来龙去脉”，不是把根因分析文字搬进 HTML。

- 主结构固定为：故障故事线 + 证据阶梯 + 假设裁决 + 行动闭环。
- 故障故事线只保留 5 到 7 个节点：现象、影响、触发点、关键证据、根因或最高可疑点、修法、验证状态；节点只写短句和状态标签。
- 故障故事线必须画成一条明确因果路径，根因或最高可疑点是视觉焦点；证据、假设、行动必须锚定到故障故事线节点，禁止各自另起一排等权重卡片。
- 证据阶梯按 E1/E2/E3 编号展示：来源、观察结果、指向的判断；日志、截图、JSONL、SQL、命令输出等长内容进入点击详情，不进入主图节点。
- 假设裁决必须分为“成立 / 排除 / 待验证”三类；每个假设必须绑定证据编号或写“待补证据”。
- 根因节点必须同时显示证据编号、置信度、验证状态；未验证只能标为最高可疑点，不能写成已确认根因。
- 主图不写长段根因分析；根因推理只用“因为 E1+E2，所以 R1 成立”这类一跳因果短句，完整推理放详情弹窗或证据表。
- 行动闭环必须展示：已做修法、验证命令/结果、回滚动作、剩余风险；验证未执行或失败时，不得写“已修复”。
- 视觉优先级：故障故事线 > 根因或最高可疑点 > 验证状态 > 证据编号 > 详细日志；避免把证据表、假设树和修法清单画成同等重量的卡片堆叠。

## 页面设计稿规则

生成页面设计 HTML 前先做 5 个极短判断：

1. 唯一核心目标是什么？
2. 用户当前最需要完成什么动作？
3. 首层必须展示什么？
4. 哪些内容应收纳到详情、抽屉、折叠区、二级 Tab 或更多菜单？
5. 是否存在卡片堆叠、标题废话、功能平铺？如何规避？

页面只能服务一个主路径；禁止把需求列表直接翻译成卡片堆叠；必须体现默认、加载、空、错误态，并覆盖移动端、平板、桌面。

## 技术设计与需求决策图规则

技术设计图默认回答“如何落地实现、有哪些契约、如何验证与回滚”；需求 / 决策沟通图默认回答“为什么做、怎么选、怎样验收”。

- 技术设计图必须覆盖：受影响目录、接口/API 契约、数据库/字段/索引、模块职责、关键流程、测试矩阵、风险与回滚。
- 需求 / 决策沟通图必须覆盖：用户场景、业务目标、核心规则、方案对比、推荐项、验收标准、未决问题。
- 方案对比只保留 2 到 4 个选项；每个选项写优点、缺点、适用边界和回滚成本。
- 推荐方案要视觉突出，但必须保留反例和风险；不要把推荐方案画成唯一事实。
- 开发设计定稿要标出 QPS/TPS、权限、幂等、重试、日志、指标、告警和灰度发布。
- 如果用户要看页面长相，应切页面设计稿；如果用户要看运行组件，应切系统架构图。

## 点击交互规则

1. 点击节点后打开小弹窗，不要让页面自动滚到底部。
2. 弹窗居中或靠近视口中心显示。
3. 点击空白处关闭弹窗，Esc 关闭弹窗。
4. 弹窗必须有关闭按钮。
5. 关闭弹窗后，焦点回到触发节点。
6. 弹窗内容包括：节点名称、类型、职责、输入、输出、风险、证据或待确认点。

## CSS 与可访问性规则

必须满足或等价实现：

- `box-sizing: border-box`。
- `overflow-wrap: anywhere`。
- 节点文字允许换行，禁止节点正文使用 `white-space: nowrap`。
- 正文字号不小于 12px，主节点标题建议 14px 以上。
- 交互元素必须有 `:focus-visible`。
- 支持 `prefers-reduced-motion` 降级。
- 使用 button/a/details 等语义元素，禁止 div 假按钮。

## 证据与不确定性

1. 事实型图必须从用户材料、仓库文件、函数名、配置、README、测试或运行证据中取证。
2. 不能确认的内容写“待确认”“未发现”或“推断”，不能画成确定事实。
3. 仓库架构图需要尽量标出文件路径或关键锚点。

## 输出前自检

交付前必须检查并修正：

- 是否只生成一个单文件 HTML？
- 是否直接发送 HTML 文件或完整 HTML 代码块？
- 是否没有外部依赖？
- 是否已按自动路由规则选择唯一主图型？
- 所选图型的必选元素是否齐全？
- 是否只有一张主图？
- 是否没有常驻“如何阅读此图”侧栏？
- 是否节点文字不溢出、无背景线穿字、无横向溢出？
- 点击节点是否打开小弹窗，空白/Esc 可关闭，且不会自动滚动页面？
- 系统架构图是否明确展示组件、中间件/运行时、DB、MQ/队列语义？
- Codex 默认场景是否提供可点击 `file://` HTML 链接和绝对路径兜底？
- Telegram 来源场景是否能看到 `.html` 文件卡片，而不是只有 Markdown 链接或路径文字？

最终回复要短，只说明 HTML 已生成/已发送；不要输出冗长过程日志。
