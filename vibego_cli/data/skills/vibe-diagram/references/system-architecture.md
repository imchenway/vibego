# vibe-diagram reference: system-architecture

> 本文件由 TASK_20260630_022 拆分自 `vibe-diagram/SKILL.md`。只有路由命中对应图型后才读取；读取失败必须 fail-closed。


## 候选全集清单

- 生图类型：系统架构图
- 首选候选：北向南分层拓扑
- 必生成备选候选：
  - 主请求中轴 + 控制/数据/兜底泳道
  - 运行时依赖拓扑
- 普通请求要求：系统架构普通请求默认只生成一张 presentation 定稿图，不得生成候选 tab、候选 panel 或候选对比矩阵。
- 校准期要求：只有用户明确要求多候选、校准、对比、视觉探索或任务文档明确要求候选全集时，才按候选 A/B/C/D…生成首选候选与全部备选候选；每个候选都必须是真图。信息不足时也必须生成该备选候选，用“待确认节点”标明缺口。

## 系统架构图 v6 路由器

系统架构图不再直接套一个固定业务 archetype。命中 `system-architecture` 后必须先做 v6 路由：**C4 主线定抽象层级 → 专项扩展补充关注面 → 必要时跨图型转交**。路由只决定骨架，不决定业务内容；所有层名、节点名、模块数、图标和文案都来自用户材料或仓库证据，证据不足写“待确认”。

### C4 主线

| selected_view | 何时选择 | 画布骨架 |
| --- | --- | --- |
| `context` | 信息很少、要说明系统边界、用户和外部系统 | 系统边界 + 外部角色 / 外部系统 + 关键关系 |
| `container` | 默认系统架构图、要说明可部署单元、数据存储和通信 | 入口/外部系统 + 可部署单元 + 数据/消息/外部依赖 + 主流向 |
| `component` | 用户要求展开某个容器内部职责、接口或模块协作 | 单容器白盒 + 内部组件 + 接口/数据关系；不混全局部署图 |
| `deployment` | 用户关注云、环境、节点、集群、托管关系、网络近邻 | 环境边界 + 运行节点 + 部署单元映射 + 托管/连接关系 |

默认规则：用户只说“系统架构图”且材料能识别多个可部署单元时选择 `container`；材料很少时选择 `context`；明确内部白盒时选择 `component`；明确部署/环境时选择 `deployment`。

### 专项扩展

专项扩展只能作为 `companion_views` 或局部叠加，不得把所有专项都塞进一张主图。可选值：

- `logical`：逻辑分层、职责边界、能力分层。
- `data-architecture`：数据域、存储、血缘、治理与质量控制点。
- `data-flow`：DFD、敏感数据流、信任边界、外部实体、进程和存储。
- `api-integration`：协议、契约、webhook、第三方集成、同步/异步语义。
- `event-driven`：事件源、队列/主题、生产者/消费者、最终一致。
- `network`：VPC/子网/网关/防火墙/专线/出入口路径。
- `security`：资产、边界、威胁、控制、审计和安全证据。
- `identity`：认证、授权、主体、token、SSO、RBAC；复杂安全审查升级到 `security`。
- `resilience`：故障域、降级、切换、备份、RTO/RPO 和恢复验证。
- `observability`：日志、指标、追踪、告警、查询和响应闭环。
- `ci-cd`：构建、制品、发布门禁、部署策略和回滚路径。

### 跨图型转交

如果用户真实问题不属于系统架构，不要强塞进系统架构图：

- `business-architecture`：业务能力、价值链、角色协作、业务边界。
- `domain-model`：领域对象、规则、聚合、概念关系。
- `er`：表、字段、主外键、数据口径。
- `state-machine`：状态生命周期、事件触发、状态约束。
- `sequence`：函数调用、请求时序、异步回调、异常返回或单一运行场景。

跨图型转交时，系统架构图只保留必要上下文；主图型应切换到对应 reference。

### 置信度与路由优先级

1. 用户显式点名图型优先。
2. 问题意图优先于关键词：先判断“读者想理解什么”。
3. 安全、合规、韧性、数据敏感等风险域命中时，可生成专项 companion。
4. `routing_confidence >= 0.75`：直接路由。
5. `0.45 <= routing_confidence < 0.75`：生成一个主图，并在图内标注建议 companion views。
6. 低于 0.45：退回 `context` 或 `container`，所有缺口标“待确认”，不得脑补。

禁止项：不要把所有模板混入一张图；不要只靠关键词路由；不要固定层名、固定业务模块或固定图标；不要把客服 / Agent / RAG / LLM / SDK / Gateway 等词当作默认业务内容；不要使用候选 tab；不要横向滚动；不要让主图退化成多列文字卡片。模板的职责是固定视觉语法和质量门禁，内容槽位可变。

## DiagramSpec 生成前置契约

生成系统架构图前必须先在脑内形成并在 HTML 结构中体现 DiagramSpec，普通请求固定为：

```text
diagram_type=system-architecture
mode=presentation
audience=大众认知架构图读者
primary_question=这个系统的边界、可部署单元、关键依赖、数据/运行支撑和风险关注面如何协作？
selected_view=context|container|component|deployment|logical|data-architecture|data-flow|api-integration|event-driven|network|security|identity|resilience|observability|ci-cd
routing_confidence=0.00-1.00
companion_views=[logical|data-architecture|data-flow|api-integration|event-driven|network|security|identity|resilience|observability|ci-cd]
handoff_target=business-architecture|domain-model|er|state-machine|sequence|none
handoff_reason=...
slot_policy=flexible
default_archetype=入口 rail + 应用/服务核心 + 支撑能力 + 数据/观测 + 基础设施 + 管理/运维侧栏
```

执行约束：

1. `mode=presentation` 时只输出一张主图；不得生成候选 tab、候选 panel 或候选对比矩阵。
2. HTML 主容器应带可检查标记：`data-diagram-type="system-architecture"`、`data-diagram-grammar="system-architecture-presentation"`、`data-system-arch-view="selected_view"`、`data-routing-confidence="0.xx"`；这些标记不能替代真实图形层。
3. `data-system-arch-view` 必须来自 v6 路由器；`data-routing-confidence` 必须是 0 到 1 的数字。
4. 若使用用户材料中的特定业务词或技术名，可在主容器标注 `data-content-source="user-material"` 或在节点详情说明来源；没有来源时不得把特定业务词当模板默认值。
5. 若确实进入候选全集校准模式，必须在说明区写明“用户显式要求校准/多候选”的来源；否则默认按 presentation 出图。
6. 主图第一眼必须是架构拓扑，不是调研报告：标题下方直接进入架构画布，摘要、证据、风险只能作为画布内角标、节点详情或下方轻量附录。

## 大众系统架构 archetype 模板

系统架构图在 `container` 或大众总览视图下可使用大众认知里的“入口 rail + 应用/服务核心 + 能力支撑层 + 数据层 + 基础设施层 + 管理/运维侧栏”骨架；槽位可变，不固定具体业务内容：

1. **外部入口 rail**：用户/渠道入口必须画成左侧竖向 rail，例如 Web/H5、APP、微信公众号、小程序、企业微信、电话/IVR、邮件；使用小图标 + 短标签，不要做成文本卡片墙。
2. **接入/网关区**：入口 rail 右侧是 API Gateway、会话入口、鉴权、限流、路由或多渠道接入，负责把外部请求接入核心系统。
3. **核心应用/服务层必须是主视觉中心**：业务应用、服务编排、领域服务、任务处理、工作流或用户材料中的核心能力应位于画布中央，并用更强描边或主色强调；不得默认写死为客服智能体或 Agent。
4. **能力支撑层必须用图标 + 短标签**：检索、规则、工作流、工具、模型、缓存、通知、评估等支撑能力放在核心层下方或附近，以能力图标表达“支撑”；RAG/LLM 等只能在材料明确出现时使用。
5. **数据层必须区分知识库、会话数据、用户数据、运营数据**：数据层不能只是“DB”一张卡；知识库、对话数据、用户画像、行为数据、质量数据、日志/trace 至少按语义分组。
6. **基础设施层必须横向铺成底座**：云计算资源、存储、网络与安全、监控与告警、CI/CD 与运维作为最底层基础设施底座，承载上层能力。
7. **管理/运维侧栏必须靠右收纳**：智能体管理、知识管理、技能/工具管理、权限安全、效果评估、日志审计等是侧向管理能力，不要混入主请求链路。
8. **人工协同/兜底可作为右侧或旁路 rail**：人工接管、会话协同、坐席辅助、知识推荐等用旁路虚线与核心 Agent 或回复阶段相连。
9. **图标是语义，不是装饰**：用户、终端、渠道、通信、数据库、模型/规则、立方体/工具、齿轮/管理、锁/安全、图表/评估等图标必须帮助读者定位类别；具体渠道图标来自材料。
10. **主请求流用实线箭头**：入口 → 接入/网关 → 核心应用/服务 → 返回或输出必须是一条可追踪主线。
11. **数据读写流用双向或下沉箭头**：核心应用与能力支撑、数据层之间用下沉/双向箭头；异步、反馈、优化闭环用虚线。
12. **运维/管理流用侧向短箭头**：管理能力对核心系统的影响用短箭头或侧栏连接，不要穿越整张图造成蜘蛛网。

## 系统架构图 presentation 版式锁定

普通系统架构图必须优先稳定成“像大众认知系统架构图”的 presentation 画布，而不是报告页或卡片墙。默认版式锁定为：

1. **主版式**：左侧入口 rail + 中央应用层 + 南向能力/数据/基础设施 + 右侧管理/运维/兜底侧栏；画布应紧凑铺满主内容宽度，不靠横向滚动展示。
2. **节点内部布局**：节点内部内容必须作为一个整体自动水平居中和垂直居中；图标与文案是一组内容，不得分别用固定 `x/y` 坐标硬摆。
3. **高节点排布**：高节点默认图标在上、文案在下，整体居中，适合外部入口 rail、管理侧栏和竖向能力列表。
4. **紧凑节点排布**：紧凑节点默认图标在左、文案在右，整体居中，适合应用层、能力支撑层、数据层和基础设施底座。
5. **承载语法**：优先使用 `foreignObject` + HTML/CSS `grid`/`flex` 承载节点内容；如果不用 `foreignObject`，必须用等价 HTML 覆盖层或 `data-node-layout="auto-centered"` 标记证明节点内容可自动居中和换行。
6. **Emoji 策略**：emoji 图标是语义锚点，不得因为局部图例丑就全局去掉 emoji；用户只指出局部图例或箭头图标问题时，只替换该局部及其图例，不重画已被确认“对味”的整体架构。
7. **箭头语义**：每条箭头必须有明确源节点、目标节点、边缘锚点和关系标签；如果读者会问“这个箭头指向哪里”，必须改名或重连，禁止孤立的“主请求入口”“外部”等悬空标签。
8. **局部反馈编辑**：浏览器标注反馈默认只修改用户选中区域及直接相邻关系；不得把“这几个图标有点丑”理解成“全图去掉 emoji”，也不得在用户认可整体后重画主结构。
9. **失败项**：横向滚动条、节点内滚动条、文字溢出、图标/文案未居中、箭头穿字、孤立箭头标签均为失败；交付前必须压缩空白、放大可读字号，并让图标和文案在各自节点中自动居中。

## 证据预算与主图降噪

系统架构图要遵守证据预算，避免再次退化为证据卡片页：

1. 主图节点禁止展示源码路径、E# 证据列表或长运行验证命令；每个节点主文案最多 3 行：组件名、职责、协议/运行语义。
2. 文件路径、行号、命令输出和长证据只能放入点击详情、折叠附录或轻量证据索引；主图里只保留“已验证 / 待确认 / 推断”等状态角标。
3. 每个系统层最多显示 4-7 个节点；超过后合并为语义分组，并在点击详情列出子模块。
4. 证据编号可以出现在节点角标，但同一屏不超过 12 个；若需要更多证据，必须放到“证据索引”附录。
5. 如果出现大面积 `.node` 卡片、`role=tablist` 候选切换、底部证据卡片区或多列文字表格，必须先重画主图再交付。

## 产物级 lint 门禁

系统架构图生成后必须按以下规则自检；仓库内置脚本 `scripts/vibe_diagram_lint.py` 可用于检查单文件 HTML：

1. 普通 `system-architecture` 不得出现候选 tab / `role="tablist"`；只有显式校准模式可通过 `--allow-candidates` 放行。
2. 必须包含真实 SVG 主画布；`data-diagram-grammar="system-architecture-presentation"` 只是机读标记，不能替代 SVG、图标、边界和连线层。
3. 必须能在正文中找到入口、应用/Agent、能力支撑、数据、基础设施、管理/运维和主流向语义。
4. 证据密度超标、源码路径外露过多、节点卡片数量过多且缺少 SVG/架构语法时，必须判失败。
5. lint 失败时不得声称“完成”或“验证通过”；应重画并重新检查。

## 系统架构图规则

presentation 版式锁定优先级高于 007 宏观拓扑基线。普通系统架构图默认使用 v6 路由器选出的 presentation 骨架；当 `selected_view=container` 时可使用“大众架构图”骨架：

`外部入口 rail → 接入/网关 → 应用/服务核心 → 能力支撑层 → 数据层 → 基础设施底座`，并把管理/运维/人工兜底收在右侧侧栏。


兼容说明：历史样板中的“入口 rail + 智能体应用层 + 能力支撑层 + 数据层 + 基础设施层 + 管理/运维侧栏”只作为可映射的示例骨架，不代表所有系统都必须出现智能体、客服、RAG 或 LLM。

系统架构图不是组件清单、证据清单、编号实现分层或接口调用链；首屏必须出现一张宽画布架构拓扑总览，让读者第一眼看出入口、应用、支撑能力、数据面、基础设施和管理侧栏如何协作。

007 仅作为内部运行时证据架构的例外形态：只有当用户明确要求“内部实现 / 运行时证据 / 代码链路 / worker 运行拓扑”，且大众架构图模板无法承载时，才可使用北向南层级 + 层间流向分隔条 + 节点内摘要 + 点击详情证据；使用前必须在图内标注“内部运行时视图”。普通系统架构图不得使用 `1.`、`2.`、`3.` 编号 lane 作为主层标题。

实现名降噪：SDK 表现层、HTTP 接入、Controller、Facade、Handler、DTO 等实现名只能进入节点详情、tooltip 或轻量附录；主图节点应映射成“渠道入口 / 接入网关 / 会话编排 / Agent 能力 / 数据与观测 / 管理侧栏”等大众语义。不得把主画布标题写成接口调用链或代码流水线，例如“入口 → SDK → /api/cs/v1/* → Facade → Agent”。

必须清晰展示：

1. 组件名称：服务、进程、worker、controller、service、repository、脚本。
2. 外部入口：用户、客户端、API、Bot、Webhook、CLI。
3. 控制面：编排器、配置管理、状态管理、调度器。
4. 工作面：业务处理模块、任务服务、命令服务、会话服务。
5. 运行与中间件：runtime、queue、tmux、worker pool、scheduler、MQ/broker、模型 CLI、第三方 SDK。
6. 数据与证据：DB、SQLite、Postgres、Redis、JSONL、日志、pointer、文件系统、对象存储。
7. 如果没有独立 MQ，也要明确画出“无独立 MQ / 由某模块承担队列语义”。
8. 层间只用短箭头和短标签表达主流向；禁止蜘蛛网连线和长距离跨层箭头。
9. 背景不得出现穿过文字、标题或节点的装饰线。

架构拓扑执行门禁：

1. **首屏必须出现一张宽画布 presentation 拓扑总览**，而不是顶部摘要 + 多行分层卡片；外部入口、接入/网关、智能体应用、能力支撑、数据/观测、基础设施、管理/兜底应在同一张图内形成清晰空间关系。
2. **层间必须画出主请求流、控制流、数据读写流或兜底流**：至少一条从入口到数据面的主流向清晰可追踪；异步、回调、兜底、人工介入用旁路短箭头或编号支线表达。
3. **同层节点只保留组件名、职责、协议/接口、运行状态和关键约束**；源码证据、文件路径和长说明进入该节点点击详情，避免节点变成证据卡片。
4. **先画系统边界，再画组件**：外部系统、第三方 SDK、运行中服务、数据库/对象存储/日志观测必须有边界差异；不能把客户端、后端、RDS、兜底人工和
   feature flag 画成同层卡片。
5. **必须显式标注运行语义**：同步 HTTP、队列/MQ、定时任务、worker pool、SDK 内嵌、进程内调用、外部托管服务要用不同线型或标签表达；如果没有独立
   MQ / 未启动服务 / 未连接运行时 DB，要直接标注为运行状态，不要藏在说明区。
6. **如果第一眼只能看到多列卡片和证据文字，看不出入口到数据面的流向，必须重画**；系统架构图的第一阅读路径必须是“从入口沿箭头走到数据/观测面”。

系统架构图例外形态不是优化方向：

除非用户明确要求或 presentation 模板无法区分两个以上独立平面，否则不要使用泳道、分段故事线或 007 内部运行时视图；不要把 007 当作普通系统架构图推荐模板，也不要把 010/011 当作推荐模板。所谓 presentation 模板不够用，是指入口、主请求、控制、数据、兜底平面彼此强冲突，或主流向与运行语义互相遮挡。此时才进入例外形态，而且必须先说明为什么大众架构图模板不够用。

只有当 presentation 模板无法表达清楚两个以上独立平面时，才考虑侧向泳道。

中等复杂系统泳道拓扑：

1. **中等复杂系统架构可以使用“主请求中轴 + 控制面 / 数据面 / 兜底面泳道”**
   ：主请求中轴负责从入口到服务/Agent/返回的第一阅读路径；控制、数据、兜底只作为侧向或南向平面，不抢主轴。
2. **控制面不要与主请求节点等权重平铺**：Feature Flag、Guard、策略、权限、调度、限流等控制能力应挂在 Gateway、Agent
   或运行层侧边，用短箭头标出“约束/开关/调度”，不要混入主请求链路。
3. **数据/知识面应作为南向或侧向依赖泳道**：DB、RAG、对象存储、缓存、日志、trace、向量库等应按读写/检索/观测语义靠近使用它们的节点，避免在底部变成同权重数据卡片墙。
4. **兜底/人工流适合画成侧边 rail**：人工客服、第三方兜底、降级服务、fallback API 等可画成贯穿右侧或下侧的虚线
   rail，并标出触发条件；不要把兜底终点误放成数据面节点。
5. **运行语义条用于收纳跨层约束**：同步 HTTP、无独立 MQ、worker pool、进程内调用、外部托管、未连接运行时服务等全局运行状态，优先放成跨层状态条或角标，避免在每个节点重复解释。
6. **状态角标必须区分证据、运行态和缺口**：E# 表示证据索引，`运行态` 表示当前已知运行语义，`缺口/待接入/待确认`
   表示风险；不要用同一种灰色小字混写所有状态。
7. **泳道不是表格，不得把层级、控制面、主请求、数据面、兜底面全部做成等权重多列网格**：如果读者需要横向逐格扫描才能理解，就是把架构图画成表格了。
8. **先保留一条粗主线，再把控制、数据、兜底折成贴近当前阶段的侧向胶囊或短注**：侧向信息只解释当前阶段为什么能走、读写哪里、何时兜底，不能抢走主线。
9. **单个视口内主路径节点建议 3-5 个**：超过后拆成分段卷轴或阶段组；每段只回答一个问题，例如“入口如何进来”“Agent
   如何决策”“状态如何落库”。
10. **如果需要 5 列以上才能表达，必须改为分段卷轴、阶段轨道或多张局部小图**：不要靠缩小字号、压窄节点或增加横向滚动来保留多列表格。
11. **侧向泳道只能服务当前主线阶段，不得形成第二张需要逐格阅读的矩阵**：控制、数据、兜底的内容应贴近对应主线节点，跨阶段公共约束用运行语义条承载。
