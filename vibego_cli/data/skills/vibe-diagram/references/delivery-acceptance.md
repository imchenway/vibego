# vibe-diagram reference: delivery-acceptance

> 只有路由命中交付验收图后才读取；读取失败必须 fail-closed。

## 候选全集清单

- 生图类型：交付验收图
- 首选候选：验收账本 / 需求到证据签收表
- 必生成备选候选：
    - 证据泳道图
    - 风险动作板
    - 交付时间线
- 校准期要求：交付验收图保留候选按钮，但按钮只是入口；面板内部必须独立可读。不得通过删除候选按钮、删除面板或只保留单一账本来回应可读性问题。旧
  R# 泳道、证据矩阵、地铁线路不清晰时，改用更易读布局。

## 交付验收图规则

交付验收图必须默认回答“原始要求是否逐条满足、证据是否足够、还剩什么动作”，让读者沿每条需求证据链判断能否验收。

- 交付验收图必须覆盖：原始需求 / AC、交付变更、验证证据、验收判定、未覆盖项、风险、回滚和待执行动作。
- 结论必须写成验收口径，例如 `R1-R4 通过；R5 待人工截图确认；无阻塞项`。不能只写“已完成”“全量门禁通过”。
- 每一条用户需求或 AC 都必须拥有独立 R# 泳道；没有证据的需求只能标 `待确认`、`warn`、`blocked`，不能并入总通过。
- 验证命令、测试结果、截图核对和打包门禁必须贴在对应 R# 泳道内；全量门禁只能作为兜底证据，不能替代逐条需求证据。
- 风险、回滚和待执行动作必须从对应 R# 泳道引出；不要放成底部孤立卡片。
- 如果用户明确讨厌或禁止某个旧模式，必须在对应 R# 泳道放“旧模式反查”节点或角标。
- 不要用移除按钮或删除面板来解决可读性问题；保留候选切换入口，但每个候选面板内部必须包含主结论、阅读顺序和关系结构。

## 交付验收必答信息

交付验收图除 R# 轨道外，还必须在首屏或紧邻主轨道的固定信息区直接外显以下答案，不得只在最终聊天收尾字段里说明：

1. **改了什么**：列出源码、测试、文档、配置或规约的实际变更；只写“优化/修复”不合格。
2. **影响哪些功能入口**：明确用户会在哪些页面、命令、菜单、Bot 指令、API、任务流或服务入口感知变化；若无业务入口影响，写“无用户入口变化，仅影响规约/内部生成规则”。
3. **用户需要执行什么脚本**：列出需要用户执行的命令或脚本；如果无需脚本或无需重启，也必须明确写“无需执行脚本”或“无需重启服务”。
4. **需要重启什么服务**：明确要重启 worker、Web 服务、桌面应用、Bot、队列消费者、开发服务器或无需重启；不得用“按需重启”代替具体对象。
5. **如何验证**：列出聚焦测试、全量测试、构建/打包、截图/人工核对、线上观察或回归路径；每条验证方式要说明验证了哪个 R# 或哪个入口。
6. **剩余未覆盖点**：明确哪些验证没有做、为什么没做、是否需要用户补验。

每个面板上方推荐“交付说明栏”：`改了什么`、`影响功能入口`、`验证方式`、`待用户执行脚本 / 重启服务`、`剩余未覆盖点`；只做总览，细节绑定
R#。

交付说明栏不能做成 5 张等权重卡片；必须收敛成一条横向或纵向连通的信息带。旧图问题只能作为旁注或反例标记。主图第一视觉必须是当前面板的关系结构，不是解释/说明/总结卡片。

交付验收主画布不得再用矩形卡片节点承载每一步；即使有箭头也仍是卡片线路图。账本/泳道/风险动作/时间线单元必须从属于清晰主轴，用列、泳道、看板列或时间线表达要求、交付、证据、判定和动作。

改了什么、影响入口、如何验证、脚本 / 重启、未覆盖点默认纵向排列；除非用户明确要求横向总控线，否则不要把这五项横向铺满首屏。总控轴每项一句结论，细节放
R# 行、泳道、风险动作列或证据索引。

交付验收样例优先使用克制的产品式验收账本或路线牌，避免重装饰 SVG 图纸感、发光渐变和密集彩色线路；背景用统一白底工程网格，以留白、字号、细线、状态章和纵向读序建层级。

## 交付验收图专用骨架

交付验收图必须长成需求到证据的验收轨道；可用验收账本、证据泳道、风险动作板或交付时间线，但每个候选面板都要有主轴、清晰读序和证据绑定。推荐主图语法：

`验收账本：原始要求 → 交付变更 → 验证证据 → 验收判定 → 待执行 / 回滚`

桌面布局建议：

1. **验收账本 / 需求到证据签收表**：每行一条 R#，列为要求、交付、证据、判定、动作 / 回滚。
2. **证据泳道图**：按证据来源或验证命令分泳道，每条泳道连回 R#。
3. **风险动作板**：按 `通过 / 待观察 / 需要用户动作 / 回滚` 分列。
4. **交付时间线**：按用户反馈、规则变更、测试验证、同步扩散、现场待观察组织。

可读性门禁：

- 保留候选按钮，但按钮不是信息主体；每个面板打开后必须不用返回其他面板也能回答“能不能验收”。
- 每个面板必须直接展示 `panel-summary` 式主结论和 `reading-order` 式阅读顺序，避免读者猜先看哪里。
- 每个 R# 默认只出现一次作为主锚点；同一条证据可重复引用，但必须标明它证明的 R#。
- 状态章必须固定在判定列、泳道尾部、看板列头或时间线终点；不要散落在每个小单元里。
- 长证据允许换行、增高行高或进入点击详情，但命令名、结果摘要和它证明的 R# 必须静态可见。

移动端把每条 R# 折叠成纵向小轨道，保持 `要求 → 交付 → 证据 → 判定 → 动作` 顺序；不得依赖横向滚动读完验收。

## 反直觉版式红线

- 不得把“验证闭环”“代码影响点”“保留与回滚”拆成彼此等权重的底部卡片区；这些信息必须绑定到 R# 验收轨道。
- 不得把“改了什么 / 影响入口 / 如何验证 / 脚本重启 / 未覆盖点”做成并列卡片墙；它们必须是主验收看板的抬头信息带或左侧控制轨。
- 不得用顶部状态 pill 代替验收轨道；状态 pill 只能做总览图例。
- 不得把 before/after 两列当成交付验收图主体，除非每条差异都继续映射到证据和判定。
- 如果读者需要在需求卡、代码卡、验证卡之间来回跳读才能判断某条需求是否通过，必须重画。
- 如果主图去掉文字后只剩一堆圆角卡片、表格行或状态徽章，必须重画为证据验收轨道。
- 如果用户说内部不清晰，优先重画面板内部结构或替换为新的布局类型；不得先删掉候选按钮。

## 状态与可访问性

- pass / warn / fail / blocked 必须同时使用文字标签和形状或图标；不能只靠颜色。
- `pass` 有直接证据；`warn` 需人工/环境确认；`fail` 证据不满足；`blocked` 缺外部条件。
- 节点正文必须可换行，状态标签不得遮挡证据文字。
- 原始命令输出、截图说明、长日志可放点击详情；但命令名、结果摘要和它证明的 R# 必须静态可见。

## 适用边界

- 小型 UI 返修：优先用 2-5 条 R# 验收轨道；before/after 截图差异只作为每条轨道里的“交付变更”节点。
- 功能迭代交付：将 AC、测试矩阵、发布门禁和回滚动作逐条绑定，避免把全量测试写成唯一验收证据。
- 缺陷修复交付：若还需要解释根因链路，应先画故障排查图；交付验收图只负责最终验收闭环。
- 方案评审或未实现前不要使用交付验收图；应切需求决策图或技术设计图。
