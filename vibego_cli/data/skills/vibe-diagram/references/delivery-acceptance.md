# vibe-diagram reference: delivery-acceptance

> 只有路由命中交付验收图后才读取；读取失败必须 fail-closed。


## 候选全集清单

- 生图类型：交付验收图
- 首选候选：需求到证据的验收轨道
- 必生成备选候选：
  - R# 泳道验收看板
  - 证据矩阵热区
  - 地铁站点式验收线路
- 校准期要求：当前图型命中后，HTML 内必须按候选 A/B/C/D…纵向展开首选候选与全部备选候选；每个候选都必须是真图。信息不足时也必须生成该备选候选，用“待确认节点”标明缺口。

## 交付验收图规则

交付验收图必须默认回答“原始要求是否逐条满足、证据是否足够、还剩什么动作”。它不是任务总结报告，也不是测试清单；它的主图要让读者沿着每条需求的证据链直接判断能否验收。

- 交付验收图必须覆盖：原始需求 / AC、交付变更、验证证据、验收判定、未覆盖项、风险、回滚和待执行动作。
- 结论必须写成验收口径，例如 `R1-R4 通过；R5 待人工截图确认；无阻塞项`。不能只写“已完成”“全量门禁通过”。
- 每一条用户需求或 AC 都必须拥有独立 R# 泳道；没有证据的需求只能标 `待确认`、`warn`、`blocked`，不能并入总通过。
- 验证命令、测试结果、截图核对和打包门禁必须贴在对应 R# 泳道内；全量门禁只能作为兜底证据，不能替代逐条需求证据。
- 风险、回滚和待执行动作必须从对应 R# 泳道引出；不要放成底部孤立卡片。
- 如果用户明确讨厌或禁止某个旧模式，必须在对应 R# 泳道放“旧模式反查”节点或角标。

## 交付验收必答信息

交付验收图除 R# 验收轨道外，还必须在首屏或紧邻主轨道的固定信息区直接外显以下问题的答案，不得只在最终聊天收尾字段里说明：

1. **改了什么**：列出源码、测试、文档、配置或规约的实际变更；只写“优化/修复”不合格。
2. **影响哪些功能入口**：明确用户会在哪些页面、命令、菜单、Bot 指令、API、任务流或服务入口感知变化；若无业务入口影响，写“无用户入口变化，仅影响规约/内部生成规则”。
3. **用户需要执行什么脚本**：列出需要用户执行的命令或脚本；如果无需脚本或无需重启，也必须明确写“无需执行脚本”或“无需重启服务”。
4. **需要重启什么服务**：明确要重启 worker、Web 服务、桌面应用、Bot、队列消费者、开发服务器或无需重启；不得用“按需重启”代替具体对象。
5. **如何验证**：列出聚焦测试、全量测试、构建/打包、截图/人工核对、线上观察或回归路径；每条验证方式要说明验证了哪个 R# 或哪个入口。
6. **剩余未覆盖点**：明确哪些验证没有做、为什么没做、是否需要用户补验。

推荐在主图上方增加一条“交付说明栏”，字段固定为：`改了什么`、`影响功能入口`、`验证方式`、`待用户执行脚本 / 重启服务`、`剩余未覆盖点`
。该说明栏只做总览，细节仍必须绑定到 R# 轨道。

交付说明栏不能做成 5 张等权重卡片；必须收敛成一条横向或纵向连通的信息带，并用线、序号、轴线或分组括号说明这些信息共同服务同一次交付。旧图问题只能作为旁注或反例标记，不得占据首屏主视觉。主图第一视觉必须是验收线路、验收看板或证据矩阵热区，而不是解释卡片、说明卡片或总结卡片。

交付验收主画布不得再用矩形卡片节点承载每一步；即使矩形之间有箭头，也仍然是卡片线路图。优先使用地铁图 / 站点图 / 状态热区：用线表示同一需求或同一交付流，用圆点、菱形、终点章表示要求、变更、证据、判定和动作；文字只作为站点短标注或下方轻量索引。

改了什么、影响入口、如何验证、脚本 / 重启、未覆盖点默认纵向排列，形成交付总控轴；除非用户明确要求横向总控线，否则不要把这五项横向铺满首屏。总控轴文案必须比
R# 站点更短、更像导览：每项一句结论，细节放 R# 站点或索引。

交付验收样例优先使用克制的产品式验收账本或路线牌：像一张可签收的交付单，而不是炫技海报。避免重装饰 SVG 图纸感、发光渐变和密集彩色线路；背景必须遵循统一白底工程网格；首屏应靠留白、字号、细线、状态章和纵向读序建立层级。

## 交付验收图专用骨架

交付验收图必须长成需求到证据的验收轨道，而不是章节报告。推荐主图语法：

`需求验收轨道：原始要求 → 交付变更 → 验证证据 → 验收判定 → 待执行 / 回滚`

桌面布局建议：

1. **左侧 R# 需求泳道**：每行一条原始要求或 AC，保留用户原话或可验收改写。
2. **中间交付变更列**：写改动点、受影响文件/模块、保留项或未改项。
3. **证据门禁列**：写聚焦测试、全量测试、截图/人工核对、打包/发布门禁；证据必须指向它证明的需求。
4. **验收判定列**：用 `pass / warn / fail / blocked` 明示状态，并写一句为什么。
5. **右侧动作列**：只放该需求的剩余动作、回滚方式或用户需执行步骤。

移动端必须把每条 R# 泳道折叠成纵向小轨道，仍保持 `要求 → 交付 → 证据 → 判定 → 动作` 顺序；不得依赖横向滚动读完验收。

## 反直觉版式红线

- 不得把“验证闭环”“代码影响点”“保留与回滚”拆成彼此等权重的底部卡片区；这些信息必须绑定到 R# 验收轨道。
- 不得把“改了什么 / 影响入口 / 如何验证 / 脚本重启 / 未覆盖点”做成并列卡片墙；它们必须是主验收看板的抬头信息带或左侧控制轨。
- 不得用顶部状态 pill 代替验收轨道；状态 pill 只能做总览图例。
- 不得把 before/after 两列当成交付验收图主体，除非每条差异都继续映射到证据和判定。
- 如果读者需要在需求卡、代码卡、验证卡之间来回跳读才能判断某条需求是否通过，必须重画。
- 如果主图去掉文字后只剩一堆圆角卡片、表格行或状态徽章，必须重画为证据验收轨道。

## 状态与可访问性

- pass / warn / fail / blocked 必须同时使用文字标签和形状或图标；不能只靠颜色。
- `pass` 表示有直接证据满足原始要求；`warn` 表示实现完成但仍需人工或环境确认；`fail` 表示证据显示不满足；`blocked`
  表示缺少必要外部条件。
- 节点正文必须可换行，状态标签不得遮挡证据文字。
- 原始命令输出、截图说明、长日志可放点击详情；但命令名、结果摘要和它证明的 R# 必须静态可见。

## 适用边界

- 小型 UI 返修：优先用 2-5 条 R# 验收轨道；before/after 截图差异只作为每条轨道里的“交付变更”节点。
- 功能迭代交付：将 AC、测试矩阵、发布门禁和回滚动作逐条绑定，避免把全量测试写成唯一验收证据。
- 缺陷修复交付：若还需要解释根因链路，应先画故障排查图；交付验收图只负责最终验收闭环。
- 方案评审或未实现前不要使用交付验收图；应切需求决策图或技术设计图。
