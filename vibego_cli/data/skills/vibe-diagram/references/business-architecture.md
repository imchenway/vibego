## 业务架构图专用骨架

业务架构图 / 领域地图必须默认回答“业务能力如何被角色、对象和规则约束”，而不是回答“系统怎么部署”或“步骤怎么执行”。用户需要先看懂这个业务能力服务谁、解决什么、靠什么依据、在哪里止步、如何沉淀，而不是阅读一份章节化调研报告。

默认主骨架：

`参与方边界 → 业务能力层 → 业务对象关系 → 规则约束热区 → 服务结果 / 决策轴`

骨架执行规则：

1. **首屏主图必须是一张领域地图**：主图必须是一张首屏可读的领域地图；禁止把摘要、事实口径、能力、对象、规则、证据拆成章节化卡片报告。摘要最多一行，不能先用大段卡片占掉主图焦点。
2. **参与方边界要按业务角色画**：写清业务发起方、自有能力承接方、依据提供方、人工/外部边界和运营闭环方；上下游系统只能作为业务边界，不得画成部署组件或系统架构图。
3. **业务能力层写服务承诺**：能力节点写服务承诺、业务目标、归属方、输入输出；禁止把接口、表、类名、DDL 或工具名当作能力主体。
4. **业务对象必须业务化命名**
   ：业务对象必须业务化命名，例如“客服触点、咨询工单、业务场景、服务依据、人工接续、服务质量资产”；技术对象名只能作为点击详情或证据补充。若技术对象名比业务对象名更醒目，必须重写节点命名。
5. **对象关系必须有业务动词**：对象关系必须用动词显式连接：拥有、提出、承接、产生、约束、查询、交接、沉淀、反哺。没有关系动词就不能称为领域地图。
6. **规则约束热区必须贴对象**：规则约束热区必须贴近被约束的对象或能力，至少包含口径、阈值、权限、风险、状态边界或回滚条件；禁止把规则堆到底部当说明表。
7. **服务结果 / 决策轴只画业务裁决**：服务结果节点只表达可自助、需补要素、需人工、需回滚等业务结果；不要把技术返回码、接口字段或实现分支当作主轴。
8. **证据进入节点详情**：证据、文件路径、源码锚点、DDL 和长说明进入节点点击详情；主图只保留业务结论、关系和必要证据编号。
9. **流程优先级裁决**：如果图的核心变成动作先后、审批流、异常回退，应切换为业务流程图；不要把流程塞进业务架构图。

反例门禁：

- 如果去掉标题后只剩“摘要卡片 + 事实卡片 + 能力卡片 + 对象卡片 + 规则卡片”，必须重画。
- 如果主图需要逐段阅读文字才能理解业务结构，而不是沿关系线和分层读懂，必须重画。
- 如果技术对象名比业务对象名更醒目，必须重写节点命名。
- 如果规则、风险和边界只出现在底部说明表，没有贴近被约束对象或能力，必须重画。

## 业务架构图规则

业务架构图 /
领域地图默认表达“业务由哪些参与方、能力、业务对象、规则和边界构成”，不是表达“操作先后顺序”，也不是表达系统部署拓扑。默认采用“参与方边界 →
业务能力层 → 业务对象关系 → 规则约束热区 → 服务结果 / 决策轴”的领域地图骨架。

- 参与方边界要区分业务发起方、自有能力承接方、依据提供方、人工/外部边界和运营闭环方；上下游系统、人工角色、第三方服务要画出业务边界，不要混在同一节点里。
- 每个能力节点写服务承诺、业务目标、归属方、输入输出；不要把接口、表、类名、DDL 或工具名当作能力主体。
- 领域对象要业务化命名并标出核心关系：拥有、提出、承接、产生、约束、查询、交接、沉淀、反哺等；技术对象名只进点击详情或证据补充。
- 业务规则必须贴近被约束的对象或能力，显式标注口径、阈值、权限、风险、状态边界和回滚条件；不能只写“按规则处理”，也不能堆到底部当规则表。
- 服务结果 / 决策轴只画业务裁决：可自助、需补要素、需人工、需回滚等；不要把技术返回码、接口字段或实现分支当主轴。
- 文件路径、源码锚点、DDL、接口清单和长证据默认进入对应节点点击详情；主图保留业务结论、关系和必要证据编号。
- 如果出现时间顺序、审批先后、异常回退，应切到业务流程图，不要在业务架构图里画长流程。
- 如果第一眼只能看到摘要、事实、能力、对象、规则的章节化卡片报告，看不出业务对象如何互相约束，必须重画。
