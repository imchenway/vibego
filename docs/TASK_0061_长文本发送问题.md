# /TASK_0061 长文本发送问题（PLAN）

## 1. 背景（现状）

用户在 Telegram 中直接发送一段**超长文本**后，Telegram 收到提示：

> 未检测到 Codex 会话日志，请稍后重试。

但在 tmux 内的 Codex CLI 可以看到提示词已被推送（且长文本被自动落盘为附件路径并提示“请阅读附件获取全文”）。

**用户期望**：无论消息长短，Telegram 的交互都应与普通消息一致——先收到一次：

> 💭 codex 思考中，正在持续监听……

并持续回传模型输出。

## 2. 证据（可验证）

以项目 `hyphamall` 为例（路径均来自本机）：

1) 会话指针文件为空（导致无法绑定 session，进而无法发送 💭 ack 与监听输出）：

- `~/.config/vibego/logs/codex/hyphamall/current_session.txt`（0 字节）

2) `session_binder` 在 worker 启动后 10 分钟超时退出（之后不会再写入指针文件）：

- `~/.config/vibego/logs/codex/hyphamall/session_binder.log`
  - `2026-02-05 13:51:02` start
  - `2026-02-05 14:01:02` timeout reached

3) 同一天确实存在符合 `cwd=/Users/david/hypha/mall` 的 Codex session JSONL（证明会话日志存在，只是 vibego 没绑定上）：

- `~/.codex/sessions/2026/02/05/rollout-2026-02-05T15-37-52-019c2cbc-6eed-78e2-a5c1-29f44f8808c7.jsonl`

## 3. 根因分析（高置信度）

### 3.1 直接原因

- vibego 依赖 `current_session.txt` 指针文件定位会话；
- 但 `session_binder.py` 在默认 `SESSION_BINDER_TIMEOUT=600` 秒后退出；
- 若用户在 **binder 超时退出之后** 才第一次触发 Codex 产生会话文件，则指针文件永远为空；
- `bot.py` 在 `SESSION_BIND_STRICT=1` 下不会自动回退扫描最新会话，因此直接回：
  - “未检测到 Codex 会话日志，请稍后重试。”

### 3.2 触发条件（常见）

- worker 启动后较长时间无人发消息（>10 分钟），Codex CLI 可能尚未生成 session JSONL；
- 用户首次发消息（尤其是长文本）发生在 10 分钟后；
- binder 已退出 → 指针无法写入 → strict 绑定失败。

## 4. 目标与验收标准（AC）

1) 无论消息长短，只要已成功推送到 tmux，Telegram 必须先返回一次 💭（不应再以“未检测到会话日志”作为常态结果）。  
2) 首次交互发生在 worker 启动 **10 分钟之后**，仍可正常绑定 session 并持续回传输出。  
3) stop/restart 时不会遗留僵尸 binder 进程造成资源浪费或错误绑定。  

## 5. 决策记录（已确认）

用户确认采用 **A + B**：

- A：延长/取消 `session_binder` 超时（避免指针文件长期为空）
- B：bot 侧增加 strict 模式下的兜底回退（指针缺失时可扫描最新 session 并自写指针）

## 6. Design（开发设计 + 用例设计）

> 说明：此处仅输出设计与用例，代码实现进入 develop 阶段后执行。

### 6.1 定位现状（技术视角）

- 启动链路：
  - `scripts/run_bot.sh` → `scripts/start_tmux_codex.sh` → `nohup scripts/session_binder.py &`
- 指针文件：
  - `CODEX_SESSION_FILE_PATH`（指向 `current_session.txt`）
- 会话绑定：
  - `bot.py::_dispatch_prompt_to_model()`
    - strict 下仅等待指针出现（`_await_session_path()` 只读 pointer），超时则报错

### 6.2 方案 A：binder 常驻等待（推荐默认）

1) 将 `SESSION_BINDER_TIMEOUT` 默认值从 `600` 调整为：
   - `0`（表示一直等待，直到绑定成功）
2) 增加 binder 生命周期管理：
   - `scripts/start_tmux_codex.sh`：写入 `session_binder.pid`，重启前先 kill 旧 PID
   - `scripts/stop_bot.sh`：按项目 log_dir 定位并 kill `session_binder.pid`

影响范围：

- `scripts/run_bot.sh`
- `scripts/start_tmux_codex.sh`
- `scripts/stop_bot.sh`
- （可选）`scripts/session_binder.py`：保持 `timeout=0` 即无限等待的语义不变

### 6.3 方案 B：bot 侧 strict 兜底回退（安全网）

当 strict 等待 pointer 超时后，按以下顺序兜底：

1) 直接扫描会话根目录，按 `cwd == MODEL_WORKDIR` 过滤并选择最新会话：
   - 复用 `bot.py::_find_latest_rollout_for_cwd()` / Gemini 分支逻辑
2) 若找到候选会话：
   - 自写 `current_session.txt` 指针文件
   - 继续正常 ack（💭）与 watcher 监听逻辑
3) 若仍找不到：
   - 返回更可操作的错误信息（包含：指针路径、期望 cwd、建议检查 tmux/codex 是否正常运行）

影响范围：

- `bot.py`（`_dispatch_prompt_to_model` / `_await_session_path` 相关逻辑）

### 6.4 用例设计 Checklist（像测试一样）

1) **指针为空 + strict=1 + 存在匹配 cwd 的 session 文件**
   - 断言：不返回“未检测到会话日志”
   - 断言：会写入 `current_session.txt`
   - 断言：会发送一次 💭 ack
2) **指针为空 + strict=1 + 不存在匹配 cwd 的 session 文件**
   - 断言：在等待超时后返回“可操作的错误信息”
3) **stop/restart 清理**
   - 断言：stop 单项目时会 kill 对应 `session_binder.pid`
   - 断言：反复 restart 不会产生多个 binder 常驻进程

## 8. 实施结果（develop）✅

### 8.1 变更点

**A. binder 常驻等待（默认不再 10 分钟超时）**

- `scripts/run_bot.sh`
  - `SESSION_BINDER_TIMEOUT` 默认值从 `600` 调整为 `0`（一直等待直到绑定成功）
- `scripts/start_tmux_codex.sh`
  - `SESSION_BINDER_TIMEOUT` 默认值从 `600` 调整为 `0`
  - 新增 `SESSION_BINDER_PID_FILE`（默认 `<log_dir>/session_binder.pid`）
  - 启动前会按 `pointer` 路径清理历史 binder（含升级前未写 pid 的 orphan 进程）
  - 启动后写入 binder PID，便于 stop/restart 清理
- `scripts/stop_bot.sh`
  - 停止项目时会 kill 对应 `session_binder.pid` 指向的进程
  - `stop_all_workers` 兜底扫描 `session_binder.pid`，避免 bot.pid 缺失时残留 binder

**B. bot strict 兜底回退**

- `bot.py`
  - 新增 `_fallback_locate_latest_session()`：pointer 未写入时扫描会话目录定位最新 session
  - `_dispatch_prompt_to_model()` / `_ensure_session_watcher()` 在 `SESSION_BIND_STRICT=1` 且等待 pointer 失败时启用兜底
  - strict 兜底仍失败时，错误提示会带 `pointer=` / `cwd=` 便于排查
- `tests/test_task_description.py`
  - 新增用例覆盖 strict 模式下「pointer 为空但可扫描到 session」与「扫描不到 session 返回带细节错误」两种路径
