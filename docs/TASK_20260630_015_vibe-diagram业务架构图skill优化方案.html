需求决策：业务架构图 skill 从卡片堆叠改为领域地图

目标：让“业务架构图”第一眼看出参与方、能力分层、领域对象关系与规则约束，而不是把调研文字分块贴成卡片。

诊断结论

这份外部项目 HTML 虽然标题叫“业务架构”，但主阅读成本仍来自逐块阅读：摘要、事实口径、参与方、能力层、对象关系、规则矩阵都像独立章节；图形语法没有形成一个可一眼读取的业务结构。它比纯 Markdown 好，但仍不是“业务架构图”。

当前失败模式

问题 1

章节化堆叠

参与方、能力、对象、规则被分成多个区块；用户要上下跳读才能拼出整体。

问题 2

对象关系太弱

领域对象之间虽然有箭头，但没有成为主图骨架；对象只是另一组卡片。

问题 3

规则没有约束对象

风险、优先级、接入状态没有贴到对应能力或对象上，像底部说明。

证据：源 HTML /Users/david/cckg/tcapp/Back-End/css/docs/TASK_20260630_002_业务架构图.html 中存在 20 个 article、22 个 h3、22 个 evidence，结构上更接近“卡片型报告”。

推荐的业务架构图应该长这样

参与方边界

客户/门店

提出业务诉求；只保留角色、诉求、触点。

内部业务能力

承接诉求；不是系统模块清单。

业务支撑方

提供知识、数据、规则和人工承接。

外部/人工边界

明确何时流出自研能力。

能力分层

入口能力

统一触点、场景露出、身份上下文。

识别能力

意图、槽位、风险、置信度。

履约能力

查询、规则计算、SOP、反馈。

接续能力

转人工、上下文交接、闭环评测。

领域对象关系

App / 用户

拥有会话与可见场景。

Session / Message

产生意图、槽位、卡片与 trace。

Scene / Tool / Knowledge

约束回答来源与风险等级。

Handoff / Feedback

沉淀结果，反哺质量。

低风险
可自助回答；来源可追溯。
中风险
必须有业务数据、权限和槽位。
高风险
默认转人工或只给解释，不直接承诺结果。

三个优化方案

方案 内容 优点 风险
A 最小补丁 只强化业务架构图规则文字，增加“禁止卡片报告”。 改动小。 模型仍可能照旧分章节堆卡片。
B 推荐：领域地图骨架门禁 新增业务架构专用骨架：参与方边界 → 能力分层 → 领域对象关系 → 规则热区；并要求首屏一张主领域地图。 能直接针对这次失败模式，约束图必须有主谓宾与对象关系。 需要补测试和同步 AGENTS。
C 强模板化 除 B 外，再提供固定 HTML 模板片段和反例库。 执行最稳定。 可能过度模板化，导致不同业务都长得太像。

我建议按 B 实施

  • 在 vibe-diagram/SKILL.md 新增“业务架构图专用骨架”。
  • 在测试中加入压力断言：必须包含“参与方边界、能力分层、领域对象关系、规则约束热区、首屏主领域地图、禁止章节化卡片报告”。
  • 同步到 AGENTS 后，后续任何触发“业务架构图 / 领域地图”的制图都会默认走这套骨架。
  • 不把业务架构图误改成业务流程图；它仍然回答“业务由哪些角色、能力、对象、规则组成”，不是回答“先做什么后做什么”。
