# /TASK_0059 优化按钮体验（PLAN）

## 1. 背景（现状）

用户在“模型答案消息”底部点击 `任务状态更新为测试中` 后，目前只会收到一条确认提示：

- `任务 /TASK_0068 状态已更新为“测试”。`

用户希望：**点击该按钮后，最后自动触发一次“查询任务列表”**，减少一次额外操作（例如再点击菜单里的任务列表）。

> 参考截图：`20260131_100018418-3ec742762ac7.jpg`

## 2. 目标（用户视角）

当用户点击“更新为测试中”按钮后：

1) 明确告知状态已更新；并且  
2) 自动展示任务列表（等同于用户自己触发一次任务列表查询）。

## 3. 边界与异常（待确认）

1) 自动展示的“任务列表”应使用哪种筛选？
2) 自动展示的“任务列表”应以什么形式出现？
3) 该行为是否只针对“更新为测试中”按钮，还是也适用于其他状态更新按钮？

## 4. 验收标准（AC）

1) 点击 `任务状态更新为测试中` 后，用户能看到“已更新为测试中”的确认提示。  
2) 在确认提示之后，自动展示一份任务列表（无需用户再点菜单/再输入命令）。  
3) 自动展示任务列表时，不应引入重复刷屏或明显的交互跳闪（例如连续弹出多条无意义提示）。  
4) 若任务不存在/权限不足/网络失败，应给出明确可理解的错误提示，并保持用户可继续操作（至少不阻塞后续对话）。  

## 5. 待决策项（请回复编号+选项）

1) **自动展示的任务列表筛选范围**
   - A（推荐） 展示“全部（默认）”任务列表（与常规入口一致）
   - B 展示“测试中”筛选后的任务列表
   - C 展示用户“上一次浏览过的列表筛选条件”（如果存在）
   - D 不做自动展示（仅保留提示）

2) **任务列表的展示形式**
   - A（推荐） 在确认提示之后，再发送一条“任务列表”消息（最直观、最稳）
   - B 尝试复用/更新当前消息为任务列表（减少消息条数，但可能影响阅读）
   - C 仅追加一个“查看任务列表”的按钮（仍需用户点击一次）
   - D 其他（请说明）

3) **生效范围**
   - A（推荐） 仅对“更新为测试中”这个按钮生效
   - B 对所有“状态更新按钮”生效
   - C 仅对“模型答案消息”中的状态按钮生效（不影响任务详情页内的状态按钮）
   - D 其他（请说明）

## 7. 决策结果（用户确认）

> 用户指令：“待决策项全部按模型推荐”

1) 自动展示任务列表筛选范围：**1A（全部/默认）**  
2) 任务列表展示形式：**2A（确认提示后再发送一条任务列表消息）**  
3) 生效范围：**3A（仅对“更新为测试中”按钮生效）**  

## 8. design（开发设计 + 用例）

### 8.1 现状定位（技术视角）

- 入口：`bot.py` → `on_model_task_to_test(...)`（回调：`model:task_to_test:<TASK_ID>`）
- 任务列表渲染：`bot.py` → `_handle_task_list_request(...)` → `_build_task_list_view(...)`

### 8.2 方案（按已选决策落地）

在 `on_model_task_to_test` 内：

1) 先完成状态更新并提示：`任务 /TASK_XXXX 状态已更新为“测试”。`
2) 随后调用 `_handle_task_list_request(callback.message)`，自动发送默认任务列表视图（等价于用户手动触发一次任务列表）。

### 8.3 用例 Checklist（像测试一样）

1) 任务从非 test → test
   - 点击按钮：回调返回“已切换到测试”
   - 消息 1：状态更新提示
   - 消息 2：自动展示任务列表（含“搜索/创建”按钮）
2) 任务已处于 test
   - 点击按钮：仅提示“任务已处于测试”，不额外刷任务列表
3) 任务不存在 / task_id 无效
   - 仅提示错误（show_alert），不发送任务列表
