# TASK_20260630_001_vibe-diagram 故障排查与功能迭代重点设计

## 1. 结论先行

用户判断是对的：`故障排查` 与 `功能迭代 / 开发设计` 是 `vibe-diagram` 里最需要重点设计的两类图。

原因不是它们更“复杂”，而是它们承担的用户任务最重：

- 用户不是只想看结果，而是要理解**现状到底是什么**。
- 用户要知道**为什么出问题 / 为什么要改**。
- 用户要判断**在当前代码设计和功能设计上如何安全修复或迭代**。
- 用户最后要看到**验证闭环、风险与回滚**，否则无法相信结论。

因此，这两类图不应只继承通用图形语法，而应单独定义专用模板。

## 2. 当前不足

现有 `vibe-diagram` 已有：

- 全图型形态契约。
- 布局、箭头与防重叠算法门禁。
- 故障排查图规则。
- 根因、修法、前后对照规则。
- 流程图语法门禁。

但对“故障排查”和“功能迭代”的用户任务仍不够明确：

1. **现状层不够重**：容易直接跳到根因或目标方案，用户看不清当前代码/功能到底怎么走。
2. **证据层和流程层容易分离**：证据表、假设、修法各自漂浮，主路径仍不够直观。
3. **功能迭代缺少差异语法**：新增/修改/删除/不变/风险/回滚没有稳定落在 before/after 或 current/target 主路径上。
4. **没有专用骨架名**：模型容易把“故障排查图”和“功能迭代图”继续画成泛流程图或文字说明图。

## 3. 推荐设计：双专用模板

### 3.1 故障排查图：`现象 → 现状链路 → 证据裁决 → 根因 → 修法 → 验证/回滚`

故障排查图的核心不是“列出根因”，而是让用户看懂一条线：**现象如何沿着当前代码/系统设计一路走到问题点，证据如何裁决假设，修法如何切断问题链路。
**

建议固定为 6 段：

1. **现象 / 影响**：用户看到什么，影响范围是什么。
2. **现状链路**：当前代码、接口、状态、配置或运行链路如何工作。
3. **证据裁决**：E1/E2/E3 贴在现状链路节点上，裁决成立/排除/待验证假设。
4. **根因节点**：必须落在现状链路上，标 `根因 R#` 或 `最高可疑点 H#`。
5. **修法节点**：必须落在“切断/替换/补偿/兜底”动作上，标 `修法 F#`。
6. **验证 / 回滚闭环**：验证命令、结果状态、回滚动作、剩余风险。

推荐主图形态：

- 小故障：单图因果链，高亮 R/F/V。
- 链路故障：代码时序排障图，参与者列 + 时间向下，证据贴在调用行。
- 逻辑修复影响两处以上：before/after 流程化对照，左/上故障逻辑，右/下修复逻辑。

### 3.2 功能迭代 / 开发设计图：`当前体验与实现 → 目标体验与实现 → 差异映射 → 验证与发布`

功能迭代图的核心不是“展示新方案”，而是让用户看懂：**当前功能和代码设计是什么，目标要改变哪里，哪些不变，新增/修改/删除如何映射到接口、DB、模块、权限、测试和回滚。
**

建议固定为 5 段：

1. **当前功能主路径**：用户现在怎么用，系统现在怎么处理。
2. **当前开发实现链路**：前端入口、接口、服务、DB/状态、中间件/异步任务。
3. **目标主路径**：用户改完后怎么用，关键体验差异是什么。
4. **差异映射**：新增 / 修改 / 删除 / 不变 / 风险 / 回滚分别落到对应节点。
5. **验收闭环**：AC、测试矩阵、灰度发布、监控、回滚脚本或人工动作。

推荐主图形态：

- 体验变更：before/after 用户主路径对照。
- 技术链路变更：current/target 技术时序或数据流对照。
- 大迭代：三层卷轴：用户体验层 → 系统实现层 → 验证发布层。

## 4. 方案选项

### A. 只在现有规则里补几句强调

优点：改动最小。

缺点：仍容易被模型理解成普通说明文字；不能稳定改变生图形态。

适用：只想轻微补丁。

### B. 新增两类专用模板规则（推荐）

优点：明确故障排查和功能迭代各自的固定骨架、信息层级、箭头和验证闭环；成本可控。

缺点：需要新增测试锁定文案与 AGENTS 注入。

适用：当前最匹配。

### C. 新增专用模板规则 + 示例 HTML 样板

优点：不只告诉模型怎么画，还给未来模型可参考的结构样板；对视觉稳定性更强。

缺点：文件和测试更多，维护成本更高；如果样板质量不够，反而会固化错误示例。

适用：B 落地后，如果仍出现图形质量漂移，再追加。

### 8.3 实施后的规则摘要

故障排查图固定骨架：

`现象 / 影响 → 当前现状链路 → 证据裁决 → 根因或最高可疑点 → 修法 → 验证 / 回滚`

功能迭代 / 开发设计图固定骨架：

`当前功能主路径 → 当前开发实现链路 → 目标主路径 → 差异映射 → 验证与发布闭环`

两类图共同原则：必须先讲清当前现状，再进入修法或目标方案；证据、差异、测试、风险、回滚必须贴到主路径节点，不能漂在底部说明区。
