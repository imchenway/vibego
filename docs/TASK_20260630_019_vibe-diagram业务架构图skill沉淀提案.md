# TASK_20260630_019 vibe-diagram 业务架构图 skill 沉淀提案

## 核心结论

业务架构图不是系统架构图，也不是业务流程图，更不是调研报告。它应该回答：业务由哪些参与方、能力、业务对象、规则和边界构成；用户沿业务主轴看一遍，就能知道这个业务能力“服务谁、解决什么、靠什么依据、在哪里止步、如何沉淀”。

## 建议写入 skill 的默认骨架

`参与方边界 → 业务能力层 → 业务对象关系 → 规则约束热区 → 服务结果 / 决策轴`

## 建议规则

1. 主图必须是一张首屏可读的领域地图；禁止拆成章节化卡片报告。
2. 参与方边界写清业务发起方、自有能力承接方、依据提供方、人工/外部边界和运营闭环方。
3. 业务能力层写服务承诺、业务目标、归属方、输入输出；禁止接口、表、类名、DDL 成为主体。
4. 业务对象必须业务化命名，如客服触点、咨询工单、业务场景、服务依据、人工接续、服务质量资产。
5. 对象关系必须用拥有、提出、承接、产生、约束、查询、交接、沉淀、反哺等动词连接。
6. 规则约束热区必须贴近被约束对象或能力，禁止堆到底部当说明表。
7. 证据、文件路径、源码锚点、DDL 和长说明进入节点点击详情。
8. 如果核心变成动作先后、审批流、异常回退，应切换为业务流程图。
