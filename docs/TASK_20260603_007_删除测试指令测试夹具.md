# TASK_20260603_007 删除测试夹具里的“测试指令”

## 1. 任务结论

本次只调整 `tests/test_plan_progress.py` 中 `test_session_ack_message_silent` 的测试输入，把真实中文短语 `测试指令` 替换为明显的测试哨兵：

```text
__VIBEGO_DELIVERY_CONFIRMATION_TEST_PROMPT__
```

不修改 `bot.py`，不新增 Telegram 文本拦截，不新增 `/ping`，不改变生产发送链路。

## 2. 现状与根因

| 项目 | 说明 | 证据 |
| --- | --- | --- |
| 目标测试文件 | `tests/test_plan_progress.py` 覆盖模型回复投递、ack 静默投递、pending offset 等行为。 | `tests/test_plan_progress.py`（锚点：`test_session_ack_message_silent`、`_deliver_pending_messages`） |
| 根因 | `test_session_ack_message_silent` 的 `DummyMessage` 和手写 JSONL 使用真实自然语言短语，容易在会话、搜索、排障时被误认为产品逻辑主动发送的真实 prompt。 | `tests/test_plan_progress.py`（锚点：`DummyMessage("测试指令")`，修改前证据；当前已替换为哨兵） |
| 生产链路边界 | Telegram 普通消息与模型投递逻辑位于 `bot.py`，本任务计划明确不修改生产链路。 | `bot.py`（锚点：`_handle_prompt_dispatch`、`_deliver_pending_messages`） |

## 3. 需求口径

### 3.1 用户/场景

- 使用者：通过搜索测试、JSONL、会话日志排查 vibego Telegram 投递链路的开发者。
- 使用时刻：排查“是否有真实 prompt 被主动发送”时。
- 业务目标：让测试夹具与真实用户输入明显区分，降低误判成本。

### 3.2 核心规则

1. 真实自然语言短语不得继续作为该测试用例的假 prompt。
2. `DummyMessage.text` 与模拟 Codex JSONL 中的 `input_text.text` 必须保持一致。
3. 测试语义仍是“模型已消费本次输入，因此 ack 静默投递不落入未确认失败分支”。
4. 不修改生产发送链路，不拦截真实用户消息。

### 3.3 验收标准

| AC | 验收点 | 验证方式 |
| --- | --- | --- |
| AC-01 | `tests/test_plan_progress.py` 不再包含 `测试指令`。 | `rg -n "测试指令" tests/test_plan_progress.py` 无结果（exit code 1） |
| AC-02 | `DummyMessage` 使用哨兵 prompt。 | `tests/test_plan_progress.py`（锚点：`__VIBEGO_DELIVERY_CONFIRMATION_TEST_PROMPT__`） |
| AC-03 | 手写 JSONL 的 `input_text.text` 与 `DummyMessage.text` 使用同一哨兵值。 | `tests/test_plan_progress.py`（锚点：`__VIBEGO_DELIVERY_CONFIRMATION_TEST_PROMPT__`） |
| AC-04 | 投递/ack/pending 聚焦测试仍通过。 | `python3.11 -m pytest -q tests/test_plan_progress.py -k 'ack or delivery or pending'` |
| AC-05 | `bot.py` 语法未受影响。 | `python3.11 -m py_compile bot.py` |
