# TASK_20260317_001_缺陷流程新增前置条件字段

## 1. 背景

用户要求在缺陷结构化录入中新增 **前置条件** 字段，并统一调整顺序为：

```text
前置条件 -> 复现步骤 -> 预期结果
```

同时要求以下入口与展示口径保持一致：

1. 任务详情里的「🚨 报告缺陷」创建缺陷任务流程
2. `/task_new` 交互式缺陷创建流程
3. `/task_new ... | type=缺陷` 命令式缺陷创建
4. Excel 批量创建缺陷模板与导入逻辑
5. 推送到终端/模型的提示词与上下文块
6. 任务详情中的缺陷结构化展示

### 关联历史文档

- `docs/TASK_0089_缺陷提交步骤改为复现步骤与期望结果.md`
- `docs/TASK_0096_通过Excel批量创建缺陷.md`

## 2. 现状证据

### 2.1 缺陷结构仍是双字段

- `bot.py`（锚点：`DEFECT_REPRODUCTION_LABEL`, `DEFECT_EXPECTED_RESULT_LABEL`, `_build_defect_description`, `_parse_defect_description`）
  - 现状仅存储：`复现步骤 / 期望结果`
- `bot.py`（锚点：`_append_task_prompt_description_fields`, `_format_task_detail`）
  - 缺陷详情与推送提示词也只拆成双字段

### 2.2 缺陷创建流程仍未采集前置条件

- `tasks/fsm.py`（锚点：`TaskDefectReportStates`, `TaskCreateStates`）
  - 现状缺少 `waiting_precondition`
- `bot.py`（锚点：`on_task_defect_report_title`, `on_task_create_reproduction`）
  - 任务详情缺陷创建和 `/task_new` 缺陷创建都直接进入复现步骤

### 2.3 Excel 模板仍是旧列头

- `bot.py`（锚点：`DEFECT_EXCEL_TEMPLATE_HEADERS`, `_build_defect_excel_template_bytes`, `_parse_defect_excel_import_file`）
  - 模板列头仍为：`缺陷标题 / 复现步骤 / 期望结果 / 关联任务编码 / 优先级`

---

## 3. vibe 结论（用户视角）

### 3.1 现状

当前缺陷信息主要是双字段：

```text
复现步骤 -> 期望结果
```

问题是：

1. 缺少“问题发生前需要满足什么条件”的上下文
2. 批量导入模板与手动创建口径不一致时，容易造成字段语义丢失
3. 推送给终端/模型的提示词无法完整表达触发场景

### 3.2 本次目标

统一改为：

```text
前置条件 -> 复现步骤 -> 预期结果
```

### 3.3 验收口径

1. 手动创建缺陷时，标题后先采集前置条件
2. 前置条件为可选，与复现步骤/预期结果一样支持“跳过”与附件
3. Excel 模板新增“前置条件”列，导入结果按同一顺序组装正文
4. 缺陷任务详情与推送提示词都展示三字段
5. 历史双字段缺陷正文要兼容展示，默认前置条件显示 `-`

---

## 4. design 方案

## 4.1 推荐方案（已实施）

继续复用 `TaskRecord.description`，但把缺陷正文升级为三字段结构化文本：

```text
前置条件：
<内容或 ->

复现步骤：
<内容或 ->

预期结果：
<内容或 ->
```

### 这样设计的原因

✅ 不需要数据库迁移  
✅ 与既有结构化缺陷正文兼容  
✅ 任务详情 / Excel / 推送提示词都能统一复用同一份正文  
✅ 可以兼容历史双字段正文，通过 parser 自动补齐 `前置条件：-`

## 8. 最终结果

### 用户可见变化

1. 缺陷创建流程新增“前置条件”输入
2. 新顺序统一为：`前置条件 -> 复现步骤 -> 预期结果`
3. Excel 批量创建缺陷模板新增“前置条件”列
4. 缺陷详情展示新增“前置条件”
5. 推送到终端/模型的提示词也新增“前置条件”
6. 历史旧双字段缺陷任务不会丢失，自动兼容为 `前置条件：-`

### 未改动边界

- 优化任务仍保持 `当前效果 / 期望效果` 双字段
- 数据库存储模型未改 schema
- 历史自由文本 defect 任务仍按普通描述回退展示
