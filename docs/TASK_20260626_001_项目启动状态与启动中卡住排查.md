# TASK_20260626_001 项目启动状态与启动中卡住排查

## 1. 任务背景

用户在 2026-06-26 08:51-08:52 通过 Telegram 截图反馈两个问题：

1. 项目列表里 `WMS2` 显示为已启动，可操作按钮为 `⛔️ 停止 (⚙️ Codex)`，但实际模型 tmux 主会话已不存在。
2. 点击启动项目时经常卡在 `⏳ 启动中/停止 (⚙️ Codex)`，截图示例为 `Vibego`。

本轮先做只读取证与方案确认，未修改业务源码。

> 重要现场说明：排查中误调用了 `master.bootstrap_manager()`；该函数按设计会执行启动清理（`master.py` 锚点：`bootstrap_manager`、`_clear_related_tmux_sessions`、`manager.stop_all(update_state=True)`），导致当时存活的 tmux 会话被清理。该动作已经影响现场，后续如需恢复运行态，应由用户确认后重新启动对应项目。

## 2. 现象 -> 影响 -> 根因候选

### 2.1 现象 A：WMS2 列表显示已启动，但 tmux 主会话缺失

- 截图证据：`/Users/david/Library/Containers/com.wiheads.paste/Data/tmp/images/Telegram 2026-06-26 08.51.25.png`，按钮显示 `WMS2 | ⛔️ 停止 (⚙️ Codex)`。
- 现场证据（误清理前已取）：`tmux ls` 仅有 `vibe-vibego`、`vibe-wms`，没有 `vibe-wms2`。
- 现场证据（误清理前已取）：`/Users/david/.config/vibego/logs/codex/wms2/bot.pid` 指向的 worker pid `30550` 当时仍存活，但对应模型 tmux 会话不存在。
- 代码证据：`master.py` 锚点 `reconcile_worker_states` 只检查 `_worker_process_alive` 与握手日志；`master.py` 锚点 `_projects_overview` 直接按 `state.status == "running"` 渲染停止按钮。

影响：Master 项目列表把“Telegram worker 进程还活着”误当成“项目可用”，但实际 Codex tmux 主会话已经丢失，继续点击项目入口会进入不可用会话或旧会话。

### 2.2 现象 B：点击启动后卡在启动中

- 截图证据：`/Users/david/Library/Containers/com.wiheads.paste/Data/tmp/images/Telegram 2026-06-26 08.52.43.png`，按钮显示 `Vibego | ⏳ 启动中/停止 (⚙️ Codex)`。
- 日志证据：`/Users/david/.config/vibego/logs/vibe.log` 中 `2026-06-26 08:51:35` 启动 `Vibego`，`2026-06-26 08:52:05` worker 记录 `Telegram 连通性检查失败：在 30.0 秒内未能与 Telegram 成功握手`。
- 日志证据：同一段异常路径中 `master.py` 行号 4292 对用户发送 `操作失败: {exc}`，随后 Telegram 返回 `Bad Request: message is too long`。
- 代码证据：`master.py` 锚点 `_health_check_worker` 会把 `run_bot.log` 尾部拼入异常；`master.py` 锚点 `on_project_action` 的 `except` 分支直接发送完整异常文本，发送失败后无法继续刷新项目概览。

影响：启动失败本应刷新项目列表并展示最新状态，但失败详情过长导致 Telegram 拒绝消息，异常打断 `_refresh_project_overview`，旧按钮停留在 `启动中`。

## 3. 已确认根因

### R1：运行态健康口径过窄

当前 Master 的健康口径主要是 `bot.pid + 握手日志`：

- `master.py` 锚点 `_worker_process_alive`：只判断 `bot.pid` 指向的进程是否存活。
- `master.py` 锚点 `reconcile_worker_states`：当进程不活时直接 `continue`，没有把旧 `running/starting` 降级；当进程活但 tmux 主会话不存在时，也不会降级。
- `master.py` 锚点 `run_worker`：若状态是 `running/starting` 且 worker pid 存活，会直接返回当前模型，不会检查 tmux 主会话。

因此出现两类假状态：

1. worker pid 活着，但模型 tmux 已丢：仍显示 running。
2. state 里保留 running/starting，但 pid 已死或缺失：刷新项目列表时也可能继续保留旧状态。

### R2：失败消息不受 Telegram 长度约束保护

- `_health_check_worker` 会拼接最近日志用于诊断，利于排障但可能超过 Telegram 单条消息上限。
- `on_project_action` 的异常分支直接 `await callback.message.answer(f"操作失败: {exc}")`。
- 一旦这条错误提示自身发送失败，后面的 callback alert 与项目列表刷新都不会执行。

## 4. 推荐修法

### 4.1 运行态健康改为“worker + tmux + handshake”三元口径

新增内部健康判定，建议命名：

- `_expected_tmux_session_name(cfg)`：按 `TMUX_SESSION_PREFIX` 与 `cfg.project_slug` 计算主会话名，需与 `scripts/models/common.sh` 锚点 `tmux_session_for` 语义一致。
- `_tmux_session_alive(session_name)`：读取 `tmux list-sessions` 或 `tmux has-session`。
- `_worker_runtime_health(cfg, model)`：返回 `pid_alive / tmux_alive / handshake_ok / reason`。

状态刷新规则建议：

| 条件 | 新状态 | UI |
| --- | --- | --- |
| pid 不存在或已死 | `stopped` | `▶️ 启动` |
| pid 活着、tmux 主会话缺失 | `degraded` 或 `stopped` | 推荐 `⚠️ 修复/重启`，回调复用 `project:run` |
| pid 活着、tmux 活着、握手未完成 | `starting` | `⏳ 启动中/停止` |
| pid 活着、tmux 活着、握手完成 | `running` | `⛔️ 停止` |

高置信度推荐：增加 `degraded` 状态，而不是直接伪装成 `stopped`。原因是它能告诉用户“worker 还活但模型会话坏了”，并让 `run` 自动先 stop 再 start，避免 `run_bot.sh` 因旧 pid 存活而 fail。

### 4.2 启动失败提示 fail-soft，不阻断刷新

- 新增 `_truncate_for_telegram(text, limit=3500)` 或等价函数，保证项目操作失败提示不会超过 Telegram 单条文本上限。
- `on_project_action` 异常分支改为：
  1. 先记录完整异常到日志；
  2. 给用户发送短错误摘要 + “完整原因见日志”；
  3. 发送失败也只记录 warning，不阻断；
  4. 最后无论提示是否成功，都 best-effort 调用 `_refresh_project_overview`。
- `_health_check_worker` 保留完整日志尾部用于日志，但用户侧错误应短化；或把日志尾部限制到固定字符数。

## 8. 风险与回滚

- 风险 1：新增 `degraded` 状态后，旧代码不认识该状态。回滚方式：把 `state.json` 中对应项目状态改回 `stopped`，或执行 stop 后重新启动。
- 风险 2：tmux 检测误判导致项目被标为异常。回滚方式：只读检查 tmux 命令返回；若 tmux 不可用，保持 fail-soft 展示 `启动` 并在错误提示说明 `tmux 未安装/不可用`。
- 风险 3：修复/重启会停止旧 worker。回滚方式：保留用户点击确认，不在项目列表刷新时主动 kill；只有用户点击 `修复/重启` 才执行 stop/start。
