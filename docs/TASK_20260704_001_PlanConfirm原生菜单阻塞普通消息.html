故障排查：PlanConfirm 原生菜单吞掉 Telegram 普通消息

结论：这不是 Codex 仍在思考，而是计划收口后终端停在原生 Implement this plan? 选择菜单；Telegram 又把普通文本或旧快捷回复当业务 prompt 投递进 tmux，输入无法写入 JSONL，用户侧却看到“思考中”。

现象 / 影响

FawnStudio worker 在 11:43 输出 <proposed_plan> 后，11:46 用户发送 ?，Bot 回复正在监听；tmux 实际仍停在 Codex 原生 Yes / Fresh / No 菜单。

用户误以为卡在 Yes 普通消息未被 JSONL 消费 同源 session 已确认

修复判定

已改为 fail-closed：PlanConfirm 待处理时，普通消息 / 快捷回复不再进入 tmux；<proposed_plan> 输出也不再挂通用“全部按推荐”按钮，避免和 PlanConfirm 决策冲突。

Before：故障逻辑

  1. 模型输出 <proposed_plan>，Bot 同时发送通用快捷回复。
  2. Codex TUI 自动显示原生 Implement this plan? 菜单。
  3. 用户发普通文本或点旧快捷回复。
  4. Bot 继续 tmux 投递并回 session ack。
  5. Codex 菜单不产生用户消息 JSONL，用户看到“思考中”。

After：修复逻辑

  1. 模型输出 <proposed_plan> 时，不再挂通用快捷回复。
  2. PlanConfirm 待处理 token 仍在时，普通业务 prompt 先 fail-closed。
  3. 旧快捷回复也提示先处理 Plan 确认菜单，不投递 tmux。
  4. 用户只能通过 Yes / Fresh / No 决策，或先 No 再补充。
  5. 不再产生“tmux 成功但 JSONL 未消费”的误导性思考中状态。
