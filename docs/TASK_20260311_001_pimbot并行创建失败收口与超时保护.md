# TASK_20260311_001 pimbot 并行创建失败收口与超时保护

## 1. 背景

### 1.1 用户现象

- pimbot 在 Telegram 中选择：
  - `新建分支 + 新 CLI 并行处理`
  - `PLAN`
  - `跳过`
- 随后消息停留在：

```text
正在创建并行副本并启动并行 CLI，请稍候……
```

- 用户感知为“卡住不动”。

### 1.2 运行时证据

- 截图证据：
  - `/Users/david/.config/vibego/data/telegram/vibegobot/2026-03-11/20260311_002559971-af9761a8d112.jpg`
- pimbot 日志证据：
  - `~/.config/vibego/logs/codex/cckgpimbot/run_bot.log`
  - 锚点：`RuntimeError: ssh: connect to host gitlab.cckggroup.com port 10002: Connection reset by peer`
  - 锚点：`TelegramBadRequest: ... query is too old and response timeout expired`

### 1.3 根因结论

1. 真正失败点在并行副本准备阶段，而不是 tmux / 并行 CLI 启动阶段。
2. `prepare_parallel_workspace(...)` 内部 `git fetch --all --prune` 失败。
3. 失败后 `on_parallel_branch_confirm_callback(...)` 先执行 `callback.answer("并行创建失败")`。
4. 由于前置重操作耗时过长，callback query 已过期，导致 fallback 聊天消息没有继续发出，用户只看到处理中消息。

仓库证据：

- `bot.py`
  - 锚点：`on_parallel_branch_confirm_callback`
- `parallel_runtime.py`
  - 锚点：`prepare_parallel_workspace`
  - 锚点：`fetch --all --prune`

## 5. 最小实现

### 5.1 callback 先确认，避免 query 过期

- `on_parallel_branch_confirm_callback(...)`
  - 在进入 `prepare_parallel_workspace(...)` 前先执行：
    - `await _answer_callback_safely(callback)`

### 5.2 阶段化状态文案

- 新增：
  - `PARALLEL_BRANCH_PREPARING_MESSAGE = "正在准备并行副本，请稍候……"`
  - `PARALLEL_BRANCH_STARTING_CLI_MESSAGE = "正在启动并行 CLI，请稍候……"`

### 5.3 并行副本准备超时保护

- 新增：
  - `PARALLEL_WORKSPACE_PREPARE_TIMEOUT_SECONDS`
- `prepare_parallel_workspace(...)` 改为：
  - `await asyncio.wait_for(asyncio.to_thread(...), timeout=...)`
- 超时时统一抛出：

```text
并行副本准备超时（xxs），请检查 Git 网络或仓库状态后重试。
```

### 5.4 失败收口调整

- 异常分支不再依赖第二次 `callback.answer(...)`
- 统一改为：

```text
并行创建失败：<原始错误>
```

- 通过 `_reply_to_chat(...)` 发到 Telegram 聊天，确保用户可见

## 8. 风险与回滚

### 8.1 风险

- `asyncio.to_thread + wait_for` 的超时属于外层超时；极端情况下后台线程可能稍后才自然退出。
- 当前默认超时值为 `45s`，若后续仓库规模明显增大，可按环境变量调节。

### 8.2 回滚

- 回滚 `bot.py` 中：
  - `_answer_callback_safely`
  - `PARALLEL_*` 新常量
  - `on_parallel_branch_confirm_callback(...)` 的超时与失败收口逻辑
- 回滚 `tests/test_parallel_flow.py` 对应新增 / 修改用例
