# TASK_20260624_005 /upgrade 最终态不明确修复

## 1. 背景与目标

用户反馈：Telegram 中执行 `/upgrade` 后看起来很慢，而且只看到“pipx 阶段完成”，不知道最终是成功还是失败。

本次目标：让 `/upgrade` 明确区分 `pipx 执行中 -> 正在重启 master -> master 已重新上线/重启日志未确认`，并保证最终通知发送失败时不删除升级报告，避免用户永远看不到最终态。

## 2. 现状与根因

现状证据：

- `master.py::_run_single_upgrade_step` 只等待 `pipx upgrade vibego`，并实时编辑 Telegram 状态。
- `master.py::_announce_upgrade_completion` 在 pipx 成功后以 detached 子进程执行 `vibego stop && vibego start`，旧 master 退出后无法继续编辑原 Telegram 消息。
- `master.py::_notify_upgrade_report` 由新 master 启动后读取 `upgrade_report.json` 并发送摘要，但旧实现无论发送成功或失败都会在 `finally` 删除报告。
- 现场日志显示 2026-06-24 13:57:45 收到 `/upgrade`，13:57:48 pipx 阶段完成并安排重启，13:57:57 新 master 已启动；实际慢点不是 pipx，而是最终态没有稳定回传。

根因：升级流程跨进程，旧 master 只能确认 pipx 阶段；最终结果依赖新 master 的启动通知，但通知发送失败后报告被清理，导致 Telegram 侧停留在“半完成”状态。

## 4. 契约变更

- pipx 成功后，旧 master 不再写“升级流程完成”，而是写：`pipx 阶段完成，正在重启 master`。
- 新 master 启动后读取 `upgrade_report.json`，发送最终态：
  - 若 `upgrade_restart.log` 尾部包含 `master 已启动`，提示 `升级流程完成，master 已重新上线`；
  - 若重启日志缺失或尾部没有 `master 已启动`，提示 `pipx 阶段完成，但重启日志未确认 master 已启动`。
- 最终通知必须附带：版本变化、pipx 阶段耗时、重启命令、重启日志路径、pipx 输出尾部、重启日志尾部。
- 发送最终通知成功后才删除 `upgrade_report.json`。
- 发送最终通知失败时保留 `upgrade_report.json`，并记录 `last_notify_error` 与 `last_notify_failed_at`，供下次 master 启动重试。
- 无效 `chat_id` 仍删除报告，因为没有可恢复的通知目标。

## 5. 实现摘要

1. 新增 `_UPGRADE_RESTART_LOG_TAIL`，控制最终通知中展示的重启日志行数。
2. 新增 `_render_upgrade_restart_log_tail()`，读取并清洗 `upgrade_restart.log` 尾部，检测 `master 已启动` 标记。
3. 新增 `_write_upgrade_report_payload()`，统一升级报告原子写入。
4. 新增 `_build_upgrade_report_message()`，集中生成新 master 最终升级状态消息。
5. 修改 `_announce_upgrade_completion()`，将 pipx 成功后的 Telegram 状态改为“正在重启 master”。
6. 修改 `_notify_upgrade_report()`，发送成功后删除报告；发送失败则保留报告并写入失败元数据。

## 8. 风险与回滚

| 风险 | 缓解 |
| --- | --- |
| Telegram 临时不可用导致最终通知重复重试 | 只在 master 启动时重试；发送成功后删除报告。 |
| 重启日志尾部刚好不含 `master 已启动` | 默认 fail-soft 警告，不误报成功；用户可查看日志路径。 |
| 报告文件长期保留 | 仅发送失败时保留，并记录失败原因；后续可追加 `/upgrade_status` 查询或 TTL 清理。 |
| 文案改变影响测试或用户习惯 | 保留 pipx 输出、重启命令、日志路径，信息更明确。 |

回滚方式：移除 `_render_upgrade_restart_log_tail()`、`_write_upgrade_report_payload()`、`_build_upgrade_report_message()`，恢复 `_announce_upgrade_completion()` 与 `_notify_upgrade_report()` 的旧文案/清理逻辑；删除新增测试与 AGENTS 事实行。
