系统架构模板：C4 主线与专项扩展 v6

v6 不再把“系统架构图”理解成一个固定业务模板，而是先走 C4 主线确定抽象层级，再按用户问题叠加专项视图；如果问题本质不是系统架构，则转交到更合适的图型，避免把所有内容混成一张卡片墙。

1. 输入意图先识别用户到底问边界、组件、部署、数据、接口、安全，还是在问业务/状态/时序。
2. C4 主线信息少选 Context；常规系统总览选 Container；容器内部选 Component；环境与托管关系选 Deployment。
3. 专项扩展按命中问题叠加 Logical、Data Architecture、Data Flow、API Integration、Event-driven、Network、Security、Identity、Resilience、Observability、CI/CD。
4. 跨图型转交业务能力、领域对象、ER、状态机、时序等不强塞进系统架构，转到对应图型。
5. 质量门禁一张主图回答一个核心问题；槽位可变；没有横向滚动；不使用固定业务文案、固定层名或固定图标。

路由矩阵：先选 C4 层级，再选专项扩展

矩阵只定义“如何选模板”，不定义最终系统内容。最终层数、模块数、层名、图标和文案都来自输入材料；证据不足时保留待确认槽位。

用户只说“系统架构图”没有明确专项诉求，材料能识别多个可部署单元。
Container / 工作负载总览
默认产出大众认知中的系统总览；若材料极少，降级到 Context。
边界、用户、外部依赖想知道这个系统和谁交互、范围在哪里。
Context
突出系统边界、外部角色、外部系统与主要关系。
内部职责、模块协作关注一个可部署单元内部如何分工。
Component
只展开目标容器内部，避免把全系统和内部白盒混在一张图里。
云、环境、节点、托管出现部署、机房、集群、运行环境、托管关系。
Deployment
画运行节点、环境边界、托管关系、网络近邻与部署单元映射。
逻辑层、职责边界用户显式要求“分层、支撑层、领域层、应用层”。
Logical
作为 C4 companion view；不要把固定层名写死到所有架构图。
数据域、存储、血缘关注数据资产、存储分区、数据生产消费关系。
Data Architecture
突出数据域、存储、流向边界、治理与质量控制点。
敏感数据如何流动用户提交、数据穿越边界、合规或威胁建模。
Data Flow / DFD
使用进程、存储、外部实体、信任边界和数据流，不画成普通依赖图。
协议、接口、集成出现 API、webhook、协议、契约、第三方集成。
API Integration
突出契约边界、调用方向、同步/异步语义、错误与重试边界。
事件、队列、订阅关注异步事件、主题、消费者、事件溯源或最终一致。
Event-driven
以事件通道和发布/订阅关系为主，不把事件画成普通实线调用。
网络连接与访问路径出现 VPC、子网、网关、防火墙、专线、出入口。
Network
突出网络边界、路由路径、连接方向和安全控制点。
认证授权、主体与权限关注 SSO、token、RBAC、主体身份、权限传递。
Identity / Security
身份是安全专项子视图；若还涉及审计、边界和威胁面，升级到 Security。
安全审查、审计、威胁面出现 WAF、加密、审计、信任区、风险控制。
Security
以资产、边界、威胁、控制和证据为主，不只画登录链路。
高可用、容灾、降级关注故障域、恢复目标、熔断、备份、切换。
Resilience
突出故障域、依赖降级、RTO/RPO、恢复路径和验证点。
日志、指标、追踪、告警关注运行可见性、监控链路、排障证据。
Observability
作为运行支撑视图，映射采集点、存储、查询、告警和响应闭环。
构建、发布、回滚关注流水线、制品、部署策略、发布门禁。
CI/CD
转为交付流水线视图，系统架构只保留其对运行环境的影响。

跨图型转交：不要把非系统架构问题强塞进系统架构图

Business Architecture用户问业务能力、价值链、角色协作或业务边界时，转业务架构图；系统架构只作为后续技术承接视图。
Domain Model用户问领域对象、规则、聚合或概念关系时，转领域/状态数据模型；不要用系统层节点替代领域对象。
ER用户问表结构、字段、主外键或数据口径时，转 ER-lite 或状态/数据模型；系统架构只标出数据存储边界。
State Machine用户问状态生命周期、流转条件、事件触发时，转状态机；系统架构只保留状态来源或状态服务。
Sequence用户问函数调用、请求时序、异步回调或异常返回时，转时序图；系统架构只保留组件关系总览。
Companion Views需要多视角时，主图仍只回答一个问题；其它视角作为 companion，不在同一画布混成大全景。

选择器置信度与禁止项

显式点名优先用户明确说某图型时优先服从；但若目标明显冲突，需要在图内标注转交理由。
问题意图第二按“想理解什么”选图，而不是按材料里的关键词堆叠模板。
风险域第三安全、合规、韧性、数据敏感等风险域命中时，可生成专项 companion。
默认策略置信度 ≥0.75 直接路由；0.45-0.75 选主图并建议 companion；低于 0.45 退回 Context / Container 并标待确认。
禁止项：不要把所有模板混入一张图；不要只靠关键词路由；不要固定层名、固定业务模块、固定图标；不要使用候选 tab；不要横向滚动；不要让主图退化成多列文字卡片。

调研来源与启发

  • C4 Model diagrams：以 Context、Container、Component、Deployment 作为架构抽象主线。
  • C4 System Context：用于系统边界和外部关系。
  • C4 Container：用于可部署单元、数据存储和通信关系。
  • C4 Component：用于容器内部组件白盒。
  • C4 Deployment：用于部署节点和运行环境。
  • Azure Well-Architected design diagrams：强调图型选择、箭头方向、图例、官方图标与渐进披露。
  • arc42 Building Block View：对应逻辑/建筑块白盒。
  • arc42 Runtime View：对应运行场景和时序 companion。
  • arc42 Deployment View：对应部署/基础设施视图。
  • AWS Architecture Icons 与 Google Cloud Icons：技术特定部署图应使用官方图标，但图标不能替代语义。
  • Mermaid Architecture syntax：架构图基础图元可抽象为 group、service/node、edge、junction。
  • 用户提供的 GPT 5.5 pro 建议：启发把系统架构拆成 C4 主线、专项扩展和跨图型转交。
