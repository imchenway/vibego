# TASK_20260601_001_命令成功结果去除标准输出摘要

## 1. 背景与问题

用户截图显示：Telegram 中执行“触发微信手机自动预览”成功后，机器人仍展示 `标准输出摘要（末尾 5 行）`、完整输出提示以及详情按钮等诊断信息。用户明确反馈：成功的时候，不需要下面那一堆标准输出摘要等废话。

当前问题本质不是命令执行失败，而是成功态消息模板过度偏运维诊断：成功时继续暴露 stdout 预览与详情入口，移动端噪音较大。

## 2. 当前实现证据

- `bot.py`（锚点：`_execute_command_definition`）：命令执行完成后统一拼接 `*命令执行结果*` 消息。
- `bot.py`（锚点：`标准输出摘要（末尾 {COMMAND_OUTPUT_PREVIEW_LINES} 行）：`）：只要存在 `stdout_text`，就追加标准输出摘要，没有区分成功/失败。
- `bot.py`（锚点：`标准错误摘要（末尾 {COMMAND_OUTPUT_PREVIEW_LINES} 行）：`）：只要存在 `stderr_text`，就追加标准错误摘要，没有区分成功/失败。
- `bot.py`（锚点：`如需完整输出，请点击下方“查询详情”下载 txt 文件。`）：当前固定追加完整输出提示。
- `bot.py`（锚点：`CommandHistoryCallback(action="detail"`、`CommandHistoryCallback(action="list"`）：当前固定提供“查询详情 / 最近执行”按钮。
- `command_center/service.py`（锚点：`record_history`）：命令 stdout/stderr 已持久化到历史记录，因此即使成功消息不展示摘要，也不会丢失审计数据。

## 3. 目标行为

### 3.1 推荐收口口径

当命令执行状态为 `success` 时，Telegram 只展示用户需要确认的结果：

- 标题
- 触发方式
- 耗时
- 状态：✅ 成功

成功态不再展示：

- 退出码 `0`
- `标准输出摘要`
- `标准错误摘要`
- “如需完整输出...”提示
- “查询详情 / 最近执行”按钮

失败、超时、异常状态保持现有诊断能力：

- 保留退出码
- 保留 stdout/stderr 摘要
- 保留详情入口与最近执行入口
- 保留微信开发者工具端口不匹配时的恢复按钮

### 3.2 明确不变项

- 命令历史仍记录 stdout/stderr，便于后续查询与审计。
- 成功时如命令生成二维码/图片，仍按原流程发送图片，不因隐藏 stdout 摘要而影响功能结果。
- 不改数据库表结构。
- 不改命令中心配置。
- 不改 Telegram Bot API 调用契约。

## 9. 推荐决策

推荐选择 A：成功态默认极简，不展示 stdout/stderr 摘要、退出码、详情提示与详情按钮；失败/超时/异常保留完整诊断。

理由：这最贴近用户截图反馈，且不丢失审计数据；诊断信息应服务失败场景，而不是污染成功反馈。

## 11. PLAN 追加：命令执行中消息改为同一条消息更新

### 11.1 用户追加反馈

用户澄清两点：

1. 成功态极简不是只针对“触发微信手机自动预览”这一条命令，而应适用于所有通用命令。
2. `命令执行中` 与 `命令执行结果` 不应默认拆成两条 Telegram 消息；更理想的交互是先发送一条“执行中”消息，命令结束后编辑/更新这一条消息为最终结果。

### 11.2 当前实现现状

- `bot.py`（锚点：`show_diagnostics = status != "success"`）：当前成功态极简已经按 `status` 判定，不按命令名判定，因此实现层面已是所有通用命令生效。
- `bot.py`（锚点：`await _answer_with_markdown(reply_message, "\n".join(progress_lines))`）：当前执行中消息发送后未保存返回的
  Telegram `Message`，后续无法编辑同一条消息。
- `bot.py`（锚点：`await _answer_with_markdown(reply_message, "\n".join(lines), reply_markup=summary_markup)`
  ）：当前最终结果总是发送新消息。
- `bot.py`（锚点：`_answer_with_markdown`）：发送 Markdown 消息后会返回 `sent_message`，具备保存执行中消息引用的基础。
- `bot.py`（锚点：`_try_edit_message`）：仓库已有 MarkdownV2 安全编辑工具，编辑失败时返回 `False`，可用于 fail-open 回退为新消息。

### 11.3 推荐交互契约

推荐采用“同一条消息更新优先，失败时降级新消息”的通用契约：

1. 所有通用命令进入执行时，先发送一条 `*命令执行中*`。
2. 保存这条执行中消息的返回值。
3. 命令结束后，优先调用 `_try_edit_message(progress_message, result_text, reply_markup=summary_markup)` 将同一条消息更新为
   `*命令执行结果*`。
4. 若 Telegram 不允许编辑（消息不存在、消息过旧、内容格式不合法、网络/BadRequest 等），再回退为发送一条新的结果消息。
5. 微信开发者工具端口自动重试等中间状态，也优先编辑同一条执行中消息；编辑失败再发新提示，并把新提示作为后续可编辑消息。
6. 成功态仍使用第 10 节已经落地的通用 `show_diagnostics = status != "success"` 规则，不展示诊断摘要。
7. 失败态、超时态、异常态仍保留详情按钮、stdout/stderr 摘要和可恢复操作按钮。
8. 二维码/图片结果仍在最终结果消息更新后发送，避免把图片前置导致用户看不到命令是否成功。

### 11.7 风险与回滚

| Telegram 编辑失败      | 用户可能看不到最终结果     | helper 必须 fail-open：编辑失败立即发送新结果                 |
