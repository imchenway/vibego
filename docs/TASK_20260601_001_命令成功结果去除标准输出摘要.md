# TASK_20260601_001_命令成功结果去除标准输出摘要

## 1. 背景与问题

用户截图显示：Telegram 中执行“触发微信手机自动预览”成功后，机器人仍展示 `标准输出摘要（末尾 5 行）`、完整输出提示以及详情按钮等诊断信息。用户明确反馈：成功的时候，不需要下面那一堆标准输出摘要等废话。

当前问题本质不是命令执行失败，而是成功态消息模板过度偏运维诊断：成功时继续暴露 stdout 预览与详情入口，移动端噪音较大。

## 2. 当前实现证据

- `bot.py`（锚点：`_execute_command_definition`）：命令执行完成后统一拼接 `*命令执行结果*` 消息。
- `bot.py`（锚点：`标准输出摘要（末尾 {COMMAND_OUTPUT_PREVIEW_LINES} 行）：`）：只要存在 `stdout_text`，就追加标准输出摘要，没有区分成功/失败。
- `bot.py`（锚点：`标准错误摘要（末尾 {COMMAND_OUTPUT_PREVIEW_LINES} 行）：`）：只要存在 `stderr_text`，就追加标准错误摘要，没有区分成功/失败。
- `bot.py`（锚点：`如需完整输出，请点击下方“查询详情”下载 txt 文件。`）：当前固定追加完整输出提示。
- `bot.py`（锚点：`CommandHistoryCallback(action="detail"`、`CommandHistoryCallback(action="list"`）：当前固定提供“查询详情 / 最近执行”按钮。
- `command_center/service.py`（锚点：`record_history`）：命令 stdout/stderr 已持久化到历史记录，因此即使成功消息不展示摘要，也不会丢失审计数据。

## 3. 目标行为

### 3.1 推荐收口口径

当命令执行状态为 `success` 时，Telegram 只展示用户需要确认的结果：

- 标题
- 触发方式
- 耗时
- 状态：✅ 成功

成功态不再展示：

- 退出码 `0`
- `标准输出摘要`
- `标准错误摘要`
- “如需完整输出...”提示
- “查询详情 / 最近执行”按钮

失败、超时、异常状态保持现有诊断能力：

- 保留退出码
- 保留 stdout/stderr 摘要
- 保留详情入口与最近执行入口
- 保留微信开发者工具端口不匹配时的恢复按钮

### 3.2 明确不变项

- 命令历史仍记录 stdout/stderr，便于后续查询与审计。
- 成功时如命令生成二维码/图片，仍按原流程发送图片，不因隐藏 stdout 摘要而影响功能结果。
- 不改数据库表结构。
- 不改命令中心配置。
- 不改 Telegram Bot API 调用契约。

## 8. 风险与回滚

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| 成功时隐藏详情按钮后，用户少一步直达详情 | 需要从历史入口查询完整输出 | stdout/stderr 仍持久化；失败态保留详情按钮 |
| 部分成功命令 stdout 含关键结果 | 可能被隐藏 | 对二维码/图片等结构化结果保持原发送流程；若命令确实依赖文本结果，应定义为专门用户消息而非 stdout 摘要 |
| 误伤失败诊断 | 故障排查信息不足 | 测试覆盖失败/端口恢复按钮，确保非 success 逻辑不变 |

回滚方式：恢复 `bot.py` 中成功态与非成功态共用摘要模板的旧逻辑，并回退对应测试预期。

## 9. 推荐决策

推荐选择 A：成功态默认极简，不展示 stdout/stderr 摘要、退出码、详情提示与详情按钮；失败/超时/异常保留完整诊断。

理由：这最贴近用户截图反馈，且不丢失审计数据；诊断信息应服务失败场景，而不是污染成功反馈。
