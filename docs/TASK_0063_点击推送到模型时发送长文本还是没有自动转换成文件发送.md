# /TASK_0063 点击推送到模型时，发送长文本还是没有自动转换成文件发送（PLAN）

## 1. vibe（问题确认）

### 1.1 现象
- 在任务详情点击「🚀 推送到模型」时，若最终推送内容过长，会直接注入 tmux，而不是自动转为本地附件提示词。
- 用户补充诉求：`任务描述` 与 `补充任务描述` 在展示时希望使用代码块样式，提升可读性。

### 1.2 根因（高置信）
- 普通对话路径（`_handle_prompt_dispatch`）已有“超长文本转附件”逻辑。
- 任务推送路径（`_push_task_to_model`）之前直接调用 `_dispatch_prompt_to_model`，缺少同等兜底。
- 任务推送 payload 中 `任务描述/补充任务描述` 之前为单行拼接，长内容阅读性差。

## 2. design（方案）

### 2.1 方案 A（采用）
1. 在任务推送链路新增统一兜底函数：超长 prompt 自动落盘为文本附件，并改写为“附件列表”提示词后再推送模型。  
2. 将 `任务描述`、`补充任务描述` 字段统一改为代码块风格展示。  
3. 新增回归测试，覆盖“任务推送超长自动转附件”。  

### 2.2 关键取舍
- 代码块分隔符使用 `~~~`（而不是 ```），避免与 Telegram 预览外层 ``` 代码块冲突。
- 推送预览返回“实际发送给模型的内容”，保证可追溯。

## 3. develop（实现与自测）

### 3.1 代码变更
- `bot.py`
  - 新增 `_convert_overlong_task_prompt_to_attachment(...)`：任务推送超长内容自动转附件。
  - `_push_task_to_model(...)` 改为发送 `dispatch_prompt`（可能是附件提示词）。
  - 新增 `_append_prompt_field_as_code_block(...)`：将字段按代码块样式写入 payload。
  - `_build_model_push_payload(...)` 与 `_build_task_context_block_for_model(...)` 应用上述字段格式化。
