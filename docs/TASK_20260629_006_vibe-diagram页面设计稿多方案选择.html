当前：单稿倾向，选择空间不足

开始：用户要页面/设计稿自动路由到“页面设计稿”。
S1 只做 5 个判断目标、动作、首层、收纳、是否卡片堆叠。

修法 F1
把“页面设计稿”定义为选择型交付

推荐：一个 HTML 内提供 3-5 个候选

开始：页面设计方向评审标题仍为“页面设计稿：主题结论”。
F1 默认给 3 个候选同一目标、同一主路径，A/B/C 三个 artboard。
D2 方向是否足够开放？开放或用户明确要多方案时扩到 4-5 个。
F2 每稿必须有本质差异信息架构、首屏锚点、交互递进或响应式结构至少一项不同。
输出：对比矩阵 + 推荐项用户可直接选方向；已定方向场景允许单稿并说明原因。
A
无条件 3-5 个

选择感最强，但已定稿或小修场景容易凑数。

B
推荐：默认 3，开放到 5

把多稿绑定到方向评审，兼容单主路径和单 HTML。

C
每次先问数量

最尊重选择，但会打断工作流，尤其不利于 Telegram。

每个候选必须包含

visual thesis、content plan、interaction thesis、首屏线框、状态、响应式与可访问性旁注。

禁止伪差异

不得只换颜色、阴影、圆角凑数量；必须有信息架构或交互层级差异。

单稿例外

用户明确只要最终稿、方向已确认、缺陷修复或最终交付时可单稿。

推荐决策：选 B

将页面设计稿升级为“选择型交付”：首次方向评审默认 3 个候选，开放场景最多 5 个；仍保持一个单文件 HTML、一种生图类型、同一用户主路径。
