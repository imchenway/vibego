# /TASK_0089 缺陷提交步骤改为复现步骤与期望结果

## 1. 背景

用户要求将“报告缺陷（创建缺陷任务）”的录入结构从单一描述，调整为：

```text
标题 -> 复现步骤 -> 期望结果 -> 确认
```

并要求以下展示面保持一致：

1. 缺陷任务详情
2. 推送到模型的提示词

用户在本次确认中的决策结果为：

- 字段结构：`标题 -> 复现步骤 -> 期望结果`
- 必填规则：`复现步骤 / 期望结果` 都可跳过
- 生效范围：`仅缺陷任务`

---

## 2. 现状证据

### 2.1 变更前缺陷创建流程

- `tasks/fsm.py`
  - `TaskDefectReportStates` 原仅包含：
    - `waiting_title`
    - `waiting_description`
    - `waiting_confirm`
- `bot.py`
  - `on_task_defect_report_title()` 原在标题后进入 `waiting_description`
  - `on_task_defect_report_description()` 原负责收集单一“缺陷描述”
  - `on_task_defect_report_confirm()` 原直接把 `description` 写入缺陷任务

### 2.2 变更前展示与提示词

- `bot.py:_format_task_detail()`
  - 原统一输出 `🖊️ 描述：...`
- `bot.py:_build_model_push_payload()`
  - 原统一输出 `任务描述`
- `bot.py:_build_task_context_block_for_model()`
  - 原统一输出 `任务描述`

---

## 3. 设计决策

### 3.1 推荐方案（已实现）

不新增数据库字段，不改 schema，继续复用 `TaskRecord.description` 存储缺陷正文，但对 **缺陷类型任务** 采用统一结构化正文：

```text
复现步骤：
<内容或 ->

期望结果：
<内容或 ->
```

### 3.2 这样做的原因

✅ 不触发数据库迁移  
✅ 兼容现有任务模型与搜索能力  
✅ 仅影响 defect 流程与 defect 展示  
✅ 历史自由文本 defect 任务仍可回退展示

### 4.4 最小实现

已完成：

1. `TaskDefectReportStates`
   - 新增：
     - `waiting_reproduction`
     - `waiting_expected_result`
2. 缺陷创建流程
   - 标题后进入“复现步骤”
   - 再进入“期望结果”
   - 两步都支持：
     - 跳过
     - 附件
     - 超长转附件占位
3. 结构化缺陷正文
   - 新增统一 builder / parser
4. 缺陷详情展示
   - defect + 结构化正文 => 展示 `复现步骤 / 期望结果`
   - 历史 defect 自由文本 => 回退展示 `描述`
5. 模型提示词 / 上下文块
   - defect + 结构化正文 => 输出 `复现步骤 / 期望结果`
   - 非 defect 或旧自由文本 => 保持原 `任务描述`

### 4.5 Auto-Repair Loop（1 轮）

第一次实现后剩余 3 个失败用例，原因是测试断言过严：

1. 旧 Markdown 路径会对 `1.` 转义为 `1\\.`
2. `"任务描述：" not in ...` 误伤了 `"补充任务描述："`

已修正测试断言，未扩大生产代码改动面。

## 6. 最终结果

### 用户侧行为变化

现在“报告缺陷”流程为：

```text
任务详情
  -> 报告缺陷
  -> 输入标题
  -> 输入复现步骤（可跳过）
  -> 输入期望结果（可跳过）
  -> 确认创建
  -> 展示缺陷任务详情
```

### 展示统一口径

仅对 `defect` 任务：

1. 任务详情展示：
   - `复现步骤`
   - `期望结果`
2. 推送到模型提示词展示：
   - `复现步骤`
   - `期望结果`

### 兼容性

历史 defect 自由文本描述未被破坏：

- 若不是新结构化正文，仍按旧 `描述` 回退展示
