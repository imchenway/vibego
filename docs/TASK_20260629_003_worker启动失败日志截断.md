# TASK_20260629_003_worker 启动失败日志截断修复

## 任务背景

- 现象：用户在 Telegram Master 中启动 `Vibego` 失败，用户侧提示只看到 2026-06-27 的旧 worker 日志，真正的 2026-06-29
  启动失败原因被截断到后面。
- 现场证据：`/Users/david/.config/vibego/logs/vibe.log` 中 2026-06-29 14:09:59 明确记录
  `Telegram 连通性检查失败：在 30.0 秒内未能与 Telegram 成功握手`，但同一条失败提示的 `最近日志` 从 2026-06-27 旧会话开始。
- 当前状态：后续 2026-06-29 14:11:47 已重新启动成功，`master_state.json` 显示 `vibego` 为 `running`，`bot.pid=63490`，tmux
  会话 `vibe-vibego` 存在。

## 根因

1. `run_bot.log` 采用追加写入，`MasterManager._log_tail()` 只按最后 N 行读取，没有按本次 `boot_id` 分段。
2. 启动失败提示使用 `_truncate_text_for_telegram()` 只保留开头，导致真正根因如果在尾部，会被截掉。
3. 两者叠加后，用户侧看到的是旧日志前半段，而不是本次启动失败的有效诊断。

## 修法

- `master.py::_log_tail()` 新增 `since_token` 参数。
    - 当健康检查传入 `boot_id` 时，`最近日志` 从 `[run-bot] boot_id=<本次>` 起截取。
    - 若找不到 token，仍保留原尾部读取策略，避免丢失未知启动失败的上下文。
- `master.py::_truncate_text_for_telegram()` 改为同时保留开头与结尾。
    - 开头保留项目/动作上下文。
    - 结尾保留根因日志。
    - 中间插入省略提示，并继续保证不超过 Telegram 单条消息上限。
- `master.py::_run_master_polling()` 新增 Telegram polling 启动网络异常重试。
    - `Dispatcher.start_polling()` 在 `bot.me()` 阶段遇到 Telegram 网络超时时，只记录 warning 并按
      `MASTER_POLLING_RETRY_DELAY` 延迟重试。
    - 避免 `vibego start` 后 master 进程因 Telegram 瞬时不可达直接退出，形成“看似已启动、实际已退出”的假启动。

## 契约变更

- 用户侧启动失败提示：仍可能截断，但必须保留开头和尾部根因。
- 健康检查失败的 `最近日志`：当存在 `boot_id` 时，优先展示本次启动日志，不再让旧追加日志抢占提示主体。
- 不改变 worker 启动/停止状态机，不改变 Telegram 握手超时阈值，不改变 tmux/pid 判定口径。

## 风险与回滚

- 风险：如果某些异常发生在 `boot_id` 写入之前，`since_token` 找不到时仍回退到原尾部读取，不会比原逻辑更少信息。
- 风险：head + tail 截断会省略中间日志；但这比只保留开头更适合启动失败诊断，因为根因通常在尾部。
- 风险：master polling 重试会在 Telegram 长时间不可达时持续保活并重试；这是预期守护行为，退出仍可通过 `vibego stop`。
- 回滚：恢复 `master.py::_log_tail()` 原签名与 `_truncate_text_for_telegram()` 原截断策略，并删除对应两个测试。

## 实施顺序

1. 锁现场：读取 `vibe.log` 与 `run_bot.log`，确认 2026-06-29 14:09:59 根因与用户提示不一致。
2. TDD 红灯：新增两个复现测试，并确认失败。
3. 最小实现：只改 `master.py` 的日志截取和提示截断策略。
4. 现场重启验证：`vibego start` 暴露 master polling 启动阶段 Telegram 超时会直接退出，新增重试保护与回归测试。
5. 绿灯验证：运行聚焦测试、语法检查、doctor 与当前运行态核对。

## 已知边界

- 本修复不解决 Telegram 代理瞬时超时本身；它解决“失败原因被旧日志和截断策略隐藏”的可观测性问题。
- 本次未修改 `bot.py::ensure_telegram_connectivity()` 的 30 秒超时。部署后再次启动 worker 仍失败，但用户侧错误已收敛为本次
  boot 内的 `Telegram 网络请求失败，请检查代理或网络策略`，不再被 2026-06-27 旧日志遮蔽。
