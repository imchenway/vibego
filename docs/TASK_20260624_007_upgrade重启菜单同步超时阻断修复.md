# TASK_20260624_007 upgrade 重启菜单同步超时阻断修复

## 1. 背景与现象

- 2026-06-24 14:57 用户在 Telegram 执行 `/upgrade` 后，只看到“pipx 阶段完成，正在重启 master”的中间态消息，没有收到新 master 的最终升级报告。
- 现场日志显示新 master 已在 14:57:16 启动并打印“Master 已启动，监听管理员指令。”，随后在启动期调用 `_ensure_master_menu_button` 时因 Telegram 代理超时崩溃。
- `~/.config/vibego/state/upgrade_report.json` 仍保留，说明最终报告没有被发送成功；下次 master 正常启动时仍可重试。

## 2. 根因

`master.py` 启动顺序为：注册 startup hooks -> 同步 Telegram 聊天菜单 -> 同步命令 -> 广播键盘 -> start_polling。

其中 `_ensure_master_menu_button` / `_verify_master_menu_button` / `_ensure_master_commands` / `_verify_master_commands` 只兜底了 `TelegramBadRequest`，没有兜底代理超时、网络异常等普通异常。Telegram 菜单/命令同步本质是启动期 UI 缓存补偿，不应成为 master 启动硬门禁；一旦代理不可用，master 会在进入 polling 和执行 `_notify_upgrade_report` 前退出，用户只能看到中间态。

## 3. 受影响目录

- `master.py`
  - 影响 master 启动期 Telegram 菜单同步、命令同步与升级最终报告触达。
- `tests/test_master_menu_sync.py`
  - 增加网络异常回归测试。
- `docs/`
  - 记录本次排障、契约、测试矩阵和回滚。
- `AGENTS.md`
  - 补充 master 启动期菜单/命令同步 fail-soft 约束。

不影响：

- `bot.py` worker 投递、任务流、图片处理、resume 列表：本次故障发生在 master 启动前置同步阶段。
- 数据库表结构：无新增字段/表/索引。
- CLI 启停脚本：`vibego stop && vibego start` 行为不变。

## 4. 契约变更

### 4.1 新契约

1. master 启动期的 Telegram 菜单同步是可选体验补偿，失败时只记录 warning，不阻断启动。
2. master 启动期的 Telegram 命令同步是可选体验补偿，网络异常时应尽快跳过后续 scope，避免多次 60 秒超时拖慢启动。
3. 菜单/命令校验失败时只记录 warning，不影响 `Dispatcher.start_polling`。
4. `MASTER_STARTUP_UI_SYNC_TIMEOUT` 默认 5 秒，用于限制启动期菜单/命令同步等待时间，避免 Telegram 代理默认 60 秒超时拖慢 `/upgrade` 重启。
5. `/upgrade` 的最终态仍由 `_notify_upgrade_report` 负责；本次修复保证 master 不会在它之前被菜单同步异常杀死。

### 4.2 非契约

- 不保证 Telegram 网络异常时菜单按钮/命令列表立即刷新成功。
- 不新增自动重启或自动代理切换逻辑。
- 不删除 `upgrade_report.json` 的保留重试机制。

## 7. 风险与回滚

- 风险：Telegram 菜单或命令同步失败时，用户端可能短时间看到旧菜单文案。
  - 接受原因：旧菜单不影响 `/upgrade`、`/projects` 等命令输入；master 可用性优先级更高。
- 风险：过宽兜底可能掩盖真实 Telegram API 异常。
  - 缓解：保留 warning 日志，异常文本会写入 `vibe.log`。
- 回滚：恢复 `master.py` 中四处宽异常兜底即可，但会重新引入“代理超时导致 master 启动崩溃”的问题。
