# TASK_20260310_005 并行会话 stale 状态自动降级与 0115 清理

## 1. 背景

用户反馈：

- `TASK_0093` 的并行会话可正常使用
- `TASK_0115` 点击旧并行消息按钮时提示：

```text
tmux错误：Command '['/opt/homebrew/bin/tmux', '-u', 'has-session', '-t', 'vibe-par-hyphamall-task_0115']' returned non-zero exit status 1.
推送失败：并行 CLI 未就绪，请稍后再试。
```

现场证据：

- `tmux ls` 仅存在：
  - `vibe-hyphamall`
  - `vibe-par-hyphamall-task_0093`
- `~/.config/vibego/data/hyphamall.db` 中仍有：
  - `TASK_0115 status=running tmux_session=vibe-par-hyphamall-task_0115`

## 2. 根因结论

### 2.1 `TASK_0115` 是 stale 并行会话

- 数据库状态仍是 `running`
- 但 tmux 会话 `vibe-par-hyphamall-task_0115` 已不存在
- 运行目录与 pointer 还残留，导致用户误以为仍可操作

### 2.2 活动并行会话判定缺少运行态健康检查

- `bot.py:8162-8171`
  - `_get_active_parallel_session_for_task(...)` 旧逻辑只检查 DB 状态
  - 未检查：
    - tmux session 是否存在
    - runtime 目录是否存在
    - pointer 文件是否存在

### 2.3 删除逻辑只允许 active session

- `bot.py:8264-8295`
  - 旧逻辑删除前先调用 `_get_active_parallel_session_for_task(...)`
  - stale / closed 会话无法被正常清理

## 6. 最小实现

### 6.1 活动会话增加运行态健康检查

- 新增 `_parallel_tmux_session_exists(session_name)`
- 新增 `_parallel_session_runtime_issue(session)`
  - 当前校验：
    - tmux session 是否存在
    - workspace_root 是否存在
    - pointer_file 是否存在

### 6.2 stale 会话自动降级

- `_get_active_parallel_session_for_task(...)`
  - 若发现运行态已失活：
    - 自动将 DB 状态更新为 `closed`
    - 写入 `last_error`
    - 清理内存绑定 / watcher / callback token
    - 返回 `None`

### 6.3 删除逻辑允许清理 closed/stale 会话

- `_delete_parallel_session_workspace(...)`
  - 改为直接读取 `PARALLEL_SESSION_STORE.get_session(...)`
  - 不再要求 session 必须是 active
