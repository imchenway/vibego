# /TASK_0073 偶现消息发送问题

## 1. 背景

- 用户反馈：普通 Telegram 图文混合消息在少数情况下仍会被拆成两次发送，表现为连续出现两条 `💭 codex思考中...`。
- 用户确认的取舍：**稍慢但更稳**，优先保证同一组图文只触发一次推送，接受首条 ack 略慢。

## 2. 仓库证据

- 普通附件消息入口：`bot.py`（锚点：`async def on_media_message`）
- 普通 media_group 聚合：`bot.py`（锚点：`async def _enqueue_media_group_message`、`async def _finalize_media_group_after_delay`）
- 当前默认等待窗口：`bot.py`（锚点：`MEDIA_GROUP_AGGREGATION_DELAY = ...0.8`，修改前）
- 相关历史任务：
  - `docs/TASK_0056_从telegram发送的消息被拆分了多条发送.md`
  - `docs/TASK_0057_文件是创建好了但是消息被拆分为了两条发送.md`
  - `docs/TASK_0061_长文本发送问题.md`

## 3. 设计决策

### 3.1 采用方案 ✅

- 仅上调默认 `MEDIA_GROUP_AGGREGATION_DELAY`
- 默认值从 `0.8` 提升到 `1.5`
- 不改普通聊天 media_group 的现有 debounce 结构：仍然是“最后一张到达后重新计时，再统一 dispatch”

### 3.2 为什么这样做

- 本次问题是“图片稍晚到达导致第一批已 finalize”，与 quiet window 偏短直接相关。
- 用户已明确接受“稍慢但更稳”，因此优先做最小且可验证的修复。

### 3.3 本次不做

- 不新增新的媒体组状态机
- 不重构任务/缺陷/附件绑定流程
- 不引入新的环境变量名称

## 7. 风险与边界

- 由于 `MEDIA_GROUP_AGGREGATION_DELAY` 是共享常量，任务/缺陷/附件绑定流程的 media_group 等待时间也会同步略增。
- 本次已有直连回归测试覆盖这些路径，确保行为正确；用户层面的变化主要是相册类输入的首条 ack 会略慢。
