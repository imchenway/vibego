# TASK_20260625_007 故障排查 HTML 图内容重设计

## 1. 背景与目标

用户反馈：当前 `html-visual-communication` 生成的故障排查图，把“根因、修法、验证”等文字直接写进 HTML 节点里，读起来像 Markdown 搬家，失去了让 skill 画图的意义。

本次目标：重设计故障排查图的内容模型，让用户打开图后能更快理解故障来龙去脉：先看懂发生了什么、影响范围、证据如何收敛、根因为什么成立，再看修法、验证、回滚和剩余风险。

## 3. 契约变更

### 3.1 修改前

故障排查图规则只有线性结构：`现象 → 证据 → 假设 → 根因 → 修法 → 验证`。

问题：该结构容易诱导模型把每个名词做成一个卡片，再把分析文字塞进节点。用户看到的是“内容分类”，不是“故障故事”。

### 3.2 修改后

故障排查图主结构固定为：

1. **故障故事线**：现象、影响、触发点、关键证据、根因或最高可疑点、修法、验证状态。
2. **证据阶梯**：E1/E2/E3 编号，展示来源、观察结果、指向判断。
3. **假设裁决**：成立 / 排除 / 待验证三列，所有假设必须绑定证据或标待补证据。
4. **行动闭环**：已做修法、验证命令/结果、回滚动作、剩余风险。

关键约束：

- 主图不写长段根因分析。
- 未验证只能标为最高可疑点，不能写成已确认根因。
- 根因节点必须同时显示证据编号、置信度、验证状态。
- 点击详情只补充日志、截图、JSONL、SQL、命令输出等长证据，不承担主线理解。

## 6. 风险与回滚

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| 规则过细导致模型输出变长 | HTML 仍可能信息过载 | 明确“主图不写长段根因分析”，长内容进入详情。 |
| 未验证根因被误画成结论 | 用户误以为已经修复 | 强制“最高可疑点”和“验证状态”标签。 |
| 视觉重构影响其他图型 | 非故障图行为被误改 | 本次只修改“故障排查图规则”小节。 |
| 已运行 worker 未同步新 skill | Telegram 侧继续使用旧指令 | 重启 worker 或重新同步 AGENTS 后生效。 |

回滚方式：恢复 `vibego_cli/data/skills/html-visual-communication/SKILL.md` 中故障排查图规则为旧线性四条规则，并移除新增测试断言与本文档中对应变更。
