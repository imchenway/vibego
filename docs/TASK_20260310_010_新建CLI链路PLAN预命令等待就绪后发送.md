# TASK_20260310_010 新建 CLI 链路 PLAN 预命令等待就绪后发送

## 1. 背景

用户反馈：

- 原生 CLI 链路选择 `PLAN` 时，tmux 右下角能进入紫色 `Plan mode`
- 新建 CLI 链路选择 `PLAN` 时，tmux 中看不到 `Plan mode` 标识

已确认前提：

- 当前仓库代码已经把 `push_mode` 透传为 `intended_mode`
- 所以问题不再是“没发 `/plan`”，而更像是“发得太早”

## 2. 根因结论

### 2.1 新建并行 tmux 会话后，没有等待 Codex UI ready

- `bot.py:8300-8323`
  - `_start_parallel_tmux_session(...)` 启动完 tmux/codex 后立即返回
- `scripts/start_tmux_codex.sh:168-169`
  - 只是把 Codex 启动命令 `send-keys` 进去
  - 没有 ready 回执

### 2.2 `/plan` 发送发生在 ready 等待之前

- `bot.py:1863-1908`
  - `_maybe_send_plan_switch_command(...)` 会直接 `tmux_send_line(..., "/plan")`
- 对于刚创建好的并行 CLI，会出现：
  - tmux 会话已存在
  - 但 Codex UI 还没进入稳定输入态
  - `/plan` 可能被发早

## 6. 最小实现

### 6.1 新增并行 tmux ready 探测

- `_capture_tmux_output_for_session(...)`
  - 指定 session 抓取尾部输出
- `_is_tmux_session_ready_for_plan_switch(...)`
  - 通过 `OpenAI Codex` / `model:` / `/model to change` 等标记判断 ready
- `_wait_tmux_session_ready_for_plan_switch(...)`
  - 异步轮询直到 ready 或超时

### 6.2 `/plan` 发送前增加并行会话等待

- `_maybe_send_plan_switch_command(...)`
  - 当 `tmux_session` 指向新建并行 session 时
  - 先等待 ready
  - 再发送 `/plan`
  - 若超时仍未 ready，则记录 warning，但仍继续尝试发送 `/plan`
