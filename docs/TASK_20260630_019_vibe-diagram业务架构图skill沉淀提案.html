需求决策：业务架构图 skill 沉淀提案

基于 TASK_016、017、018 的试画反馈，沉淀“业务架构图 / 领域地图”应该如何生成。此页是提案，不修改 skill。

可沉淀的核心结论

业务架构图不是系统架构图，也不是业务流程图，更不是调研报告。它应该回答：业务由哪些参与方、能力、业务对象、规则和边界构成；用户沿着业务主轴看一遍，就能知道这个业务能力“服务谁、解决什么、靠什么依据、在哪里止步、如何沉淀”。

016 / 018 的布局

四层领域地图较稳定：参与方边界、业务能力层、业务对象关系、规则约束热区，再加一个服务结果决策轴。它能保持全局可读，不会像 017 那样拥挤。

避免

017 的对象生命周期强排版

单轴生命周期在业务架构里容易把横向关系塞得太满，读者会感觉布局乱。该形态可作为局部解释，不适合作为默认业务架构图。

保留

018 的业务化内容

把 AppConfig / Session / Scene 等技术对象改写成客服触点、咨询工单、业务场景、服务依据、人工接续、服务质量资产，读起来更像业务图。

避免

原图的章节卡片报告

摘要、事实口径、能力层、对象关系、规则矩阵各自成章节，会迫使用户上下拼图；这不是“一眼看懂业务结构”。

建议写入 skill 的规则清单

  • 先业务化命名：不要把代码对象原样放进主图。AppConfig、Session、Scene、Tool 等应翻译为客服触点、咨询工单、业务场景、服务依据等业务对象；技术名可进点击详情。
  • 首屏只保留一张主领域地图：不要在主图前先放大段摘要卡片、事实口径卡片、证据卡片；摘要最多一行，证据放节点详情。
  • 主图四层结构：默认采用“参与方边界 → 业务能力层 → 业务对象关系 → 规则约束热区”，必要时加“服务结果/决策轴”。
  • 规则必须贴对象：身份、权限、真实数据、财务红线、人工接续、回滚等规则要贴在相关对象/能力旁，不得独立堆成底部规则清单。
  • 能力不是功能清单：能力节点写“服务承诺、业务目标、归属方、输入输出”，不要把接口、表、类名、DDL 当主内容。
  • 对象关系必须有动词：拥有、提出、承接、产生、约束、查询、交接、沉淀、反哺；没有关系动词就不是领域地图。
  • 不要误画成流程图：业务架构可以有主轴，但不强调先后办理步骤；如果重点是“先做什么后做什么”，应切换到业务流程图。
  • 证据进入点击详情：文件路径、行号、DDL、源码锚点、长说明只做节点详情；主图保留证据编号或极短摘要。

## 业务架构图专用骨架

业务架构图 / 领域地图必须默认回答“业务能力如何被角色、对象和规则约束”，而不是回答“系统怎么部署”或“步骤怎么执行”。

默认主骨架：
`参与方边界 → 业务能力层 → 业务对象关系 → 规则约束热区 → 服务结果 / 决策轴`

执行规则：
1. 主图必须是一张首屏可读的领域地图；禁止把摘要、事实口径、能力、对象、规则、证据拆成章节化卡片报告。
2. 参与方边界写清业务发起方、自有能力承接方、依据提供方、人工/外部边界和运营闭环方；上下游系统只能作为业务边界，不得画成部署组件。
3. 业务能力层写服务承诺、业务目标、归属方、输入输出；禁止把接口、表、类名、DDL 或工具名当作能力主体。
4. 业务对象必须业务化命名，例如“客服触点、咨询工单、业务场景、服务依据、人工接续、服务质量资产”；技术对象名只能作为点击详情或证据补充。
5. 对象关系必须用动词显式连接：拥有、提出、承接、产生、约束、查询、交接、沉淀、反哺。没有关系动词就不能称为领域地图。
6. 规则约束热区必须贴近被约束的对象或能力，至少包含口径、阈值、权限、风险、状态边界或回滚条件；禁止把规则堆到底部当说明表。
7. 证据、文件路径、源码锚点、DDL 和长说明进入节点点击详情；主图只保留业务结论、关系和必要证据编号。
8. 如果图的核心变成动作先后、审批流、异常回退，应切换为业务流程图；不要把流程塞进业务架构图。

反例门禁：
- 如果去掉标题后只剩“摘要卡片 + 事实卡片 + 能力卡片 + 对象卡片 + 规则卡片”，必须重画。
- 如果主图需要逐段阅读文字才能理解业务结构，而不是沿关系线和分层读懂，必须重画。
- 如果技术对象名比业务对象名更醒目，必须重写节点命名。
