# TASK_20260309_二维码回传顺序修复

## 1. 背景

- 用户反馈：`wx-dev-preview` 执行成功后，Telegram 中只看到了“命令执行结果”摘要，没有收到二维码图片。
- 用户新增口径：**二维码图片需要放在最后发送到 Telegram。**

## 2. 关键证据

- `bot.py:4584-4607`
  - 旧逻辑会在摘要前直接 `send_photo`。
- `bot.py:4617-4737`
  - 摘要消息在图片发送之后才发送。
- `command_center/defaults.py:41-44`
  - 旧默认命令把 `OUTPUT_QR` 指向 `~/Downloads`。
- `~/.config/vibego/logs/codex/hyphamall/run_bot.log:47243,47248`
  - 运行期两次出现 `命令输出图片发送失败`。

## 3. 结论

高置信度问题分为两层：

1. 默认二维码输出目录走 `~/Downloads`，更容易触发回传失败。
2. 旧消息链路是“先图片、后摘要”，与用户期望不一致，也让失败场景更难理解。

## 4. 本次改动

### 4.1 默认路径修复

- 文件：`command_center/defaults.py`
- 改动：
  - `wx-dev-preview` 默认输出目录从 `~/Downloads` 改为 `/tmp`
  - 描述文案同步说明：避免 macOS Downloads 权限影响回传

### 4.2 发送顺序修复

- 文件：`bot.py`
- 改动：
  - 先解析 `TG_PHOTO_FILE`
  - 先发送“命令执行结果”摘要
  - 再发送二维码图片，确保图片消息位于最后

### 4.3 失败兜底

- 文件：`bot.py`
- 改动：
  - `send_photo` 失败时，自动降级为 `send_document`
  - 避免二维码彻底丢失
