# TASK_20260624_006 /upgrade 重启后项目列表重复推送修复

## 1. 背景与目标

用户反馈：执行 `/upgrade` 后，Telegram 连续出现 `pipx 阶段完成`、项目列表、`Master bot 已启动` 项目列表、最终升级报告，看起来像系统在重复执行或状态混乱。

目标：升级重启场景只保留一条升级最终报告作为主反馈，不再额外推送重启成功项目列表和 CLI 自动 `/start` 项目列表，避免噪音和误解。

## 2. 现状与根因

现状证据：

- `master.py` 启动阶段注册了 `_notify_restart_success`、`_notify_start_signal`、`_notify_upgrade_report` 三个通知入口。
- `/upgrade` 成功后会写入 `upgrade_report.json`，新 master 由 `_notify_upgrade_report` 发送最终升级状态。
- 同一次升级重启还可能存在 `restart_signal.json` 与 `start_signal.json`，分别触发 `_notify_restart_success` 和 `_notify_start_signal` 推送项目列表。
- 现场日志显示 2026-06-24 14:40:36 重启通知失败后走管理员项目列表兜底，14:40:42 自动启动通知又推送一次项目列表，随后最终升级报告才发出。

根因：升级最终报告已经承担“master 重新上线”的职责，但其他启动通知入口没有识别升级场景，导致三路通知并发输出。

## 3. 受影响范围

| 范围 | 是否影响 | 说明 |
| --- | --- | --- |
| `master.py` | 是 | 新增升级报告待发送判断；升级场景下跳过重启成功通知和自动 `/start` 概览，并清理对应 signal。 |
| `tests/test_master_restart.py` | 是 | 新增两个回归用例，锁定升级报告存在时不重复推送项目列表。 |
| `docs/` | 是 | 新增本文档记录契约、风险与验证。 |
| `AGENTS.md` | 是 | 补充 `/upgrade` 最终态约束，明确升级报告接管启动通知。 |
| `tests/test_master_update_notifications.py` | 否 | 既有最终升级报告测试保持不变。 |
| Worker `bot.py` | 否 | 本次只影响 Master 启动通知分流。 |
| 数据库/项目配置 | 否 | 不新增字段；只清理已有 signal 文件。 |

## 4. 契约变更

- 当 `_UPGRADE_REPORT_PATH` 存在时，表示新 master 仍有升级最终报告待发送。
- 在升级报告待发送期间：
  - `_notify_restart_success` 不发送重启成功通知，不推送项目列表；会清理 `restart_signal.json` 与兼容旧路径，避免下次启动补发。
  - `_notify_start_signal` 不发送自动 `/start` 概览；会清理 `start_signal.json`，避免下次启动补发。
  - `_notify_upgrade_report` 仍负责发送唯一的最终升级状态。
- 若最终升级报告发送失败，`upgrade_report.json` 会保留；后续启动仍继续抑制重复项目列表，只重试最终升级报告。
- 非升级场景不变：普通 restart/start 仍按原逻辑发送项目列表。

## 5. 实现摘要

1. 新增 `_upgrade_report_pending()`，判断是否存在待发送升级报告。
2. 新增 `_discard_restart_signal_files()`，升级报告接管通知后清理 restart signal。
3. 新增 `_discard_start_signal_file()`，升级报告接管通知后清理 start signal。
4. 在 `_notify_restart_success()` 开头检测升级报告，命中则跳过重启通知与项目列表。
5. 在 `_notify_start_signal()` 开头检测升级报告，命中则跳过自动 `/start` 概览。

## 8. 风险与回滚

| 风险 | 缓解 |
| --- | --- |
| 升级报告文件损坏但存在，导致普通启动通知被抑制 | `_notify_upgrade_report` 会读取并清理无效报告；下一次启动恢复普通通知。 |
| 最终升级报告发送失败后没有项目列表 | 这是预期取舍：避免重复噪音，保留报告供下次重试；用户仍可手动发 `/projects`。 |
| 清理 signal 影响普通 restart/start | 只在 `upgrade_report.json` 存在时清理，普通场景不受影响。 |
