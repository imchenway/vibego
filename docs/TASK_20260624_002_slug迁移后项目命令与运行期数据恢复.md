# TASK_20260624_002 slug 迁移后项目命令与运行期数据恢复

## 1. 背景与目标

用户反馈：`vibego` 的“命令管理”打开后，之前的 patch 命令不见了。现场只读取证显示：

| 项目 | 旧运行期 DB | 新运行期 DB | 现象 |
| --- | --- | --- | --- |
| Vibego | `~/.config/vibego/data/vibegobot.db`：`commands=4`、`tasks=108`、`command_history=224` | `~/.config/vibego/data/vibego.db`：`commands=0`、`tasks=0`、`command_history=0` | `pipx-publish-patch/minor/major` 等命令仍在旧库，新 worker 读新库所以看不到。 |
| WMS | `cckgwmsappcore.db`：`tasks=24` | `wms.db`：不存在 | 任务历史也会受影响。 |
| WMS2 | `cckgwms2bot.db`：`tasks=2` | `wms2.db`：不存在 | 同上。 |
| Blog | `imchenwayblogbot.db`：`commands=2`、`tasks=4`、`history=4` | `blog.db`：不存在 | 同上。 |

根因：`TASK_20260621_002` 已把项目 `project_slug` 从长 bot slug 收敛到展示名 slug，例如 `vibegobot -> vibego`，但当时只迁移了项目配置与状态 key，没有迁移 `~/.config/vibego/data/{slug}.db`。因此 worker 切到新 slug 后读的是新空库。

本任务目标：Master 启动时自动恢复旧 slug 对应的运行期 DB，使命令、任务、命令历史随新 slug 可见；若新旧库都有业务数据则 fail-closed，避免覆盖用户新数据。

## 2. 现状与根因

### 2.1 代码现状

- Worker 运行期 DB 路径由 `bot.py` 的 `TASK_DB_PATH = DATA_ROOT / f"{PROJECT_SLUG}.db"` 决定。
- Master 的 slug 修复由 `project_repository.py::_repair_records()` 完成，并通过 `ProjectRepository.slug_migrations` 暴露旧 slug 到新 slug 的映射。
- Master 启动入口 `master.py::bootstrap_manager()` 已把 `slug_migrations` 传给 `StateStore`，但此前没有消费该映射迁移运行期 DB。
- 已经升级过的环境里，`master.db/projects.json` 可能已经只有新 slug；此时单靠本次启动的 `slug_migrations` 会为空，需要从 `projects.json*.bak*` 备份恢复旧 slug 映射。

### 2.2 根因归纳

这是“配置身份迁移”与“运行期数据身份迁移”不一致：

1. 配置身份：`project_slug` 已从旧长 slug 改为新短 slug。
2. 运行期数据：命令、任务、历史仍保存在旧 `{old_slug}.db`。
3. Worker 入口：只按新 `{new_slug}.db` 读写，因此旧数据表现为“消失”。

## 3. 受影响目录与边界

| 范围 | 是否影响 | 说明 |
| --- | --- | --- |
| `project_repository.py` | 是 | 新增运行期 DB 迁移工具；从 `projects.json` 备份恢复已执行过的旧 slug 映射；迁移时复制旧 DB、改写 `project_slug` 列、写幂等标记。 |
| `master.py` | 是 | `bootstrap_manager()` 初始化仓库后调用运行期 DB 迁移；冲突时阻断启动。 |
| `tests/test_master_project_management.py` | 是 | 覆盖复制迁移、空目标库替换、目标库冲突、幂等二次启动、从备份恢复映射、Master 启动集成。 |
| `docs/` | 是 | 记录本次调研、设计、测试矩阵与回滚。 |
| `bot.py` | 否 | Worker 仍按 `PROJECT_SLUG.db` 读写，不改消息/命令中心逻辑。 |
| SQLite schema | 否 | 不新增表字段；只复制 DB 文件并改写已有表的 `project_slug` 值。 |
| 真实运行期 DB | 否（当前开发阶段） | 本次实现与测试只操作临时目录；真实迁移将在用户重启 Master 后由启动流程触发。 |

## 4. 契约变更

1. `migrate_runtime_data_for_slug_migrations(data_dir, slug_migrations)`：
   - 输入旧 slug 到新 slug 的映射。
   - 若旧 `{old_slug}.db` 不存在：跳过并记录 `skipped_missing_source`。
   - 若新 `{new_slug}.db` 不存在：复制旧库到新库，并将所有包含 `project_slug` 列的表中旧 slug 改写成新 slug。
   - 若新库存在但无业务行：先备份新空库，再替换为旧库副本。
   - 若新库已有业务行且没有迁移幂等标记：抛出 `RuntimeDataMigrationConflictError`，Master 启动 fail-closed。
   - 若新库已有业务行且已存在迁移幂等标记：返回 `skipped_already_migrated`，允许重复启动。
2. `ProjectRepository`：
   - 继续保留 `_repair_records()` 产生的 `slug_migrations`。
   - 额外读取 `projects.json*.bak*`，按 `bot_name` 或唯一 `workdir` 匹配当前项目，恢复“旧版本已经修过配置但没迁移运行期 DB”的旧 slug 映射。
3. `master.py::bootstrap_manager()`：
   - 仓库初始化后立即迁移运行期 DB，再构建 `MasterManager`。
   - 迁移冲突时记录错误并退出，避免覆盖新库业务数据。

## 7. 风险与回滚

| 风险 | 影响 | 控制方式 |
| --- | --- | --- |
| 新旧库都有业务数据 | 自动覆盖会丢数据 | 直接 fail-closed，不做覆盖；需人工比对合并。 |
| 旧库保留导致重复启动误判 | 每次启动都看到旧库 | 写 `.migrated-to-*.marker` 幂等标记，后续跳过。 |
| `projects.json` 备份匹配错误 | 可能迁错项目 | 只按稳定 `bot_name` 或唯一 `workdir` 匹配；若旧 slug 已被当前项目占用或映射冲突则抛错。 |
| 真实迁移后回滚 | 新库已复制旧数据 | 旧 `{old_slug}.db` 不删除；空目标库替换前会备份为 `{new_slug}.db.bak-*`；可停止服务后手工恢复。 |
