# /TASK_0078 报错了

## 1. 背景

### 1.1 用户现象

- 并行推送链路中，选择 `PLAN` 模式并跳过补充描述后，进入“共同分支批量选择”页。
- 点击共同分支按钮后，Telegram 弹窗报错：`共同分支已失效，请改用逐个选择。`
- 关联历史任务：`/TASK_0077`。

### 1.2 附件证据

- 错误截图：`/Users/david/.config/vibego/data/telegram/vibegobot/2026-03-10/20260310_164316166-b83ff0856a7c.jpg`

### 1.3 仓库证据

- 共同分支计算已支持忽略“无远端分支的本地根仓库”：`bot.py`（锚点：`filter_common_branch_repo_options`、`collect_common_branch_refs`）
- 共同分支点击后仍遍历 `session.repo_options` 全量仓库重新匹配：`bot.py`（锚点：`on_parallel_common_branch_select_callback`）
- 确认阶段仍要求 `len(session.selections) == len(session.repo_options)`：`bot.py`（锚点：`on_parallel_branch_confirm_callback`）

## 2. 已确认口径

1. 被共同分支计算忽略的根仓库，不自动排除出并行创建。
2. 用户点击共同分支后：
   - 先批量应用到参与共同分支计算的仓库；
   - 再自动进入“逐个补选”流程，补选被忽略仓库。
3. 本次不仅修报错，还顺带补强并行分支流程的边界回归。

## 6. 最小实现

### 6.1 会话建模

- `ParallelLaunchSession` 新增 `common_branch_repo_keys`
- `_begin_parallel_launch` 在构建会话时，记录真正参与共同分支计算的仓库键集合

### 6.2 共同分支点击逻辑

- 仅对 `common_branch_repo_keys` 内的仓库应用共同分支
- 若仍存在未选仓库，直接跳到首个未补选仓库页
- 不再把“被忽略根仓库”错误当成“共同分支失效”

### 6.3 逐个补选链路补强

- “逐个选择”不再清空现有已选结果
- 单仓库选分支后，自动跳过已完成仓库，定位下一个未选仓库
- 摘要页若仍有未选仓库：
  - 展示待补选列表
  - 按钮改为“🧩 继续补选剩余仓库”

### 6.4 确认前保护

- `on_parallel_branch_confirm_callback` 先校验是否还有未选仓库
- 若未补选完，则保留会话，不提前 `pop`

## 8. 用户可见结果

1. 共同分支页中，若根仓库仅因“无远端分支”被忽略：
   - 点击共同分支后不再报“共同分支已失效”
2. 系统会：
   - 先批量应用共同分支到参与计算的仓库
   - 再自动跳转到剩余仓库的逐个补选页
3. 补选过程中：
   - 已批量选择结果会保留
   - 已完成仓库会被自动跳过
4. 若仍有仓库未选：
   - 摘要页会明确提示待补选仓库
   - 不会错误进入并行创建
