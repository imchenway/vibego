需求决策：系统架构图还有优化空间

当前版本已经从“证据墙 + 卡片目录”向架构图靠近，但仍应继续减少同屏实现清单，把主流向、系统边界和运行语义变成第一视觉层。

判断：有优化空间，而且值得继续收敛

最关键的下一步不是加更多节点，而是做减法：首屏只保留 6-8 个宏节点，让用户不读正文也能看出系统从入口到数据/观测面的运行方式。

推荐：两级架构图 收敛主图节点数 三条流向显性化 避免同屏实现清单

3 秒钟测试

遮住节点正文，只看层、边界和箭头：如果说不出“请求从哪来、经过哪里、读写什么、失败怎么兜底”，这张系统架构图就还要重画。

推荐目标形态：首屏宏观拓扑，细节二级展开

入口用户 / 客户端微信小程序、Webhook、CLI、外部调用方。
接入SDK / Channel统一入口、appCode、trace、降级入口。
网关API / Gateway鉴权、路由、门面、feature flag。
编排Agent / Service识别意图、补槽、执行业务或转人工。
工具Adapter / ToolGateway业务工具、外部 API、Guard 约束。
状态DB / Vector / StorePostgreSQL、pgvector、消息与会话状态。
观测/兜底Trace / Human日志追踪、告警、人工接管路径。

主请求流

入口 → SDK → 网关 → Agent → 返回用户。

数据 / 知识流

Agent → 工具 / RAG → DB / pgvector → 回写状态。

兜底 / 异常流

低置信、关闭自助、缺工具或超时 → 人工 / 七鱼。

A. 强化系统架构图规则 🌟

  • 限制首屏宏节点 6-8 个。
  • 要求先画三条流向。
  • 组件细节进入二级展开或详情。

B. 同时做技术设计图

  • 系统架构图负责全貌。
  • 技术设计图承载 Controller / Handler / DB 细节。
  • 适合落地评审。

C. 只优化视觉样式

  • 成本低。
  • 但无法解决主图仍像实现清单的问题。
  • 不推荐单独做。
问题 优化规则 判断标准
节点太多，像清单 首屏只放宏节点；超过 8 个必须收纳。 不读正文也能说出架构层次。
线条不够主导 主请求流、数据流、兜底流三条路径必须显性化。 沿箭头能走到数据/观测面。
状态混杂 区分已落地、待接入、待确认、当前缺口。 用户不用读说明也能看出风险点。
架构图和实现图混在一起 系统架构图看全貌；技术设计图看实现细节。 主图不再塞满 Controller / Facade / Handler 清单。
