# TASK_20260630_006 vibe-diagram 系统架构图进一步优化建议

## 结论

当前版本比“分层卡片 + 底部证据墙”更接近系统架构图，但仍有优化空间。主要问题不是细节不够，而是*
*架构图仍然承载了太多实现清单，第一眼的系统边界、主链路和运行语义还不够强**。

## 推荐优化方向

### 1. 首屏只保留一张 6-8 个宏节点的架构总览

系统架构图第一屏应先回答：谁进来、经过什么边界、到哪些核心服务、依赖哪些运行/数据面、如何兜底。

推荐总览层级：

`用户/客户端 → 接入层/SDK → 网关/API → Agent/业务编排 → 工具/外部依赖 → 状态/数据/观测 → 人工兜底`

每层 1-2 个宏节点，细节通过展开区或节点详情进入。

### 2. 主流向必须比节点更显眼

当前节点已经可读，但线条和系统流向仍偏弱。下一版应把主请求流、数据读写流、兜底流做成三条可追踪路径：

- 蓝色粗线：主请求流
- 绿色细线：数据/知识检索流
- 琥珀虚线：兜底/人工接管流

用户不读节点文字，也应该能跟着三条线看懂系统怎么跑。

### 3. 组件细节从“同屏卡片”改成“二级展开”

系统架构图不应在主图里列出所有 Controller、Facade、Handler、Feature Flag。推荐规则：

- 主图：只放宏节点和运行语义。
- 节点详情：放源码路径、行号、文件证据。
- 节点下方轻量展开：只放该层最关键的 2-4 个子组件。
- 独立矩阵：只放接口/数据/风险/测试这类跨节点关系。

### 4. 明确“设计态 / 已落地 / 待确认 / 缺口”的状态编码

架构图经常混合仓库静态设计、已落地代码、运行时状态和推断。下一版应把状态作为视觉语法，而不是说明文字：

- 实线边框：已落地代码/DDL
- 虚线边框：设计态或待接入
- 红色角标：当前缺口
- 灰色问号：运行时待确认

### 5. 系统架构图与技术实现图分离

如果用户要一眼看架构，就先画系统架构图；如果要看 Controller/Facade/Handler/DB 细节，再切技术设计图或代码时序图。不要把二者压在一张主图里。

## 推荐下一步

推荐把 vibe-diagram 继续强化成两级架构图规则：

1. 系统架构图默认先输出“总览拓扑层”。
2. 如果组件超过 8 个，必须把实现细节降级为节点详情或二级展开，不得继续同屏堆节点。
3. 每张系统架构图必须先标三条路径：主请求流、数据流、兜底/异常流。
4. 输出前做“3 秒钟测试”：不读正文，只看层、边界和箭头，能否说出系统怎么运行；不能则重画。
