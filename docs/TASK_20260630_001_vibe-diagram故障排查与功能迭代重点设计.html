需求决策：故障排查与功能迭代必须重点设计

这两类图的任务不是“画结论”，而是让用户理解当前代码/功能现状，并在此基础上判断修复或迭代是否可信。

推荐：为故障排查与功能迭代新增专用骨架。二者都必须先画“当前现状”，再画证据裁决、差异映射、修法/目标、验证与回滚。

故障排查：从现象追到根因

用户要看懂故障如何沿当前链路发生，以及修法如何切断问题链路。

1. 现象 / 影响用户看到什么，影响范围和触发条件是什么。
2. 当前代码 / 系统链路接口、函数、状态、配置或运行路径如何一步步走到问题点。
3. 证据裁决E1/E2/E3 贴在链路节点上，区分成立、排除、待验证。
4. 根因 R# / 可疑点 H#根因必须落在主路径节点上，不能漂在说明区。
5. 修法 F#明确切断、替换、补偿或兜底哪一段故障链路。
6. 验证 / 回滚 V#验证命令、结果状态、回滚动作和剩余风险形成闭环。

功能迭代：从当前设计走到目标设计

用户要看懂当前功能和实现怎么工作，目标改变哪里，风险如何控制。

1. 当前功能主路径用户现在入口、动作、状态和结果是什么。
2. 当前开发实现链路前端入口、API、服务、DB/状态、中间件或异步任务。
3. 目标主路径改完后用户如何操作，体验和业务规则怎样变化。
4. 差异映射新增 / 修改 / 删除 / 不变 / 风险 / 回滚都落到对应节点。
5. 实现与发布闭环接口、DB、模块、权限、幂等、观测、灰度和回滚动作。
6. AC / 测试矩阵验收标准、关键用例、边界、失败态和未覆盖风险。
A. 轻补丁只补强调语句，成本低，但仍可能画成普通说明图。
B. 专用骨架 🌟新增两类固定骨架与测试，成本可控，最匹配当前问题。
C. 骨架 + 样板再追加示例 HTML，稳定性更强，但维护成本更高。
验收口径两类图都必须先画现状，再把证据、差异、测试、回滚贴到主路径。
