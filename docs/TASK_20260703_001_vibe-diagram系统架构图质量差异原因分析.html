不是“没有规则”

现有规则已经写了“不是组件清单”“不是卡片目录”，但没有把大众系统架构图的视觉模板固化成可执行约束。

不是“证据不重要”

证据重要，但应进入附录或点击详情；默认主图应先完成抽象分层、边界和流向。

不是“HTML 做不到”

HTML 可以内联 SVG、图标和版式模板；问题是当前生成路径允许 div 网格替代架构画布。

两种输出的核心差异

附件图强在哪里

  • 先给总体架构，再给 RAG、多 Agent、数据流变体，符合常见架构图阅读顺序。
  • 用户接入、应用层、能力层、数据层、基础设施、运营管理有稳定空间位置。
  • 图标承担类别识别，节点短文本只做补充。
  • 箭头表达系统边界、调用方向和反馈环，而不是只做层间装饰。

vibe-diagram 输出弱在哪里

  • 把“源码事实覆盖”优先级抬得过高，架构抽象被组件清单挤掉。
  • 候选全集让图像变成可切换报告，而不是一个定稿架构图。
  • 节点内容接近审计段落，读者需要逐卡阅读。
  • 缺少图标语义和经典架构图骨架，第一眼无法形成大众认知里的系统地图。
