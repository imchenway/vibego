# TASK_0010 并行工作区正常开发与正常提交修复

## 1. 背景

用户确认的目标：

1. 并行目录必须是可正常开发的真实工作区。
2. Codex 改动必须真实落入业务子仓库，而不是只形成根仓库空壳提交。
3. 当前分支场景下，源项目未提交改动要带入并行目录。
4. 有远端的仓库要能正常推送任务分支；无远端仓库不能再默认报 `origin` 错误。

仓库证据：

- `parallel_runtime.py`
  - 锚点：`prepare_parallel_workspace`
  - 旧行为：整目录复制后统一 `checkout -B <task_branch> <selected_ref>`，会把复制过来的脏工作区重置掉。
- `parallel_runtime.py`
  - 锚点：`commit_parallel_repos`
  - 旧行为：一律 `push_remote = repo.selected_remote or "origin"`。
- `parallel_runtime.py`
  - 锚点：`merge_parallel_repos`
  - 旧行为：一律 `fetch --all` + `(repo.selected_remote or "origin")`。

## 5. 最小实现

文件：`parallel_runtime.py`

变更：

1. 新增 `_current_branch_name(...)`
   - 读取当前本地分支
2. 新增 `_extract_status_paths(...)`
   - 从 `git status --short` 解析变更路径
3. 新增 `_resolve_push_remote(...)`
   - 优先取 branch remote / `selected_remote` / 唯一 remote
4. 新增 `_direct_child_repo_names(...)` / `_is_meta_only_status(...)`
   - 识别父仓库仅包含子仓库指针/共享目录元信息变化的场景
5. 调整 `prepare_parallel_workspace(...)`
   - `selected_ref == HEAD` 或 “所选本地分支 == 当前分支” 时，使用 `checkout -b` 保留脏工作区
   - 其它基线分支场景在 `checkout -B` 后补 `reset --hard` + `clean -fd`
6. 调整 `commit_parallel_repos(...)`
   - meta-only 仓库直接 `skipped`
   - 无 remote 仓库改为“已本地提交，跳过推送”
7. 调整 `merge_parallel_repos(...)`
   - 无 remote 仓库直接 `skipped`
   - 不再默认回退 `origin`

## 7. 用户可见结果

1. 当前分支场景下，新建并行目录会保留未提交改动。
2. 切到其它基线分支的新并行目录不会混入当前工作区脏改动。
3. 父仓库仅有嵌套 Git 指针变化时，不再生成误导性的空壳提交。
4. 真实业务子仓库即使未显式记录 `selected_remote`，只要仓库存在唯一 remote，仍可正常推送任务分支。
5. 无 remote 仓库不再报 `origin` 错误，会明确标记为本地提交/跳过自动合并。

## 8. 风险与回滚

### 8.1 风险

- 当前“meta-only” 判定基于：
  - 当前仓库下首层变更路径是否只落在嵌套仓库目录
  - 根仓库额外容忍 `docs` / `.idea`
- 若未来要把根仓库 `docs` 也作为正式交付物，需要单独细化判定规则。

### 8.2 回滚

- 回滚 `parallel_runtime.py` 中：
  - 当前分支保留脏工作区逻辑
  - meta-only 跳过逻辑
  - remote 解析逻辑
- 回滚 `tests/test_parallel_runtime.py` 五条新增测试
