# TASK_0010 并行工作区正常开发与正常提交修复

## 1. 背景

用户确认的目标：

1. 并行目录必须是可正常开发的真实工作区。
2. Codex 改动必须真实落入业务子仓库，而不是只形成根仓库空壳提交。
3. 当前分支场景下，源项目未提交改动要带入并行目录。
4. 有远端的仓库要能正常推送任务分支；无远端仓库不能再默认报 `origin` 错误。

仓库证据：

- `parallel_runtime.py`
  - 锚点：`prepare_parallel_workspace`
  - 旧行为：整目录复制后统一 `checkout -B <task_branch> <selected_ref>`，会把复制过来的脏工作区重置掉。
- `parallel_runtime.py`
  - 锚点：`commit_parallel_repos`
  - 旧行为：一律 `push_remote = repo.selected_remote or "origin"`。
- `parallel_runtime.py`
  - 锚点：`merge_parallel_repos`
  - 旧行为：一律 `fetch --all` + `(repo.selected_remote or "origin")`。

## 13. 补充修复：Git push 继承 worker 代理导致 HTTP 503

### 13.1 背景

用户进一步提供证据：

1. 在 IDEA 中直接对 `web-base` 执行 `git push` 可以成功。
2. 但 Telegram 按钮触发的“提交并行分支”会对同一仓库报：
   - `The requested URL returned error: 503`

运行期与现场证据：

- `~/.config/vibego/logs/codex/cckgpimbot/run_bot.log`
  - 锚点：`使用代理(https_proxy): http://127.0.0.1:8234`
- 现场复核：
  - 对 `TASK_0011/workspace/web-base` 执行
    - **清理代理后** `git push --porcelain ... --set-upstream` 成功
    - **注入 worker 代理后** 同一命令复现 `503`

结论：

- 不是分支问题，也不是 GitLab 服务本身不可用
- 是 `parallel_runtime.py` 内部 `git` 子进程继承了 worker 代理环境
