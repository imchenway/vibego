# TASK_20260605_001 task_id 变量名误触并行会话路由

## 1. 现象

- 用户在 `CckgWmsAppCoreBot` 普通直聊中发送：
  - `task_id + srcBatchStockId 不能用吗`
- Telegram 可见异常提示：
  - `/TASK_ID 当前没有活动中的并行会话。`
- 用户补充：tmux 中对应会话仍存在。

附件截图：`/Users/david/.config/vibego/data/telegram/vibegobot/2026-06-05/20260605_025202058-00933e793616.jpg`。

## 2. 当前只读取证结论

### 2.1 不是 tmux 主会话不存在

本机只读检查显示：

### 2.2 cckgwmsappcore 当前没有并行会话记录

因此“没有活动中的并行会话”不是在说主 tmux，而是在说 `parallel_sessions` 没有 `TASK_ID` 对应的活动并行会话。

### 2.3 真正触发点是 `task_id` 被误识别为任务前缀 `TASK_ID`

当前源码链路：

- `bot.py:25870-25875`
  - `on_text(...)` 会先调用 `_extract_task_prefixed_prompt(prompt)`。
  - 只要解析出任务前缀，就优先按“手动并行会话路由”处理。
  - 找不到活动并行会话时直接回复 `/{prefixed_task_id} 当前没有活动中的并行会话。`
- `bot.py:12707-12715`
  - `_extract_task_prefixed_prompt(...)` 会把第一段 token 交给 `_normalize_task_id(...)`。
- `bot.py:8886-8903`
  - `_normalize_task_id(...)` 会把首 token 转为大写并调用 `TaskService._convert_task_id_token(...)`。
- `tasks/service.py:1026-1040`
  - `_convert_task_id_token(...)` 兼容旧格式，会把 `TASKID/TASK_ID/TASK-...` 等统一成任务 ID 格式。
- `bot.py:5512`
  - 当前合法任务 ID 正则为 `^TASK_[A-Z0-9_]+$`，因此 `TASK_ID` 被认为合法。

本轮用当前代码复现：

```text
_extract_task_prefixed_prompt("task_id + srcBatchStockId 不能用吗")
=> ("TASK_ID", "+ srcBatchStockId 不能用吗")
```

## 3. 根因

这是“普通文本任务 ID 前缀路由”过宽导致的误判：

```text
普通业务变量名 task_id
  -> 首 token 被大写为 TASK_ID
  -> 满足 TASK_ID_VALID_PATTERN
  -> 被当成“发往 /TASK_ID 并行会话”的路由前缀
  -> parallel_sessions 中没有 TASK_ID 活动并行会话
  -> Telegram 回复 `/TASK_ID 当前没有活动中的并行会话。`
```

因此，用户看到的提示不是 tmux 主会话不存在，也不是当前主会话无法收消息，而是普通业务文本被错误拦截到并行会话路由。

## 5. 契约变更建议

### 推荐契约

普通文本中只有在“明确像用户要路由到任务并行会话”的前缀时，才触发并行路由：

1. `/TASK_0010 继续补充`：继续支持并行路由。
2. `TASK_0010 继续补充`：是否继续支持需二选一确认；若保留，至少必须要求数字型任务号后缀。
3. `task_id + srcBatchStockId 不能用吗`：必须作为普通模型 prompt，不能触发并行路由。
4. `TASK_ID foo`：不应被当作真实任务 ID，除非任务体系明确允许非数字任务编号。

### 推荐实现方向

- 在 `_extract_task_prefixed_prompt(...)` 层收窄“手动并行路由前缀”，不要直接复用过宽的 `_normalize_task_id(...)`。
- 新增一个专用判定函数，例如 `_normalize_parallel_route_task_prefix(...)`：
  - 优先要求前缀以 `/TASK_` 开头；或
  - 至少要求 canonical ID 匹配 `TASK_\d+(?:_\d+)?`，避免 `TASK_ID` 这种变量名误触。
- `_normalize_task_id(...)` 保持历史兼容，避免影响任务查询、旧链接与已有命令。

## 6. 测试矩阵

| 场景 | 输入 | 预期 |
|---|---|---|
| 业务变量名 | `task_id + srcBatchStockId 不能用吗` | 普通发送到主模型，不回复并行会话缺失 |
| slash 并行路由 | `/TASK_0093 继续补充` | 继续路由到 TASK_0093 活动并行会话，入模文本为 `继续补充` |
| 裸数字任务路由（若保留） | `TASK_0093 继续补充` | 继续路由，或按新契约转普通 prompt |
| 非数字占位符 | `/TASK_ID 继续` / `TASK_ID 继续` | 不应路由到并行会话；可按任务详情查询规则单独处理 |
| 普通 slash 命令 | `/task_show TASK_0001` | 不受影响 |
| 无活动并行会话 | `/TASK_0093 继续` 且 DB 无 running 会话 | 保持 fail-closed 提示 |

## 8. 风险与回滚

### 风险

- 收窄前缀后，用户如果习惯裸写 `TASK_0093 继续` 来路由并行会话，可能需要改为显式 `/TASK_0093 继续`。
- 如果历史任务 ID 允许非数字后缀，不能简单全局收紧 `_normalize_task_id(...)`，否则会影响任务查询。

### 回滚

- 回滚 `_extract_task_prefixed_prompt(...)` / 新 helper 的变更即可恢复旧行为。
- 不涉及 DB 迁移和外部依赖，回滚成本低。
