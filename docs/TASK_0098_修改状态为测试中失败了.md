## 5. TDD Gate（先红）

先补 3 条测试，覆盖：

1. callback 过期时，写库成功后仍继续确认消息 / 任务列表 / 按钮移除
2. 已是 `test` 时，callback 过期后仍继续任务列表 / 按钮移除
3. 自动任务列表发送失败时，不回滚“写库成功”

首次执行：

```bash
python3.11 -m pytest -q tests/test_task_description.py -k 'model_task_to_test_callback and (expired or task_list_send_fails or updates_status_even_if_callback_answer_expired or already_test_still_continues_when_callback_answer_expired or keeps_success_when_task_list_send_fails)'
```

结果：

```text
3 failed
```

首红原因符合预期：

- `on_model_task_to_test` 成功分支仍直接 `callback.answer("已切换到测试")`
- `on_model_task_to_test` 已是 `test` 分支仍直接 `callback.answer("任务已处于“测试”状态")`
- `_handle_task_list_request(...)` 抛 `TelegramNetworkError` 时未降级处理

## 6. 最小实现

仅修改 `bot.py:on_model_task_to_test`：

1. 成功分支与“已是 test”分支改为调用 `_answer_callback_safely(...)`
2. 状态确认消息发送失败时仅记录 warning，不中断主流程
3. 自动任务列表失败时仅记录 warning，不中断主流程
4. 无论确认消息 / 任务列表是否失败，仍继续尝试 `_try_remove_clicked_inline_button(...)`

### 6.1 最终业务口径

- **只要 `TASK_SERVICE.update_task(..., status="test")` 成功，就视为主流程成功**
- callback toast / 成功提示 / 任务列表 / 按钮移除均属于 best-effort 反馈层

## 8. 用户可见结果

现在点击模型答案消息底部的：

```text
🧪 任务状态更新为测试中
```

时：

1. 若任务原本不是 `test`
   - 只要写库成功，就按成功处理
   - callback 过期不会再把整条链路打成失败
   - 后续确认消息 / 自动任务列表 / 按钮移除会尽力执行
2. 若任务已是 `test`
   - callback 过期也不会阻断自动任务列表与按钮移除
3. 若任务列表发送异常
   - 不会回滚“状态已成功写库”的结果

## 9. 风险与边界
