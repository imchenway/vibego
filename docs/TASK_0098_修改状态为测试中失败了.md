## 11. 追加修复（2026-03-14）：批量推送到同一主会话后，多条返回消息串绑到最后一个任务

### 11.1 新增问题描述

用户追加反馈：

> 在批量任务场景里，一次发送了十条排队消息到同一主会话；十条消息回来后，依次点击“更新为测试中”，只有一条任务能正确改成测试中，而且更改成功后按钮需要消失。

### 11.2 根因定位

根因已在仓库内复现并定位为：

1. 原生主会话当前只有单值绑定：
   - `bot.py`（锚点：`SESSION_TASK_BINDINGS`、`_bind_session_task`）
   - 后一次批量推送会覆盖前一次任务绑定
2. 模型返回多条消息时，整批消息共用同一个 `bound_task_id` 构造按钮：
   - `bot.py`（锚点：`_deliver_pending_messages` 中 `bound_task_id = SESSION_TASK_BINDINGS.get(session_key)`）

因此，在“同一主会话 + 批量排队推送”场景下，多条返回消息底部的：

- `✅ 全部按推荐`
- `🧩 部分按推荐（需补充）`
- `⬆️ 提交分支`
- `🧪 任务状态更新为测试中`

都会串绑到最后一个任务。

### 11.6 最小实现

1. 新增 `SESSION_PENDING_TASK_BINDINGS`
   - 用于记录原生主会话里“尚未被最终答案消息消费”的任务队列
2. `bot._push_task_to_model(...)`
   - 对**原生主会话**成功推送后，追加一条待消费任务绑定
   - 并行会话逻辑保持不变
3. `bot._deliver_pending_messages(...)`
   - 不再在整批 events 开始前只生成一次 keyboard
   - 改为对每一条 `DELIVERABLE_KIND_MESSAGE` 单独解析：
     - 当前并行任务绑定，或
     - 原生主会话待消费任务队列的队首任务
   - 每条消息按各自任务生成独立 callback payload
   - 当消息成功发送，或因重复 hash 被明确跳过时，消费对应任务绑定，避免后续消息继续串绑
4. 运行时清理与运行中图标
   - 清理 session 级状态时同步移除待消费队列
   - 任务列表“运行中”图标补充读取待消费任务队列

### 11.8 最终行为（本轮）

现在在**批量推送到同一主会话**场景下：

1. 每条返回的模型答案消息，底部按钮都会绑定到各自原始任务
2. 逐条点击 `🧪 任务状态更新为测试中` 时，会分别更新对应任务
3. 哪条消息写库成功，哪条消息上的“测试中”按钮就消失
4. 若任务原本已是 `test`，或写库失败，该条消息按钮保留
