# /TASK_0072 并行执行任务问题

## 1. 需求结论（最终确认）

- 不改任务详情按钮布局，继续复用 `🚀 推送到模型`
- 点击后先选：
  - 当前 CLI 处理
  - 新建分支 + 新 CLI 并行处理
- 并行链路覆盖当前项目根目录下全部 Git 仓库
- 分支基线按仓库分别选，来源为本地 + 远端分支
- 并行目录与原工作目录隔离，不能影响原仓库当前分支
- 并行会话每次模型回复底部增加：
  - 来源任务按钮
  - 回复按钮（下一条消息自动补 `/TASK_xxxx` 前缀）
  - 提交并行分支按钮
- 提交成功后询问是否自动合并，并给“删除并行目录”选项
- 自动合并成功 / 失败后，都给“删除并行目录”选项

## 5. 实现摘要

### 5.1 交互链路

- `🚀 推送到模型` 前新增“当前 CLI / 并行 CLI”选择
- 并行会话回复底部新增：
  - `🏷 /TASK_xxxx 标题`
  - `↩️ 回复 /TASK_xxxx`
  - `⬆️ 提交并行分支`
- 回复按钮采用一次性前缀注入模式

### 5.2 并行运行时

- 新增 `parallel_runtime.py`
- 持久化表：
  - `parallel_sessions`
  - `parallel_session_repos`
- 为每个任务创建独立并行目录与独立 tmux/session pointer

### 5.3 提交与自动合并

- 提交范围：并行任务下所有有改动仓库
- commit message 改为任务语义：
  - `fix/feat/chore(TASK_xxxx): 标题`
- 提交成功后给：
  - `🔀 尝试自动合并`
  - `🗑️ 删除并行目录`
  - `稍后再说`
- 自动合并失败后仍允许删除并行目录

## 7. 风险与已知限制

- 当前并行目录删除为显式按钮触发，已加二次确认
- 默认主 CLI 与并行 CLI 共存，消息路由需依赖：
  - 显式 `/TASK_xxxx ...`
  - 或回复按钮触发的一次性前缀模式
- 当前未新增“关闭但保留”独立按钮；目录保留主要通过“不删除”实现
