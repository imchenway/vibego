系统架构：多小程序客服智能体的可读主线图

修正 010 的问题：不再把系统画成五列表格。读者只沿蓝色主线看 5 个阶段，控制、数据、兜底都贴在当前阶段旁边，作为解释而不是第二张矩阵。

一眼读法

多小程序入口进入 SDK，经 Gateway / Facade 到 Agent，Agent 调工具和知识后返回答案；任何阶段遇到低置信、关闭自助或工具不可用，都进入七鱼兜底。

蓝：唯一主线 琥珀：控制 / 兜底 绿：数据 / 知识 红：缺口

为什么比 010 更可读

010 同时展示 5 列，读者要横向逐格扫描。本版把泳道折回到阶段内部，先读主线，再看每段的控制、数据、兜底说明。

主线分段拓扑

每段只回答一个问题：入口如何进来、服务如何接住、Agent 如何决策、工具与知识如何支撑、结果如何闭环。

这次明确承认 010 的失败点

  • 五列同屏导致读者要横向扫表。
  • 控制、数据、兜底变成同权重信息面。
  • 主请求中轴不够强，第一眼不知道该从哪读。

本版采用的可读性原则

  • 先读 5 个主节点，再看每段的三类胶囊。
  • 侧向信息只服务当前阶段，不另起矩阵。
  • 证据默认折叠，关键结论仍静态可见。
