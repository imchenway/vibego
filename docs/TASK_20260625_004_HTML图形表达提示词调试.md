# TASK_20260625_004 HTML 图形表达提示词调试

## 1. 任务目标

沉淀一套可复用提示词，让 Codex 在收到作图、沟通复杂逻辑、设计定稿、故障排查、开发设计等需求时，优先输出**单文件 HTML 图形化表达**，并可导出 PNG 供 Telegram 快速预览。

目标不是“HTML 代替 PNG”，而是建立一套 **HTML 可视化沟通协议**：

- Codex 内可点击、可编辑、可迭代；
- Telegram 内优先用 PNG fallback 快速预览，若后续 HTML 文件/URL 预览稳定，再以 HTML 为主；
- 图形必须比普通 Markdown 更清晰，而不是把混乱节点搬进 HTML；
- 所有复杂图必须解决文字遮挡、溢出、箭头穿节点、路径混乱、主图过载问题。

## 3. 初始质量红线

以下任一问题出现，HTML 图不合格，必须重排后再交付：

1. 节点文字超出边框。
2. 节点互相遮挡。
3. 箭头穿过节点正文或关键标签。
4. 主路径无法在 10 秒内看懂。
5. 一张图塞入所有细节，没有主图/详情分层。
6. 只输出静态图片，没有可编辑源。
7. 没有 Telegram 可预览 PNG fallback。
8. 代码/架构事实没有证据锚点。
9. 故障图把假设画成结论。
10. 页面设计稿把功能清单直接平铺成卡片集合。

## 6. 子代理结果汇总

### 6.1 统一共识

六个子代理结论一致：HTML 的优势不是自动更美，而是**可点击、可编辑、可迭代、可截图 fallback**。要让 HTML 真正优于 PNG，必须把图形表达收敛为规则化布局，而不是自由摆节点。

统一共识：

1. 主图只表达核心路径，详细说明进入详情面板、折叠区、表格或附录。
2. 单文件 HTML，禁止外部依赖、CDN、构建工具。
3. 复杂图必须先裁剪信息：超过 25 个核心节点或 35 条主连线，必须拆分、分层或分页。
4. 图必须有静态可读版本，不能依赖点击后才理解核心关系。
5. PNG fallback 必须能表达标题、主路径、核心节点和图例。
6. 文字溢出、遮挡、箭头穿正文、节点重叠属于直接 fail 红线。
7. 所有事实型图必须带证据锚点；无证据只能写 TODO / 待验证。

### 6.2 图型分支结论

| 图型 | 最适合结构 | 核心规则 | 不适合方式 |
| --- | --- | --- | --- |
| 系统架构图 | 分层容器 + 节点摘要 + 右侧详情栏 + 正交连线 | 节点只放短名和职责，主/异步/失败/配置链路线型区分 | 自由散点布局、斜线乱连、所有细节塞节点 |
| 业务流程图 | 泳道/阶段 + 主路径 + 异常区 + 规则表 | 主路径 5-9 步，异常回流用“返回 Sx”，多条件规则表格化 | 状态机画成一团圆圈，多对多关系画蜘蛛网 |
| 代码时序图 | 时间轴 + 参与者列 + 每行一个主调用 | 7-12 个关键步骤，每步绑定源码证据，返回值单独成行 | 交叉箭头、异步回调用大弧线、每行源码都画箭头 |
| 故障排查图 | 三栏：证据链 / 假设树 / 根因修复验证闭环 | 现象 -> 影响 -> 证据 -> 假设 -> 根因 -> 修法 -> 验证；假设不能画成结论 | 复杂连线、隐藏未验证点、验证未跑却写已修复 |
| 页面设计稿 | 工作台/主路径原型 + 状态切换 + 详情/抽屉 | 先回答唯一目标、首层展示、收纳内容、卡片堆叠检查；展示空/错/加载态 | 功能清单卡片化、KPI 卡片矩阵、静态设计说明 |
| 质量评审 | 100 分 rubric + 红线 fail | 无红线且 ≥85 合格，≥92 优秀 | 只看视觉不看 Telegram/Codex/PNG 三端可读 |

### 6.3 质量红线

出现任一项，HTML 图不合格：

1. 节点文字溢出边框、被裁切、互相压住。
2. 节点、分组、浮层发生可见重叠。
3. 箭头穿过节点正文区，或箭头头部落在文字上。
4. 箭头标签压住节点、线条或其他标签。
5. 主图信息过载：核心节点超过约 25 个或主连线超过约 35 条，且没有分层/分页/聚合。
6. 主路径起点、终点、方向不明确。
7. 默认静态视图不可读，只能靠点击理解。
8. PNG fallback 丢失关键节点、箭头、图例或标题。
9. 事实型图没有证据锚点；假设被画成结论。
10. 页面设计稿把功能清单平铺成卡片集合。

## 7. 统一提示词 v0

下面是可直接复制给 Codex 的统一提示词。使用方式：放在用户“画图/出设计稿/解释逻辑/排查故障”的需求前或后，作为强约束。

```text
你要把用户需求转化为【单文件 HTML 可视化图】。HTML 是主交付，PNG 只是 Telegram 预览 fallback。不要输出静态图片、Mermaid、PlantUML、Graphviz、Figma 描述或纯 Markdown 说明，除非用户明确要求。

# 0. 总目标

生成一个可直接保存为 `.html` 打开、可点击、可编辑、可迭代、可截图导出 PNG 的单文件 HTML。

阅读环境：
- Codex：打开 HTML 阅读、交互、后续修改。
- Telegram：优先看 PNG fallback 或 HTML 预览截图。
- 浏览器：本地打开、截图、打印、归档。

# 1. 硬性输出要求

1. 输出完整单文件 HTML，包含 `<!doctype html>`、内联 CSS、少量原生 JS。
2. 不新增依赖，不使用外部 CDN、远程字体、图片资源、React、Vue、D3、Mermaid、Graphviz、PlantUML。
3. 图必须默认静态可读，交互只能增强理解，不能成为理解主路径的必要条件。
4. 复杂细节不得塞进主图节点，必须进入详情面板、折叠区、规则表、证据区或 TODO 区。
5. 所有事实型内容必须有证据锚点；没有证据写 `TODO`、`待验证` 或 `推断`，不能装成确定结论。
6. 必须提供 Telegram 可用 PNG fallback 方案：页面可截图、打印友好，或提供导出/截图说明。
7. 最终 HTML 内必须包含“自检结果”注释，列出红线是否存在、已知限制和建议截图尺寸。

# 2. 先判断图型

你必须先根据用户需求选择一种主图型，不要混画：

A. 系统架构图：模块、进程、服务、数据存储、外部系统、运行边界、主链路、旁路链路。
B. 业务流程图：角色、用户旅程、审批流、状态机、异常分支、规则矩阵。
C. 代码时序图：请求链路、事件链路、异步任务、回调、重试、幂等、事务边界。
D. 故障排查图：现象、影响、证据、假设、根因、修法、验证、风险、回滚。
E. 页面设计稿：页面主路径、布局层级、交互状态、空/错/加载态、响应式、可访问性。

如果材料同时包含多种内容：
- 只选一个主图型做主图；
- 其他内容放辅助分区或详情面板；
- 不要把所有信息堆进一张复杂大图。

# 3. 通用页面结构

HTML 页面必须包含：

1. 顶部摘要区：标题、图型、目标读者、核心结论、阅读顺序。
2. 主图区域：只展示主路径/核心结构。
3. 详情面板：点击节点、步骤、状态、证据或连线时展示完整信息。
4. 图例区：解释颜色、线型、节点类型、状态标签。
5. 辅助区：规则表、状态表、证据清单、异常清单、TODO & Known Issues。
6. 截图/打印友好区：保证 PNG fallback 包含标题、主图、图例、关键说明。

推荐 DOM：
- `<header class="topbar">`：标题、摘要、图例、状态/截图提示。
- `<main class="layout">`
  - `<section class="canvas">`：主图。
  - `<aside class="details-panel">`：详情。
- `<section class="supporting-sections">`：规则、证据、异常、TODO。
- `<script type="application/json" id="diagram-data">`：集中保存图数据，方便后续修改。
- `<script>`：原生 JS 交互。

# 4. 通用布局红线

生成前必须避免：

1. 文字溢出节点边框。
2. 文字被裁切、互相重叠或压住箭头。
3. 节点、分组、浮层重叠。
4. 箭头穿过节点正文区。
5. 箭头标签压住线条、节点或其他文字。
6. 主图节点超过 25 个或主连线超过 35 条仍不分层。
7. 主路径方向不明确。
8. 静态视图不可读，只靠点击才能理解。
9. PNG fallback 看不清标题、主路径、核心节点或图例。

如果出现以上任一问题，必须重排、拆图、分层或减少主图信息，不得交付。

# 5. 通用 CSS/可读性要求

必须包含或等效实现：

- `box-sizing: border-box;`
- `overflow-wrap: anywhere;`
- 合理 `line-height`，正文至少 1.35。
- 普通正文 ≥ 14px，关键标题 ≥ 16px。
- 节点内边距：水平 ≥ 12px，垂直 ≥ 10px。
- 节点高度随内容增长，禁止固定高度导致裁切。
- 小屏时允许主图分段或纵向排列，不允许整体缩小到不可读。
- 颜色不能作为唯一信息表达，必须有文字/标签/图例。
- 交互元素必须有 hover、focus-visible、active、disabled 样式。
- 支持 `prefers-reduced-motion` 降级。

# 6. 图型分支规则

## A. 系统架构图

使用：分层容器 + 节点摘要 + 右侧详情栏 + 正交连线。

规则：
- 分层优先：入口层、应用/编排层、运行时层、数据层、外部系统层。
- 每层有边界框和标题。
- 同层节点横向排列，跨层链路自上而下或左到右。
- 节点只放短名 + 一句职责；完整职责、输入、输出、依赖、风险进详情栏。
- 连线使用正交折线，从节点边缘到节点边缘。
- 主链路实线，异步链路虚线，失败/降级链路红/橙虚线，配置链路细点线。
- SVG 连线层必须在节点层下方，避免盖住文字。
- 如果无法保证连线避让，减少主图连线，改用编号关系表。

## B. 业务流程图

使用：泳道/阶段 + 主路径 + 异常区 + 规则/状态表。

规则：
- 主路径只保留 5-9 个核心步骤；超过 9 个步骤必须拆阶段。
- 每个步骤只写步骤编号、动词短句、关键结果。
- 角色/系统用泳道；每个步骤只能落在唯一泳道内。
- 异常分支放在主流程下方，回流用“返回 Sx”，不要画长回环箭头。
- 状态机必须额外提供状态流转表，不要画成一团圆圈。
- 多条件、多角色、多状态规则必须用规则矩阵表，不要画蜘蛛网。
- 不确定业务点进入 TODO & Known Issues。

## C. 代码时序图

使用：时间轴 + 参与者列 + 每行一个主调用。

规则：
- 时间从上到下，参与者从左到右。
- 参与者建议 3-6 个；超过 6 个必须合并为逻辑边界。
- 主路径只放 7-12 个关键步骤。
- 每一行最多一个主调用动作，箭头只在同一行水平移动。
- 返回值单独占下一行。
- 异步回调作为新的时间点展示，不画大弧线跨半张图。
- 同步调用、异步消息、返回值、错误、重试、幂等、事务边界必须有不同视觉编码。
- 每个步骤必须绑定源码证据：文件路径、函数名、调用点、输入、输出、异常。
- 事务边界用半透明区块标注 begin/commit/rollback；证据不足写 TODO。

## D. 故障排查图

使用：三栏结构：左证据链，中假设树，右根因/修法/验证闭环，底部风险与回滚。

规则：
- 必须覆盖：现象 -> 影响 -> 证据 -> 假设 -> 根因 -> 修法 -> 验证 -> 风险/回滚。
- 每条证据编号 E1/E2；每个假设编号 H1/H2；根因 R；修法 F；验证 V。
- 每个根因必须引用证据编号。
- 假设必须区分：成立 / 排除 / 待验证。
- 未验证点必须显式展示。
- 验证未执行或失败时，不得写“已修复”。
- 故障图尽量少用复杂连线，优先使用编号引用和折叠详情。

## E. 页面设计稿

使用：产品级单页 HTML 原型 + 主路径 + 状态切换 + 详情/抽屉。

生成 HTML 前必须先给 5 个极短设计判断：
1. 唯一核心目标是什么？
2. 用户当前最需要完成什么动作？
3. 首层必须展示什么？
4. 哪些内容应收纳到详情、抽屉、折叠区、二级 Tab 或更多菜单？
5. 是否存在卡片堆叠、标题废话、功能平铺？如何规避？

规则：
- 一个页面只服务一个主路径。
- 高频动作放首层，低频配置/历史/说明/详情收纳。
- 管理后台优先使用导航 / 工具栏 / 主工作区 / 上下文详情，不要 KPI 卡片矩阵。
- 必须展示默认、加载、空、错误态，必要时补成功、禁用、选中态。
- 必须覆盖 mobile ≤640px、tablet 641-1024px、desktop ≥1025px。
- 使用语义元素 button/a/input/select/details，禁止 div 假按钮。
- 支持 Tab、Enter/Space、Esc，focus 可见。
- 禁止把需求清单逐条翻译成卡片。

# 7. 交互规则

- 点击节点/步骤/状态/证据/连线时，详情面板必须更新。
- 当前选中项必须高亮。
- 详情面板内容必须可复制。
- 交互必须使用原生 JS 或 HTML 原生元素 `<details><summary>`。
- 无 JS 时仍应能阅读主图和主要内容。
- 不允许用复杂动画掩盖信息结构问题。

- 布局与空间：16 分。
- 文字可读性：16 分。
- 连线与箭头质量：16 分。
- 视觉编码一致性：10 分。

合格线：无红线且 ≥85。

# 9. 最终输出格式

如果用户要求直接生成文件：必须把 `.html` 文件本身发给用户；文件路径只能作为补充，不能只给路径。
如果当前平台不支持直接发送文件：必须输出完整 HTML 代码块，让用户可直接复制保存为 `.html`。
如需 PNG fallback：可同时发送 PNG，但不能替代 HTML 文件本身。
如果用户要求直接回复代码：只输出完整 HTML 代码块。
如果用户要求先评审方案：先给 5 行以内设计判断，再给 HTML。
不要输出冗长过程日志。
```

## 15. Telegram HTML 交付方式修正：从 Markdown 链接改为文件附件（2026-06-25）

### 15.1 现象

用户在 Telegram 中看到的不是可点击文件卡片，而是一段普通文本：

- `打开新版业务流程图 HTML`
- `!新版业务流程图预览`
- 后续 Markdown 说明

这说明上一次交付虽然在 Codex 里用了本地 Markdown 链接和图片预览语法，但 Telegram 展示层没有把 `.html` 作为文件附件发送。

### 15.2 影响

- 用户无法在 Telegram 中直接点击打开 HTML 文件。
- “直接发送 `.html` 文件”的 skill 目标没有真正达成。
- 后续若把设计定稿、系统架构、业务流程、故障排查都沉淀为 HTML 图，这个问题会反复出现。

### 15.3 根因

`bot.py` 已支持模型回复引用本地图片时自动补发 Telegram 图片或文件，但只覆盖 PNG/JPEG/WebP：

- 图片路径提取：`MODEL_RESPONSE_LOCAL_IMAGE_TOKEN_RE`、`MODEL_RESPONSE_MARKDOWN_IMAGE_RE`
- 图片收集：`_collect_model_response_local_images`
- 图片发送：`_send_model_response_local_images`

`.html/.htm` 文件没有同等的路径提取、白名单校验和 `send_document` 回传逻辑，因此 Markdown 链接只作为文本进入 Telegram。

### 15.4 修法

新增“模型回复本地 HTML 文件附件”链路：

1. 从模型最终回复中提取 `.html/.htm` 普通路径与 Markdown 链接路径。
2. 只允许发送当前项目根目录内的 HTML 文件。
3. 校验扩展名、文件存在性、MIME 与大小上限。
4. 文本发送成功后，补发 Telegram `send_document` 附件。
5. skill 交付铁律从“直接发送 HTML 文件”细化为“必须作为文件附件发送，Telegram 中必须看到文件卡片，禁止只发送 Markdown 链接”。

| 路径 | 影响 |

| `bot.py` | 新增本地 HTML 文件路径提取、项目内白名单校验、`send_document` 发送函数，并接入模型输出投递流程。 |

## 17. AGENTS 与 HTML 制图 skill 配合协议（2026-06-25）

### 17.1 目标

用户希望后续沟通默认通过 HTML 图完成：

1. 开发设计前能先看到现状、目标方案和前后差异。
2. 缺陷排查时能一眼看到完整链路、证据链和高亮根因节点。
3. 系统/功能理解能看到组件、依赖、中间件、DB、MQ/队列语义和关键调用链。
4. 方案定稿、需求评审、最终交付都能用一张可点击、可编辑、可在 Telegram 阅读的 HTML 图承载核心信息。

### 17.2 第一性原理结论

AGENTS 和 skill 不应该重复定义同一批画图细节，否则后续会出现规则冲突和维护漂移。

推荐分工：

| 层级 | 职责 | 不负责 |
| --- | --- | --- |
| `AGENTS-template.md` | 判断“什么时候必须用 HTML 图沟通”，把非琐碎任务默认路由到 `html-visual-communication`。 | 不写具体布局、颜色、节点、箭头和弹窗规则。 |
| `html-visual-communication` skill | 判断“画哪一种图、如何画、如何交付”，覆盖自动路由、视觉红线、交互、移动端和附件发送。 | 不决定所有任务是否必须画图；触发优先级由 AGENTS 定。 |
| `bot.py` HTML 附件链路 | 把模型回复里引用的项目内 `.html/.htm` 自动作为 Telegram 文件附件发送。 | 不理解图内容质量，只负责安全发送。 |

### 17.3 契约变更

| 场景 | 新契约 |
| --- | --- |
| 功能迭代 / 开发设计 | 必须先用 HTML 图展示现状、目标方案、前后差异、影响面、测试矩阵、风险与回滚。 |
| 缺陷排查 / 故障分析 | 必须用 HTML 图展示现象、影响、证据链、可疑节点、已确认根因、高亮根因节点、修法、验证与回滚；未验证只能标注为假设。 |
| 系统 / 功能理解 | 必须用 HTML 图展示组件、依赖、中间件、DB、MQ/队列语义、关键调用链或业务规则。 |
| 需求评审 / 方案选择 / 设计定稿 | 必须用 HTML 图展示问题、目标、选项、推荐方案、AC、待决策项和风险。 |
| 琐碎任务 / 用户明确不要图 | 可不生成 HTML，但不能把例外扩大到开发设计、排障、架构理解和定稿类任务。 |
| 最终交付 | 如果产出或修改了 HTML 图，必须作为 `.html` 文件附件发送；禁止只发 Markdown 链接或路径。 |
