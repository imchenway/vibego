# TASK_20260703_003 系统架构图通用模板预览

## 目标

基于用户确认“对味”的复刻版抽象出通用模板预览。当前重点不是把某个业务内容做成模板，而是比较两条稳定方向：

1. 宏观骨架模板：只固定整体画布、关系语法和视觉质量门禁，不固定层数、层名、模块数量和区域定义。
2. 多 archetype 模板选择器：预置多个纯画法骨架，生成时先按材料选择，再填入真实节点。

## 2026-07-06 三次反馈修正

用户指出：即使使用可变槽位，模板仍然会把每层内容限制得太死；最终多少层、多少模块、每层内部多少元素、每层定义是什么，都不应该由模板固定。

本次改为双方案样板：

### 方案一：宏观骨架模板

- 固定：系统边界、主结构区、可选关系区、箭头线型、节点自动居中、无横向滚动、无文字溢出。
- 放开：层数、层名、模块数、区域数量、区域位置、是否存在外部区/旁路区/依赖区。
- 适用：用户明确反感固定层定义，或材料不足以判断架构形态时。

### 方案二：多 archetype 模板选择器

内置以下纯画法骨架，由材料选择：

1. 分层拓扑型
2. 主链路中轴型
3. 内外边界型
4. 网状依赖型
5. 多平面协作型
6. 阶段轨道型

选择器只决定画法，不决定业务内容；所有层名、节点名和模块数量仍必须来自材料或标注待确认。

## 2026-07-06 十一次反馈修正：非系统图型模板 DOM 骨架差异化

用户指出“每个类型里的多个模板图都一样，只是文案不一样”。复核后确认反馈成立：上一轮虽然给 10 类非系统图型补了 42 个 HTML 模板，也写入了不同 `data-template-layout`，但生成器主体仍大量复用同一段 `slot + arrow + slot + arrow` DOM 骨架，视觉上很容易表现为同一个卡片流程换文案。

### 根因

- 测试只校验 `data-template-layout` 是否唯一，没有校验去掉文案后的主画布 DOM 结构是否不同。
- 生成器把“模板种类”落实成 CSS 类名和文案差异，没有落实成不同图形语法。
- reference 写了“不能共享宏观骨架”，但没有明确到“不能共享同一 DOM 骨架换文案”。

### 修正

- 新增 `_template_structure_signature()`：去掉 `data-slot` 名称、`<b>` / `<span>` 文案和 `{{...}}` 占位后，对主画布 DOM 结构做签名。
- 新增测试 `test_vibe_diagram_non_system_html_templates_have_distinct_dom_skeletons`，要求同一图型内所有模板结构签名不重复。
- 重写 42 个非系统图型模板：
  - 流程类改为 BPMN 事件 / 网关 / 泳道 / 异常旁路。
  - 时序类改为 lifeline、event bus、transaction box、retry loop。
  - 状态数据类改为状态机、实体盒、生命周期、数据处理框、状态事件矩阵。
  - 故障类改为排障时序、因果链、before/after、断点图。
  - 页面类改为 artboard、filmstrip、响应式状态板、页面流。
  - 技术设计、决策、验收类分别改为拓扑、泳道、边界、决策树、象限、账本、看板、时间线。
- 10 个非系统图型 reference 均补充：同一图型内模板不能共享同一 DOM 骨架换文案。
