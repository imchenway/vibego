# TASK_20260703_003 系统架构图通用模板预览

## 目标

基于用户确认“对味”的复刻版抽象出通用模板预览。当前重点不是把某个业务内容做成模板，而是比较两条稳定方向：

1. 宏观骨架模板：只固定整体画布、关系语法和视觉质量门禁，不固定层数、层名、模块数量和区域定义。
2. 多 archetype 模板选择器：预置多个纯画法骨架，生成时先按材料选择，再填入真实节点。

## 2026-07-06 三次反馈修正

用户指出：即使使用可变槽位，模板仍然会把每层内容限制得太死；最终多少层、多少模块、每层内部多少元素、每层定义是什么，都不应该由模板固定。

本次改为双方案样板：

### 方案一：宏观骨架模板

- 固定：系统边界、主结构区、可选关系区、箭头线型、节点自动居中、无横向滚动、无文字溢出。
- 放开：层数、层名、模块数、区域数量、区域位置、是否存在外部区/旁路区/依赖区。
- 适用：用户明确反感固定层定义，或材料不足以判断架构形态时。

### 方案二：多 archetype 模板选择器

内置以下纯画法骨架，由材料选择：

1. 分层拓扑型
2. 主链路中轴型
3. 内外边界型
4. 网状依赖型
5. 多平面协作型
6. 阶段轨道型

选择器只决定画法，不决定业务内容；所有层名、节点名和模块数量仍必须来自材料或标注待确认。

## 2026-07-06 五次反馈修正：调研基准 v5

用户要求继续调研并完善系统架构图模板。本次基于外部资料把模板从经验归纳版升级为来源对齐版。

### 主要来源

- C4 Model：System Context、Container、Component、Deployment，以及 Dynamic / System Landscape 等支持视图。
- Microsoft Azure Well-Architected：架构图实践、图型选择、方向箭头、图例、官方图标、渐进披露，以及 Context / High-level system / Component / Deployment / DFD / Sequence / Network / Resilience / Identity 等图型说明。
- arc42：Building Block View、Runtime View、Deployment View。
- AWS / Google Cloud 官方图标与参考架构中心：技术特定部署图应使用最新官方图标，且图标不能替代语义和图例。
- Mermaid Architecture：架构图基础图元可抽象为 group、service/node、edge、junction。

### v5 模板集

1. 上下文 / 范围模板
2. 容器 / 工作负载总览模板

### 选择器原则

- 默认从“上下文 / 范围”或“容器 / 工作负载总览”开始。
- 只有用户问题明确指向部署、数据、网络、身份、韧性或单一运行场景时，才切换到专项模板。
- 任何模板都不得固定层名、业务模块、图标或技术栈；所有内容必须来自材料或标注待确认。
- 如果一个模板需要强行脑补才能填满，必须降级为更宽的上下文或容器总览。
