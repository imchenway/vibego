# TASK_20260310_004 原生会话与并行会话并存时的串会话与消息丢投递修复

## 1. 背景

用户反馈：

1. 同一 Telegram chat 内同时存在一个原生会话和一个并行会话
2. 点击并行消息底部 `✅ 全部按推荐` 后，内容被错误发送到原生 CLI
3. 点击 `↩️ 回复 /TASK_0093` 后，提示未找到并行会话
4. 原生会话后续有一条模型回复没有回推到 Telegram

## 2. 根因结论

### 2.1 chat 级单例 watcher / session 造成会话互相覆盖

- `bot.py` 旧逻辑仅维护：
  - `CHAT_SESSION_MAP: Dict[int, str]`
  - `CHAT_WATCHERS: Dict[int, asyncio.Task]`
- `_dispatch_prompt_to_model(...)` 在新 prompt 入模时会先取消当前 chat 的旧 watcher
- 结果：同一 chat 下原生会话与并行会话无法稳定并存

### 2.2 并行按钮只依赖 task_id，缺少 session-scoped 路由

- 并行消息底部按钮旧逻辑使用 `task_id` 作为 callback payload
- `on_model_quick_reply_all(...)` 若查不到并行会话，会静默回退到默认分发链路
- 导致“并行按钮串到原生 CLI”

### 2.3 回复模式依赖查库，丢失实时上下文

- `on_parallel_reply_callback(...)` 旧逻辑只按 `task_id` 查活动并行会话
- 若活动会话记录缺失或已切换，点击老消息按钮就会提示“未找到活动中的并行会话”

## 6. 最小实现

### 6.1 并行按钮路由升级为 session-scoped

- 新增：
  - `ParallelCallbackBinding`
  - `PARALLEL_CALLBACK_BINDINGS`
  - `PARALLEL_SESSION_CONTEXTS`
  - `PARALLEL_TASK_SESSION_MAP`
- 并行消息底部按钮 callback payload 改为：
  - `TASK_xxxx:<token>`
- 这样按钮点击时能优先命中原始并行上下文，而不依赖当前 chat 的默认 session

### 6.2 并行按钮 fail-closed

- `on_model_quick_reply_all(...)`
- `on_model_quick_reply_partial(...)`
- 若并行上下文已失效：
  - 直接提示“并行会话已失效，请在最新并行消息中重试”
  - 不再静默回退到原生 CLI

### 6.3 回复模式改为携带并行上下文

- `on_parallel_reply_callback(...)`
  - 优先使用按钮绑定的 dispatch_context
- `_set_parallel_reply_target(...)` / `_consume_parallel_reply_target(...)`
  - 改为携带 `task_id + dispatch_context + token`
- `on_text(...)`
  - 回复模式优先使用缓存的并行上下文，不再强依赖查库

### 6.4 并行 watcher 与原生 watcher 解耦

- 原生会话仍使用：
  - `CHAT_SESSION_MAP`
  - `CHAT_WATCHERS`
- 并行会话新增：
  - `PARALLEL_TASK_WATCHERS`
  - `_watch_parallel_and_notify(...)`
  - `_start_parallel_task_watcher(...)`
- 并行 dispatch 不再覆盖 chat 当前原生会话映射

## 8. 风险与边界

1. 历史旧消息上的“并行按钮”若来自 bot 重启前，token 绑定可能已丢失：
   - 现逻辑会优先按 token 查
   - 查不到时再按 `task_id` 尝试活动并行会话
   - 若仍失败，会 fail-closed 提示重试，不会串到原生 CLI
2. 本次仅修复“会话精确路由与 watcher 并存”问题，不改动数据库 schema。
3. `bot.py` 仍有与本任务无关的 `DeprecationWarning: invalid escape sequence '\*'`，未纳入本次范围。
