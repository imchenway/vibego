# TASK_20260625_006 普通直聊移除强制 PLAN

## 1. 背景与目标

- 现象：Telegram 普通直聊消息进入 worker 后，会在正文发送/排队前自动确保 Codex 处于 PLAN 模式。
- 用户目标：普通直聊只投递用户业务提示，不再隐式发送 `/plan`；PLAN 由 `🧭 PLAN:` / `/plan_mode` 或 Codex PlanConfirm 显式控制。
- 第一性原理：普通直聊是“输入投递通道”，不应偷偷改变模型协作模式；协作模式切换必须是用户可见、可控的动作。

## 2. 受影响目录与边界

| 范围 | 影响 | 说明 |
| --- | --- | --- |
| `bot.py` | 有影响 | `_handle_prompt_dispatch(...)` 普通直聊 dispatch 参数调整。 |
| `tests/test_task_description.py` | 有影响 | 调整普通直聊行为测试，按 TDD 先红后绿。 |
| `AGENTS.md` | 有影响 | 更新 Facts Table，避免后续实现继续按旧自动 PLAN 事实执行。 |
| `docs/` | 有影响 | 新增本任务文档，并保留历史文档作为历史语境。 |
| DB / SQLite | 无影响 | 本次不新增表、字段、索引，不涉及迁移。 |
| Telegram Bot API 契约 | 无影响 | 不改变消息收发 API；只改变进入 tmux 前的本地 dispatch 参数。 |
| tmux / Codex TUI | 局部影响 | 普通直聊不再自动发送 `/plan`；显式 PLAN 入口仍可发送 `/plan`。 |

## 3. 契约变更

### 3.1 修改前

普通 Telegram 直聊进入 `_handle_prompt_dispatch(...)` 后：

- `intended_mode=PUSH_MODE_PLAN`
- `force_plan_switch_for_queued=True`
- `probe_plan_mode_before_switch=True`
- `send_mode=_resolve_business_prompt_send_mode()`
- `confirm_delivery=True`

效果：即使正文按 queued 发送，也会先尝试让 Codex 进入 PLAN。

### 3.2 修改后

普通 Telegram 直聊进入 `_handle_prompt_dispatch(...)` 后：

- 不传 `intended_mode`
- 不传 `force_plan_switch_for_queued`
- 不传 `probe_plan_mode_before_switch`
- 保留 `send_mode=_resolve_business_prompt_send_mode()`
- 保留 `confirm_delivery=True`

效果：普通直聊不改变终端模式；默认仍按业务提示 queued 投递，并保留 JSONL 消费确认。

### 3.3 不变契约

- `/plan_mode`、底部 `🧭 PLAN:` 按钮仍负责显式模式切换。
- PlanConfirm 的 `Implement this plan` 仍负责从 Plan 收口进入实现动作。
- `_dispatch_prompt_to_model(...)` 仍支持显式 `intended_mode=PLAN`，供非普通直聊入口复用。
- 不支持 queued 的模型仍按既有逻辑回退 immediate。

## 6. 风险与回滚

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| 用户误以为普通直聊仍自动进 PLAN | 可能在普通消息中直接要求开发，模型若当前不是 Plan 就会按当前模式执行 | 通过 `🧭 PLAN:` / `/plan_mode` 明确模式；PlanConfirm 仍处理计划收口。 |
| 排队语义被误删 | 可能出现忙碌时投递失败或覆盖 | 本次保留 `_resolve_business_prompt_send_mode()` 与 `confirm_delivery=True`。 |
| 历史文档冲突 | 后续 agent 读旧任务文档可能误判 | 更新 `AGENTS.md` Facts Table 指向本任务为最新事实。 |

回滚方式：恢复 `_handle_prompt_dispatch(...)` 中 `intended_mode=PUSH_MODE_PLAN`、`force_plan_switch_for_queued=True`、`probe_plan_mode_before_switch=True`，并恢复对应测试期望。
