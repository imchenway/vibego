系统架构：多小程序客服智能体的泳道式拓扑

按用户选择的 B 方案中度重排：保留北向南层级，但把控制面、数据/知识面和七鱼兜底面拆成泳道；第一眼沿蓝色主请求中轴读，侧边只看约束、依赖和兜底。

一眼读法

用户从多小程序入口进入 Miniapp SDK，再到 Gateway / Facade，进入 Agent 编排并返回答案；控制面只约束主链路，数据/知识面只提供依赖，七鱼兜底是右侧贯穿 rail。

蓝：主请求中轴 琥珀：控制 / 兜底面 绿：数据 / 知识面 红：缺口 / 待接入

B 方案的边界

这不是展开实现清单；Controller、Facade、Handler、代码路径仍进入节点详情。泳道只用于区分系统平面，避免所有组件同权重平铺。

泳道拓扑总览

从左到右是控制面、主请求中轴、数据/知识面、兜底面；从上到下是入口、接入、服务、Agent、工具、状态与观测。

控制面泳道开关、权限、策略、风险

数据 / 知识面泳道读写、检索、观测
兜底面 rail人工 / 降级边界
Layer 01外部入口用户侧与宿主边界
Layer 02接入层SDK 与 REST 协议
控制不在此层展开接入层只负责协议，策略留到服务层约束。

Layer 06状态 / 观测RDS、向量、日志
审计与风险控制面可读取运行状态与日志告警，但不直接承载数据。
同步语义主链路 HTTP REST；无独立 MQ 时必须显式标注。
控制语义Feature Flags + Guard 作为侧向约束，不抢主请求中轴。
缺口语义Adapter / RAG / OpenAPI 未接入用红色角标表达。
追踪语义traceId 贯穿 SDK、Gateway、Agent、store 与日志。

主请求中轴

蓝色纵向节点是第一阅读路径；用户无需读侧边泳道也能理解主流程。

平面分离

控制面、数据/知识面、兜底面拆开后，Feature Flag、RAG、七鱼不会被误读为同层业务节点。

B 方案收益

  • 比 007 更容易分清主链路、控制策略、数据依赖和人工兜底。
  • 复杂系统的“面”比“层”更清楚，适合存在多个跨层能力的架构。
  • 运行语义集中在底部状态条，节点正文更聚焦。

B 方案风险

  • 桌面宽度利用更多，移动端需要降级为逐层纵向阅读。
  • 如果真实系统很简单，泳道会显得过度设计。
  • 仍不能把实现类、源码路径和证据重新铺回主画布。
