# TASK_20260624_008 /upgrade 成功态瘦身并自动项目列表

## 1. 背景与现象

用户在 Telegram 执行 `/upgrade` 后看到两类问题：

1. 成功路径消息包含大量 pipx 输出、重启日志路径和重启日志尾部，阅读成本高。
2. master 已重新上线后没有自动推送项目列表，需要用户手动点“项目列表”。

现场日志显示：`_notify_restart_success` 与 `_notify_start_signal` 在升级报告存在时会跳过项目列表，这是上一轮为避免重复项目列表做的修复；但 `_notify_upgrade_report` 只发送最终升级报告，没有补发项目列表。

## 2. 根因

- 噪音根因：`TASK_20260624_005` 为解决“最终态不明确”，把成功态也设计成包含 pipx 输出和重启日志尾部；这对排障有用，但对正常成功路径是垃圾信息。
- 项目列表缺失根因：`TASK_20260624_006` 为消除重复列表，升级报告存在时抑制了 `restart_signal` 和 `start_signal` 对应的项目列表；但唯一剩下的 `_notify_upgrade_report` 没有承担“成功后刷新项目列表”的职责。

## 3. 受影响目录

| 目录/文件 | 是否影响 | 说明 |
| --- | --- | --- |
| `master.py` | 是 | 调整 `/upgrade` 成功态文案；成功最终报告发送后自动推送项目列表。 |
| `tests/test_master_update_notifications.py` | 是 | 增加成功态无日志垃圾、最终报告后项目列表自动推送的回归断言。 |

| `bot.py` / worker | 否 | 本次只影响 master 端升级通知与项目列表。 |
| SQLite / 配置结构 | 否 | 无数据结构变更。 |

## 4. 契约变更

### 4.1 成功路径

- 进行中消息只展示步骤、动作、状态，不展示命令和 stdout 尾部。
- pipx 成功、即将重启的中间态只展示：pipx 阶段完成、耗时、即将重启、新 master 上线后会发送最终结果和项目列表。
- 新 master 最终成功态只展示：升级完成、版本、pipx 耗时、正在刷新项目列表。
- 最终成功态发送成功后，`_notify_upgrade_report` 自动调用 `_send_projects_overview_to_chat` 推送一次项目列表。
- 项目列表推送失败不保留升级报告，避免下次启动重复刷最终报告；失败只写日志，用户仍可手动 `/projects`。

### 4.2 警告/失败路径

- pipx 失败仍展示失败步骤、命令、最近输出，便于排障。
- 重启日志未确认 master 启动时仍保留重启日志路径和尾部日志，避免误报成功。
- 最终升级报告发送失败仍保留 `upgrade_report.json`，下次启动重试。

## 6. 实施顺序

1. 调整测试，先确认当前成功态仍包含日志垃圾，且最终报告后没有项目列表推送。
2. 修改 `master.py`：成功路径瘦身；最终报告成功后自动推送项目列表。
3. 保留失败/警告诊断信息，不削弱排障能力。
4. 更新 `AGENTS.md` 当前契约。
5. 运行聚焦测试和基础门禁。

## 7. 风险与回滚

- 风险：成功态不再直接展示日志，用户排查成功路径细节需要查看本地日志文件。
  - 接受原因：成功路径应面向行动，不应把 stdout 当用户界面。
- 风险：最终报告成功但项目列表发送失败时，用户没有自动列表。
  - 缓解：日志记录失败；用户仍可手动 `/projects`。不保留升级报告，避免下次启动重复刷升级完成。
- 回滚：恢复 `_build_upgrade_report_message()` 与 `_announce_upgrade_completion()` 的日志尾部展示，并移除 `_notify_upgrade_report()` 的项目列表发送即可。
