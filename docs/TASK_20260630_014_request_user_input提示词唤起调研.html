调研：提示词不能直接唤起 Codex 原生 request_user_input 提问器

本页只回答“能否通过 AGENTS/提示词让 Default mode 自动弹出类似 Plan mode 的点击提问器”。本轮不修改 AGENTS。

结论：不能只靠提示词稳定唤起。

在当前 Codex Desktop 会话里，request_user_input 是宿主按协作模式暴露的工具能力，不是 AGENTS.md 或普通 prompt 能自行开启的 UI 能力。提示词最多能要求“如果工具可用就必须用；不可用就停下来用单个阻塞问题等待用户确认”。

证据链

  • 当前会话实测：在 Default mode 调用 request_user_input 返回 “request_user_input is unavailable in Default mode”。这说明工具暴露受当前协作模式控制。
  • 当前工具契约：开发者上下文声明该工具 “only available in Plan mode”，Default mode 下不可用。
  • 官方 Codex 文档：Plan mode 被定义为让 Codex 先收集上下文、提澄清问题、形成计划再实现；入口是 /plan 或 Shift+Tab，而不是 AGENTS prompt 自动开启。
  • 官方 App Server 文档：存在实验性的 tool/requestUserInput，用于让客户端向用户展示 1-3 个短问题；但文档没有说明普通 prompt 可以把该工具注入当前 turn。
  • 官方配置文档：只看到 Plan mode 的推理强度配置 plan_mode_reasoning_effort，未看到“Default mode 自动开启 Plan 提问器”的配置项。

可行路径对比

不可行

AGENTS 直接召唤提问器

仅通过提示词写“必须使用 request_user_input”，在工具未暴露时会失败；模型不能改变当前协作模式或工具列表。

可行

能力感知门禁

AGENTS 规定：若提问器可用，必须用点击确认；若不可用，必须停止执行并问一个阻塞问题，等用户确认后再继续。

需产品/入口支持

默认进入 Plan-like 模式

如果要真正每次都有原生点击提问器，需要 Codex 入口层支持默认 Plan mode，或由客户端/桥接层显式提供 requestUserInput 能力。

推荐落地设计

用户提出需求
默认认为尚未授权实施。
→
能力检查
如果当前工具列表有 request_user_input，就用提问器；否则不执行，只问一个阻塞问题。
→
用户确认下一步
确认后才进入方案、计划、实现或验证；未确认则停在对话阶段。

这能接近 Plan mode 的协作效果，但不能把 Default mode 伪装成 Plan mode。

官方来源

  • Codex Best Practices：Plan mode 用于先收集上下文、提澄清问题、形成计划。
  • Codex App Server：列出 tool/requestUserInput 与 collaborationMode 等接口事实。
  • Codex Configuration Reference：只发现 Plan mode 推理强度配置，未发现 Default mode 自动开启提问器配置。
  • Codex Changelog：提到 Plan mode questions 的通知改进，进一步说明这是 Plan mode 问题流的一部分。
