## 2. 用户场景与业务目标

用户从 vibego Telegram Worker 的 `📟 命令管理` 触发微信小程序真机调试，希望在手机端无需扫描另一个屏幕上的二维码，即可让同账号微信进入真机调试，并让 Telegram 只在调试连接已经建立后显示成功。

业务目标：

1. 在默认通用命令中新增 `wx-remote-debug`，标题为“启动微信自动真机调试”。
2. 使用微信公开的 `miniprogram-automator` SDK，不伪装成不存在的 CLI/HTTP 真机调试命令。
3. 默认调用 `miniProgram.remote(true)`，适配 Telegram 移动端，无需扫码。
4. 成功口径严格 fail-closed：只有 SDK 收到真机调试连接事件、`remote(true)` Promise 正常完成且运行时探测通过，才返回成功。
5. 不承诺强制进入“真机调试 2.0”；公开 SDK 未提供这一保证。

## 5. 官方能力裁决

### 5.1 已确认

- 微信 CLI V2 没有 `remote-debug` / `real-device-debug` 子命令。
- 微信 HTTP V2 没有 `/v2/remote-debug` 接口。
- 官方公开的自动真机调试入口是 `miniprogram-automator` 的 `miniProgram.remote(auto?)`。
- `remote(true)` 自动调起同账号手机，无需扫码；Promise 在收到真机调试连接事件后完成。
- `cli auto --auto-port` 用于启动 automation WebSocket；`automator.connect({wsEndpoint})` 连接该 WebSocket；IDE HTTP 服务端口仍通过 CLI 全局 `--port` 参数传递，二者不能混用。

官方依据：

- <https://developers.weixin.qq.com/miniprogram/dev/devtools/cli.html>
- <https://developers.weixin.qq.com/miniprogram/dev/devtools/http.html>
- <https://developers.weixin.qq.com/miniprogram/dev/devtools/auto/remote.html>
- <https://developers.weixin.qq.com/miniprogram/dev/devtools/auto/miniprogram.html#miniProgram-remote>
- <https://developers.weixin.qq.com/miniprogram/dev/devtools/auto/automator.html#automator-launch>

### 5.2 明确边界

- `preview`、`auto-preview`、`upload` 都不能代替真机调试。
- `cli auto` 只开启自动化 WebSocket，仍需 SDK 调用 `miniProgram.remote(true)`。
- 官方未声明 `miniProgram.remote(true)` 强制使用真机调试 2.0，因此本需求不把“强制 2.0”作为 AC。
- `remote()` 没有公开扫码/连接超时参数，vibego 包装层必须自行增加超时。

## 7. 推荐设计

### 7.1 用户主路径

1. 用户在 `📟 命令管理` 点击 `wx-remote-debug`。
2. bot 扫描小程序目录：单候选自动选择；多候选让用户选择。
3. bot/脚本复用当前 IDE HTTP 服务端口配置与“唯一候选端口自动重试一次”规则。
4. 首次使用时，将随 vibego 包发布的锁定 npm 清单复制到 Vibego runtime root 下的 `wx-remote-debug/0.12.1/` 临时目录（默认 runtime root 为 `~/.config/vibego/runtime/`，支持 `VIBEGO_RUNTIME_ROOT`）；执行 `npm ci --omit=dev --ignore-scripts --no-audit --no-fund`，成功后原子 rename，失败清理临时目录。配置根优先级与 worker 保持一致：`MASTER_CONFIG_ROOT` → `VIBEGO_CONFIG_DIR` → `XDG_CONFIG_HOME/vibego` → 默认目录，并展开 `~/`。
5. Node 执行器在全局端口分配锁内，从 `19420-19519` 选择空闲 automation WebSocket 端口；若动态端口在 `cli auto` 启动阶段返回明确占用证据，则释放旧租约、换端口且仅重试一次。允许 `WX_AUTOMATION_PORT` 显式覆盖，但显式端口冲突不自动换端口，也不得把 WS 端口写入 `wx_devtools_ports.json`。
6. 显式执行 `cli auto --project <path> --auto-port <ws-port> --port <ide-http-port>`，最长等待 automation WS 就绪 45 秒。
7. 调用 `automator.connect({wsEndpoint})`，再调用 `miniProgram.remote(true)`，最多等待真机连接 120 秒。
8. `remote(true)` 完成后，在 10 秒内重试 `systemInfo()` 做运行时探测；返回非空对象且包含可识别的 `platform` 或 `system` 字段才算通过。连接事件和运行时探测必须同时成立，才输出结构化成功标记。
9. 最终调用 `miniProgram.disconnect()`、终止并等待本轮 `cli auto` 直接子进程、释放端口与项目锁；禁止调用 `miniProgram.close()`、`cli close` 或 `cli quit`，避免关闭用户开发者工具窗口。
10. Telegram 成功态专门解析结构化标记并显示“真机已连接且运行时探测通过”；项目、平台与连接证据进入成功文案和命令历史。

### 7.2 失败与异常

以下任一情况必须非零退出，禁止显示成功：

- `node` / `npm` 不存在；
- 首次依赖安装失败或离线；
- 小程序目录不存在；
- IDE 服务端口缺失/不匹配且自动重试与人工恢复均失败；
- 开发者工具未登录；
- 同账号手机未满足自动调起条件或微信未保持前台；
- SDK 启动、编译、上传或远程连接失败；
- 120 秒内没有收到真机调试连接事件；
- 连接事件成立但 10 秒内运行时探测失败。

连接超时只断开 automation WebSocket，不主动关闭开发者工具；由于官方没有公开取消真机调试准备态 API，Telegram 需明确提示“连接未确认，开发者工具可能仍停留在准备态，可手动关闭”。

超时预算：CLI/WS 启动 45 秒、自动真机连接 120 秒、运行时探测 10 秒；默认命令总 timeout 保持 600 秒，为首次依赖安装和清理留出余量。

### 7.3 重复触发

- 使用项目 realpath 哈希生成非阻塞排他锁，阻止同一项目的并发触发；第二次立即失败，不排队、不重启 IDE。
- 不同项目允许并行；automation WS 端口通过全局端口分配锁避免冲突，冲突时最多重新选择一次。
- 锁只保护“命令正在等待连接”的阶段；公开 SDK 没有可靠的“已有真机调试会话查询”接口，因此无法阻止用户在命令结束后再次触发。
- 重复触发限制属于已知边界，不把推断包装成已解决事实。

## 8. 依赖、打包与运行期策略

这是本需求的高风险变更：新增官方 Node 包 `miniprogram-automator`。

推荐策略：

- 锁定当前已核实版本 `miniprogram-automator@0.12.1`，使用专用 `package.json` + `package-lock.json`，不复用仓库根部历史 Node 依赖。
- 清单与 Node 执行器放入 `vibego_cli/data/wx-remote-debug/`，随 wheel/sdist 发布。
- 要求 Node.js `>=16`，执行器会显式校验；首次触发才原子安装到 Vibego runtime root 下的 `wx-remote-debug/0.12.1/`，不拖慢 `vibego start`，不污染项目目录和 pipx venv。
- 通过 package-lock 哈希判断是否需要重装；离线或完整性失败时 fail-closed。

收益：适配 pipx/PyPI 安装形态，版本可复现，升级与回滚边界清晰。

成本：首次运行需要 npm 网络下载并占用本地磁盘；npm registry 或代理异常会导致命令失败。

## 12. 风险与回滚

### 12.1 风险

1. 新增 npm 依赖和构建资产，可能受 registry、代理、Node 兼容性影响。
2. 首次运行需要网络安装，耗时可能显著高于后续运行；安装过程必须使用临时目录 + 原子 rename，避免半安装缓存。
3. 官方 SDK 未暴露“强制 2.0”“取消准备态”“查询已有会话”的稳定接口。
4. `remote(true)` 依赖开发者工具登录状态、同账号手机与微信前台状态。
5. SDK 源码显示 `disconnect()` 只释放 automation 连接、不会发送 `Tool.close`；但官方未明确承诺断开 automation 后真机调试窗口持续可用。该项必须通过 opt-in 真机验收；若失败，本轮不能宣称功能完成，需回到 PLAN 设计受控后台 session 与显式 stop 命令。
6. `scripts/gen_preview.sh` 增加 action 分支，必须用既有 preview/auto-preview 测试防回归。
7. 锁定的官方最新包 `miniprogram-automator@0.12.1` 自 2023-11-07 后未再发布；2026-07-13 执行 `npm audit --omit=dev --package-lock-only --json` 报告其旧版 Jimp/QR 依赖树存在 `6 moderate / 1 high / 3 critical` 共 10 项传递依赖 advisory，且 npm 没有给出保持 `0.12.1` 的兼容修复。当前链路使用 `remote(true)`，不处理二维码输入，并以 `--ignore-scripts` 安装，降低但不能消除风险；本轮不采用未经真机兼容验证的跨主版本 override，不宣称依赖安全风险已清零。

### 12.2 回滚

1. 移除默认命令、bot 常量/识别/文案和 `remote-debug` action。
2. 删除 `vibego_cli/data/wx-remote-debug/` 及打包配置条目。
3. 删除实际 Vibego runtime root 下的 `wx-remote-debug/`（默认 `~/.config/vibego/runtime/wx-remote-debug/`）。
4. 通过 Master 命令管理或受控数据清理删除已 seed 的 `wx-remote-debug` 记录；仅从 defaults 删除不会自动删除已有 DB 行。
5. 恢复 README/AGENTS 与相关测试。
