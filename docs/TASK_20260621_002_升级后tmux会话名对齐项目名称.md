# TASK_20260621_002 升级后 tmux 会话名对齐项目名称

## 1. 背景与目标

用户反馈项目列表显示名已改短，但 tmux 会话仍是 `vibe-hyphafawnstudiobot`、`vibe-vibegobot` 这类长名称。确认后的目标口径是：**不迁移当前运行中的 tmux 会话，只保证升级到新版本后，Master 重启并再次启动项目时，tmux 会话名按项目名称生成**。

目标示例：

| 项目显示名 | 旧 project_slug | 新 project_slug | 新 tmux 会话 |
| --- | --- | --- | --- |
| FawnStudio | `hyphafawnstudiobot` | `fawnstudio` | `vibe-fawnstudio` |
| Vibego | `vibegobot` | `vibego` | `vibe-vibego` |
| WMS | `cckgwmsappcore` | `wms` | `vibe-wms` |

## 2. 现状与根因

- 项目列表展示使用 `ProjectConfig.display_name`，优先读取 `legacy_name/name`。
- worker 启动使用 `project_slug` 作为 `PROJECT_NAME`，`scripts/models/common.sh::tmux_session_for` 再生成 `vibe-${project_slug}`。
- 前一轮只改了 DB/JSON 的 `legacy_name/name`，并明确不影响 `project_slug`、日志目录和 tmux session，因此 tmux 会话仍使用旧长 slug。

## 3. 受影响目录与边界

| 范围 | 是否影响 | 说明 |
| --- | --- | --- |
| `project_repository.py` | 是 | 启动修复时将已有项目 `project_slug` 对齐展示名 slug，并记录旧 slug 到新 slug 的迁移映射。 |
| `master.py` | 是 | 新项目默认 slug 改用项目显示名；`StateStore` 根据迁移映射继承旧状态。 |
| `tests/test_master_project_management.py` | 是 | 覆盖新项目默认 slug、已有项目升级修复、冲突 fail-closed。 |
| `tests/test_worker_health_boot_id.py` | 是 | 覆盖状态 key 从旧 slug 迁移到新 slug。 |
| `docs/` | 是 | 记录本次设计、测试矩阵、风险与回滚。 |
| 当前运行 tmux 会话 | 否 | 本次不主动停止、不改名、不重启当前 worker。 |
| 历史日志目录 | 否 | 不移动旧日志目录；升级后再次启动写入新 slug 目录。 |
| SQLite schema | 否 | 复用既有 `projects.project_slug` 与 `projects.legacy_name`，不新增字段。 |

## 4. 契约变更

- `project_slug` 新默认来源：优先 `_sanitize_slug(legacy_name/name)`，无展示名时才回退 `bot_name`。
- Master 升级启动时，`ProjectRepository._repair_records()` 会自动把旧长 slug 修正为展示名 slug，并同步导出 `projects.json`。
- 修复过程中如果两个项目展示名归一到同一个 slug，直接抛出异常 fail-closed，避免运行目录和 tmux 会话互相覆盖。
- `ProjectRepository.slug_migrations` 暴露本次启动的旧 slug 到新 slug 映射，`StateStore` 用该映射迁移旧 `master_state.json` key，保留 `model/status/chat_id/actual_username`。
- 当前已运行 worker 不被迁移；正确性从升级后 Master 重启并再次启动项目开始生效。

## 7. 风险与回滚

- 风险：若已有两个项目展示名归一后相同，升级启动会失败。这是刻意 fail-closed，避免两个项目共用同一 tmux/log 目标。
- 风险：旧日志目录不会自动改名，历史日志需要按旧 slug 查；新启动日志按新 slug 写入。
- 风险：当前正在运行的旧 tmux 会话不迁移；升级前仍可能看到旧会话名，升级并重启 Master、再次启动项目后才生效。
- 回滚：回退 `project_repository.py`、`master.py` 与对应测试；若已启动新版本并写出新 slug，可从升级前备份或手工将 DB/JSON/state 中 slug 改回旧值。
