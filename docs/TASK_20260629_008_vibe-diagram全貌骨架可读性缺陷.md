# TASK_20260629_008 vibe-diagram 全貌骨架可读性缺陷

## 1. 用户反馈

用户基于截图反馈：这种故障排查图想要看懂全貌、来龙去脉仍然非常吃力。

截图主题：`故障排查：WARN 级别仍打印 ERROR 日志`。

## 2. 现象 -> 影响 -> 根因 -> 修法 -> 验证

### 2.1 现象

截图已经满足部分既有规则：有标题、有结论、有根因、有证据、有修法、有箭头、有泳道式分区。

但实际阅读仍困难：

1. 首屏没有一条“5 秒能读懂”的完整因果骨架。
2. 顶部先给大段根因解释，读者还没建立全貌就被迫读细节。
3. 中间主图区分为“业务异常链 / 日志输出链”，但跨列箭头和纵向箭头混用，读者要自己拼顺序。
4. 证据、规则、判断、结果、修法的视觉权重接近，主谓宾不突出。
5. “为什么 WARN 会变 ERROR”没有被压缩成一条明确句式：
   `业务想 WARN -> 没有业务异常 handler -> 命中兜底 -> 兜底按 ERROR 记录 -> 补 handler`。

### 2.2 影响

- 用户需要逐块阅读文字，才能重新脑补整条链路。
- 图看起来比纯 Markdown 更好看，但“读懂速度”没有显著提升。
- 对故障排查图来说，这会削弱最关键的价值：快速建立来龙去脉和修复闭环。

### 2.3 根因

当前 `vibe-diagram` 已有规则主要约束：

- 不许卡片堆叠。
- 要有图形语法。
- 关键细节要外显。
- 故障图要有故事线、证据阶梯、假设裁决、行动闭环。

但缺少更明确的“全貌优先门禁”：

1. 没要求首屏必须先出现一个低文字量的全貌骨架。
2. 没要求故障排查图先回答四个问题：发生了什么、为什么、改哪里、怎么验证。
3. 没有“5 秒读图测试”：遮住节点正文后，读者仍应能通过节点标题、编号、箭头标签读出主因果链。
4. 没限制顶部根因说明的体量，导致根因大段文字抢在全貌骨架之前。

证据锚点：

- `vibego_cli/data/skills/vibe-diagram/SKILL.md`（锚点：`## 图形语法硬约束`、`## 关键细节外显规则`、`## 故障排查图规则`、
  `## 输出前自检`）。
- 截图：`/var/folders/2m/dswmfzbd7wx_39zxs4kvd3fc0000gn/T/codex-clipboard-169e12de-0f50-44b6-8780-b73fcec10cf7.png`。

### 2.4 修法建议

推荐在 `vibe-diagram` 增加一组全局规则和故障图专项规则：

1. 新增 `## 全貌优先与 5 秒读图规则`。
2. 规定所有非页面设计图，首屏必须先有“全貌骨架”，再展开证据和细节。
3. 全貌骨架必须是 5-7 个短节点，按因果/时序/状态/层级表达，节点标题能串成一句话。
4. 故障排查图首屏必须先回答：
    - 发生了什么？
    - 为什么会发生？
    - 改哪里？
    - 怎么验证？
5. 顶部结论区禁止放长段根因说明；长根因进入证据/旁注/详情，但必须由骨架节点锚定。
6. 输出前自检新增：遮住节点正文，只看节点标题、编号、箭头标签，是否仍能读出主链路；不能则重画。

### 2.5 验证建议

- 新增测试：`test_vibe_diagram_requires_overview_spine_before_details`。
- 新增测试：`test_vibe_diagram_fault_diagram_must_answer_four_questions_first`。

  `WARN 意图 -> 缺 handler -> 兜底 ERROR -> 补 handler -> 验证 WARN`。

## 3. 推荐实施方案

推荐方案：**加全局“全貌优先 / 5 秒读图”门禁 + 故障排查专项四问骨架**。

不建议只调整颜色、间距、箭头或换一种视觉风格；这些只能改善观感，不能解决“用户要自己拼来龙去脉”的根因。

## 6. 第 2 轮纠偏：问题不是缺摘要，而是缺层级递进图形语法

### 6.1 用户纠偏

用户明确指出：上一轮把问题理解成“缺少先看全貌的骨架”仍然不对。真正问题是：

- 图不够直观。
- 完全没有层级递进关系。
- 本质仍然是文字平铺。
- 时序图和流程图之所以好懂，是因为它们有非常清晰的逻辑推进。

### 6.2 修正后的根因

之前的规则虽然写了“不要卡片堆叠”“要有图形语法”，但仍给模型留下逃逸空间：

1. 把节点做得漂亮，不等于有图形语法。
2. 把文字放进流程节点，不等于有流程推进。
3. 两列或多区块并排，不等于有层级递进。
4. 箭头存在，不等于箭头承担了逻辑语义。
5. 首屏摘要骨架也可能继续变成“短文字平铺”。

真正要补的是：**每种图型必须先选定并严格执行对应的逻辑语法，图的可读性来自语法，而不是来自节点文案。**

### 6.3 新的规则方向

推荐废弃“全貌骨架”作为核心修法，改为新增 `## 图形语法执行门禁`：

1. 生成前必须先声明本图采用哪一种语法：流程图、时序图、状态机、分层拓扑、ER-lite、因果链、决策树、页面线框等。
2. 不同语法必须满足不同的结构要求：
    - 流程图：开始/结束、活动、判断、分支、回流、异常路径。
    - 时序图：参与者列、时间从上到下、调用/返回/异常/异步分行。
    - 状态机：状态节点、进入/退出条件、线上动作、终态。
    - 分层拓扑：层级、边界、层间短箭头、依赖方向。
    - 因果链：症状、触发、机制、结果、修法、验证闭环。
3. 禁止“伪流程图”：只是把文字节点用箭头串起来，但节点之间没有状态变化、动作结果、判断条件或因果关系。
4. 对故障排查图，如果重点是“为什么从 WARN 变 ERROR”，应优先用因果链或代码时序图，而不是两列文字区块。
5. 输出前自检：把节点正文遮住，只看图形符号、位置、箭头标签、泳道/时间轴/状态转换线，是否仍能读出逻辑；如果不能，必须重画。

### 6.4 对截图的正确重画方向

这张 WARN/ERROR 图更适合两种图型之一：

- **代码时序图**：业务代码抛出 `DpsException(WARN)` → Spring handler 查找 → 未命中业务 handler → 命中 `Exception.class` →
  `logBusinessException(..., ERROR)` → Logback 输出 ERROR。
- **因果链图**：业务期望 WARN → handler 缺口 → 兜底机制 → ERROR 覆盖 → 输出 ERROR → 补 handler → 验证 WARN。

二者都比“两列文字块 + 箭头”更直观，因为它们的逻辑推进来自时间或因果，而不是读者自己读文案拼出来。
