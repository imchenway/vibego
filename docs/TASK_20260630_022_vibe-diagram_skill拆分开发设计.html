从“一个长 skill 全量常驻”改为“薄内核常驻 + 图型 reference 按需读取”，保留 HTML-first 与所有制图红线，但降低互相干扰。

设计结论：按 B 方案实施，但用 TDD 分阶段落地

核心不是删除规则，而是把规则放回正确层级：AGENTS 管触发，vibe-diagram 薄内核管路由与共性门禁，各图型 reference 管专属形态。

本轮边界

这是开发设计，不改源码。确认后再进入实现：先写失败测试，再拆文件、改打包、同步 AGENTS 并跑回归。

目标架构：四层职责，避免规则互相污染

LAYER 01AGENTS 触发层

决定何时使用 HTML / vibe-diagram。

文件AGENTS-template.md

保留 HTML-first、Skill routing、交付信封与全局工作契约。

以后优化某一种图，不需要反复修改 AGENTS。

LAYER 02vibe-diagram 薄内核

进入制图后的总调度。

保留交付、路由、共性红线

新增reference 索引表

把每个图型映射到 references/*.md，并写成硬门禁：选中图型后必须读取。

限制不展开每类图细节

核心目标 ≤ 300 行；页面设计稿不再背着故障排查长规则。

LAYER 03图型 reference

一类图一个文件。

架构/业务系统架构 + 业务架构

系统架构保留 007 宏观拓扑基线；业务架构保留领域地图、对象关系与规则热区。

流程/时序/状态流程图、代码时序、状态数据

每类图明确“应该长成什么样”、节点/箭头/布局规则和失败样式。

故障/迭代/页面现状链路与候选设计

故障与迭代先画当前链路；页面设计稿保留 3-5 候选与 Web/移动排版规则。

LAYER 04同步 / 打包 / 测试

保证拆完后真的可用。

同步AGENTS 只注入薄内核

当前同步器注入 SKILL.md 全文；只要 SKILL.md 变薄，即可避免全量常驻。

打包发布包包含 references

pyproject.toml 补 data/skills/*/references/*.md；MANIFEST.in 现有递归规则作为兜底。

测试从关键词堆转为结构断言

内核测共性；reference 测专属；同步测不全量注入；包测引用文件存在。
