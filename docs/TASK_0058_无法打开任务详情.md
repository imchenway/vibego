# /TASK_0058 无法打开任务详情（PLAN）

## 1. 背景（现状）

用户在「任务列表」点击某条任务进入详情时，出现提示：

> ⚠️ 任务详情显示失败，可能包含特殊字符。任务ID: TASK_0019 请联系管理员检查任务内容。

导致用户无法查看详情，也无法在详情页执行后续操作（如归档/删除）。

## 2. 证据（可验证）

从本机日志可定位到“任务详情发送失败”的高相关原因：

- `~/.config/vibego/logs/vibe.log`
  - 2026-01-31 17:44:53：`Telegram server says - Bad Request: message is too long`
  - 2026-01-31 17:45:34：`Telegram server says - Bad Request: message is too long`
  - 2026-01-31 17:48:32：`Telegram server says - Bad Request: message is too long`

说明“打不开详情”并不一定是“特殊字符”，更高概率是 **详情内容过长触发 Telegram 4096 字符限制**。

参考官方文档：
- Telegram Bot API `sendMessage`（文本长度限制）：https://core.telegram.org/bots/api#sendmessage
- Telegram Bot API Formatting options（Markdown/HTML 转义规则）：https://core.telegram.org/bots/api#formatting-options

## 3. 根因假设（vibe 结论）

### 假设 A（高置信度）
任务详情内容（标题/描述/备注/附件/历史等）拼接后超出 Telegram `sendMessage` 限制，返回 `message is too long`，导致消息发送失败。

### 假设 B（中高置信度）
任务详情存在 Markdown 格式化解析失败（转义不足/实体不合法），触发 Telegram `can't parse entities` 等错误，导致发送失败。

## 4. 已确认决策（来自用户）

1) 当任务详情打不开时：**提供“删除”触发方式**（只要能点击查询任务详情的人就可见）。✅  
2) 删除语义：**2A 归档/隐藏（可恢复）** ✅

> 备注：现有实现中 `tasks/service.py::delete_task()` 即通过 `archived=True` 实现逻辑删除（归档）。

## 5. 验收标准（AC）

1) 从任务列表点击任意任务，任务详情 **不因超长/格式问题而无法打开**（必要时自动降级为附件/分页/纯文本等）。  
2) 当任务详情仍无法展示时，失败提示中提供可行动入口：至少包含 **删除（归档）**。  
3) “删除（归档）”成功后，该任务默认不再出现在「未归档」任务列表中；如系统支持展示归档任务，应可恢复。  

## 6. design 阶段待确认项（下一步）

为解决“message is too long”导致的详情不可用，需要确认详情超长时的展示策略：

- A🌟 详情超长 → 发送“摘要 + 附件（全文）”，摘要消息挂载操作按钮（刷新/归档/返回等）
- B 详情超长 → 自动分页拆分为多条消息（仍可在消息内阅读全文）
- C 详情超长 → 截断展示 + 仅提供导出全文
- D 仅在失败时提供删除，不处理超长问题（不推荐）

## 7. design 结论（已确认，准备进入 develop）

> 用户指令：“待决策项全部按模型推荐”

### 7.1 决策落地

1) 详情超长策略：**A（摘要 + 全文附件）** ✅  
2) 删除（二次确认）：**需要二次确认** ✅  
3) 删除按钮范围：**失败提示 + 正常详情页都可见** ✅  
4) 删除语义：**归档/隐藏（可恢复）** ✅  

### 7.2 现状定位（技术视角，可验证）

- 任务列表入口：`task:detail:<TASK_ID>` → `bot.py` `on_task_detail_callback(...)`
- 详情渲染：`bot.py::_render_task_detail()` → `bot.py::_format_task_detail()`
- 当前失败提示：`⚠️ 任务详情显示失败...`（当 `edit_text/answer` 均失败时兜底）
- 高相关日志：`Bad Request: message is too long`（Telegram 4096 限制触发）

### 7.3 设计方案（推荐并选中）

#### 7.3.1 统一“展示任务详情”的出口（解决超长/格式失败）

新增一个仅用于“任务详情场景”的发送函数（伪代码级别）：

- 输入：`message/callback`、`task_id`、`detail_text`、`markup`
- 行为：
  1) 预判 `detail_text` 是否超出 Telegram 限制（按当前 parse_mode 的 payload 长度计算）
  2) 未超长：沿用原逻辑（优先 `edit_text`，失败则 `answer`）
  3) 超长：改为
     - 在原消息上（可编辑时）显示 **短摘要**（挂载详情操作按钮）
     - 另发一条 **全文附件（txt/md）** 供下载查看
  4) 若发生未知发送失败：显示错误原因（尽量从 TelegramBadRequest 提取），并挂载兜底按钮（至少含删除）

> 说明：现有 `reply_large_text(...)` 已能把“超长文本”降级为附件，但其摘要消息默认不挂键盘；任务详情需要保留“返回/删除”等按钮，因此将实现独立的“详情专用”发送策略（摘要挂键盘）。

#### 7.3.2 删除（归档）触发方式（任何可查看详情的人可见）

1) **正常详情页**：在 `_build_task_actions(task)` 中新增按钮  
   - 文案：`🗑️ 删除（归档）`
   - 回调：`task:delete_prompt:<TASK_ID>`
2) **详情展示失败兜底提示**：提示消息同样挂载  
   - `🗑️ 删除（归档）`（同回调）
   - `🔄 重试`（或 `刷新`）
   - `⬅️ 返回任务列表`

#### 7.3.3 删除二次确认（避免误触）

- 点击 `task:delete_prompt:<TASK_ID>` 后，只更新键盘为确认态（不改文本，避免超长时 edit_text 再失败）：
  - `✅ 确认删除（可恢复）` → `task:delete_confirm:<TASK_ID>`
  - `❌ 取消` → `task:refresh:<TASK_ID>`（恢复原详情键盘）

#### 7.3.4 删除实现语义（可恢复）

- 后端调用：`TaskService.delete_task(task_id, actor=...)`
  - 实现等价于 `update_task(... archived=True)`，属于逻辑删除（归档）
- 归档后反馈：
  - 提示“已归档（可恢复）”，并展示 `/{TASK_ID}` 与返回列表入口
  - 说明恢复方式：`/task_delete TASK_0001 restore=yes`

### 7.4 影响范围（预估）

- `bot.py`
  - 任务详情入口：`on_task_detail_callback` / `on_refresh_callback` / `_reply_task_detail_message`
  - 任务详情按钮：`_build_task_actions`
  - 新增回调 handler：`task:delete_prompt:*`、`task:delete_confirm:*`
  - 统一发送：新增“详情发送（含超长降级）”辅助函数
- `tasks/service.py`：无需改（已有 `delete_task`）
- 测试
  - 更新 `tests/test_task_detail_actions_layout.py`（现在详情会出现删除按钮）
  - 新增用例：超长详情不会触发“显示失败”，且会发送附件/摘要
  - 新增用例：删除二次确认与归档结果（mock TaskService）

### 7.5 用例设计 Checklist（像测试一样）

1) 超长详情（描述超长）
   - 点击详情：不出现“显示失败”
   - 收到摘要（含按钮）+ 全文附件
2) 正常详情（不超长）
   - 点击详情：仍可编辑原消息展示
   - 删除按钮可见
3) 删除流程
   - 点击删除：键盘变为“确认/取消”
   - 点击确认：任务 archived=True，并提示“可恢复”
   - 点击取消：回到详情操作键盘
4) 失败兜底
   - 模拟发送失败：提示中仍可点击删除（归档）

## 8. 全局长文本策略（已确认）

> 用户指令：“待决策项全部按模型推荐”

### 8.1 目标

用户在 Telegram 任意阶段一次性粘贴 **10000+ 字符**，Telegram 客户端可能会拆成多条消息；vibego 需要在“瞬时”窗口内把多条分片聚合为一次逻辑输入，并继续原本流程（落库 / 推送模型 / FSM 交互）。

### 8.2 决策（按推荐落地）

当“需要写入任务/缺陷描述等数据库字段”的输入超出系统限制（当前 `DESCRIPTION_MAX_LENGTH=3000`）时：

- ✅ 写入占位文本（提示“内容过长，详见附件”）
- ✅ 全文落盘为本地 txt，并作为任务附件绑定
- ✅ 流程继续推进（不要求用户手动拆分/重发）

### 8.3 设计要点（后续 develop 实现）

1) **前置聚合（覆盖任意场景）**
   - 位置：Telegram 消息入站最前置（建议 middleware/统一入口）
   - 行为：短窗口内聚合拆分文本，输出“合并后的逻辑文本”再交给原有 handler/FSM

2) **推送模型的统一降级**
   - 位置：进入 `_dispatch_prompt_to_model(...)` 前
   - 行为：超长 prompt → 落盘 txt → 复用“附件列表 → 文件路径”的提示词推送模型（等价于“发文件给模型读”）

3) **落库字段的统一降级**
   - 位置：任务描述/缺陷描述等写库前的长度校验点
   - 行为：超长文本 → 绑定为任务附件 + DB 写入占位 → 继续流程

### 8.4 参考资料（官方/可验证）

- Telegram Bot API `sendDocument`（文件消息）：https://core.telegram.org/bots/api#senddocument
- Telegram Bot API Formatting options（Markdown/HTML 解析规则）：https://core.telegram.org/bots/api#formatting-options

## 9. develop 实施结果（已完成 ✅）

> 完成日期：2026-01-31

### 9.1 变更摘要

1) ✅ **任务详情超长不再打不开**
   - 详情渲染文本在发送前做长度预判；超过 Telegram 单条限制时自动降级：
     - 发送「摘要（挂操作按钮）」+「全文附件（md）」。

2) ✅ **失败兜底提示可操作**
   - 不再误导为“特殊字符”，改为提示“内容过长/格式异常/网络波动”等可能原因；
   - 失败提示内联键盘提供：**删除（归档）/重试/返回任务列表**。

3) ✅ **删除（归档）二次确认**
   - 任务详情页新增 `🗑️ 删除（归档）`；
   - 点击后仅更新键盘为确认态（避免 edit_text 失败）；
   - 确认后调用 `TaskService.delete_task(...)`（归档语义，可恢复），并提示恢复命令：`/task_delete TASK_0001 restore=yes`。

4) ✅ **全局长文本（10000+）处理**
   - 新增全局中间件：入站最前置聚合 Telegram 拆分的长文本分片，延迟窗口后注入“合成消息”，继续原有 FSM/命令/普通对话逻辑；
   - 推送模型：当待推送文本 > 4096（通常意味着聚合后的合成消息）时，自动转为“本地附件 + 附件提示词”推送；
   - 写库字段（任务描述/缺陷描述等）：超过 `DESCRIPTION_MAX_LENGTH=3000` 时自动落盘为附件，并写入占位文本继续流程。
