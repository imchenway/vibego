# /TASK_0094 消息底部成功按钮按点击项消失

## 1. 背景

用户要求：

1. 模型答案消息底部的以下按钮，在**后续逻辑真正执行成功后**才消失：
   - `✅ 全部按推荐`
   - `🧩 部分按推荐（需补充）`
   - `⬆️ 提交分支`
   - `⬆️ 提交并行分支`
   - `🧪 任务状态更新为测试中`
   - `↩️ 回复 /TASK_xxxx`
2. 点哪个按钮，只移除哪个按钮
3. 任务详情按钮不在本次范围内

补充确认：

- **不是**点击瞬间就移除
- **必须等该按钮后续业务逻辑完整成功后**才移除
- 若业务失败 / fail-closed / 参数无效，则按钮保持不变

## 2. 关键证据

- 模型答案消息底部按钮由同一个 inline keyboard 构造：`bot.py`（锚点：`_build_model_quick_reply_keyboard`）
- `全部按推荐` 成功链路：`bot.py`（锚点：`on_model_quick_reply_all`）
- `部分按推荐` 两阶段链路：`bot.py`（锚点：`on_model_quick_reply_partial`, `on_model_quick_reply_partial_supplement`）
- `更新为测试中` 链路：`bot.py`（锚点：`on_model_task_to_test`）
- 原生 / 并行提交链路：`bot.py`（锚点：`on_session_commit_callback`, `on_parallel_commit_callback`）
- 并行回复模式链路：`bot.py`（锚点：`on_parallel_reply_callback`）
- Plan Yes 成功后会清空整组按钮，可作为“成功后再回写按钮”的参考：`bot.py`（锚点：`on_plan_confirm_callback`）

## 6. 最小实现

### 6.1 通用按钮裁剪能力

新增：

- `_remove_inline_button_from_markup(...)`
  - 从 inline keyboard 中精确移除一个 callback 对应按钮
- `_try_remove_clicked_inline_button(...)`
  - 仅在业务成功后尝试 `edit_reply_markup`
  - 若按钮编辑失败，仅记录日志，不回滚业务成功

### 6.2 成功后移除按钮的回调

在以下成功分支末尾追加“移除当前点击按钮”：

- `on_model_quick_reply_all`
- `on_model_quick_reply_partial_supplement`
- `on_model_task_to_test`
- `on_parallel_reply_callback`
- `on_session_commit_callback`
- `on_parallel_commit_callback`

### 6.3 部分按推荐两阶段处理

- `on_model_quick_reply_partial`
  - 新增 state 字段：`partial_callback_data`
  - 首次点击仅进入补充态，不移除按钮
- `on_model_quick_reply_partial_supplement`
  - 只有最终推送成功后，才按 `partial_callback_data` 回写原消息按钮

## 8. 用户可见结果

现在模型答案消息底部按钮会遵循以下规则：

1. 点哪个按钮，成功后只移除哪个按钮
2. 其它未点按钮保留
3. 任务详情按钮保留
4. `部分按推荐（需补充）` 只有在最终提交成功后才会消失
5. 任何失败 / 会话失效 / 参数无效场景，按钮都不会提前消失

## 9. 风险与回滚

### 风险

- 若业务成功但 Telegram `edit_reply_markup(...)` 失败，用户会看到“动作成功但按钮仍在”；本次按需求优先保证业务成功，不回滚业务结果。

### 回滚点

- `bot.py`
- `tests/test_model_quick_reply.py`
- `tests/test_parallel_flow.py`
