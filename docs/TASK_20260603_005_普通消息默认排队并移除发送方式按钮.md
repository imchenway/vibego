# TASK_20260603_005 普通消息默认排队并移除发送方式按钮（DEVELOP）

## 1. 需求与目标

用户已确认范围：仅影响普通 Telegram 直聊。

目标：移除 Worker 底部 `✉️ 立即/排队` 切换按钮；普通消息不再读写发送方式状态文件，改为按模型能力自动选择默认发送方式：

1. Codex/Copilot：默认 `queued`。
2. 不支持 queued 的模型：自动回退 `immediate`。
3. 普通消息仍沿用 `TASK_20260603_004` 的规则：发送正文前先确保 PLAN；若默认 queued，也必须先发送必要的 PLAN 预命令，再排队正文。
4. 任务推送、批量推送、PlanConfirm、并行任务等非普通直聊入口保持原显式发送方式选择。

## 2. 当前实现证据

| 事实 | 证据 |
| --- | --- |
| Worker 主键盘由 `_build_worker_main_keyboard` 生成 | `bot.py`（锚点：`_build_worker_main_keyboard`） |
| 普通直聊最终由 `_handle_prompt_dispatch` 统一投递模型 | `bot.py`（锚点：`_handle_prompt_dispatch`） |
| 普通直聊发送方式通过 `_get_worker_direct_send_mode()` 传给 `_dispatch_prompt_to_model` | `bot.py`（锚点：`_get_worker_direct_send_mode`、`_handle_prompt_dispatch`） |
| 只有 Codex/Copilot 支持 queued | `bot.py`（锚点：`_supports_queued_send_mode`） |
| 任务推送仍有独立发送方式选择 | `bot.py`（锚点：`_build_push_send_mode_keyboard`、`on_task_push_model_send_mode`、`on_task_batch_push_mode_choice`） |

## 3. 方案选择

### 方案 A：移除底部按钮，普通直聊默认排队（本次采用）

- 主键盘恢复一行两列：`📟 命令管理` + `🧭 PLAN: ...` / `🧭 MODE: ...`。
- `_get_worker_direct_send_mode()` 改为运行时派生，不再读写 `*_direct_send_mode.json`。
- 旧 `✉️` 按钮 handler 保留为兼容入口，只提示按钮已移除并刷新主键盘。

优点：符合用户最新确认；减少 Telegram 小屏误触和状态认知成本；避免旧状态文件导致普通消息不是默认排队。  
缺点：普通直聊不再提供“手动立即发送”快捷切换；如果临时需要 immediate，只能通过任务推送流程显式选择或后续另设命令入口。

### 方案 B：保留隐藏命令 `/send_mode`，移除底部按钮

优点：保留高级手动切换。  
缺点：仍会保留运行期状态与额外契约，和“默认排队、去掉切换按钮”的目标不完全一致。

### 方案 C：仅隐藏按钮但继续读取旧状态文件

优点：代码改动最小。  
缺点：用户看不到按钮却仍受旧状态影响，容易出现“为什么不是默认排队”的隐性问题，因此不采用。

## 5. 契约变更

1. Worker 底部常驻键盘必须是一行两列：`📟 命令管理`、`🧭 PLAN: ...`；Copilot 为 `🧭 MODE: ...`。
2. 新主键盘不得展示 `✉️ 立即` 或 `✉️ 排队`。
3. `_get_worker_direct_send_mode()` 不再读取或写入运行期状态文件：
   - `_supports_queued_send_mode() == True` 时返回 `queued`。
   - 否则返回 `immediate`。
4. 普通直聊继续传 `intended_mode=PLAN`、`force_plan_switch_for_queued=True`、`probe_plan_mode_before_switch=True`、`confirm_delivery=True`。
5. 旧 `✉️ 立即/排队` 键盘残留仍被 handler 接收，但只回复“按钮已移除；普通消息默认排队/立即”，并刷新主键盘，不切换状态、不写文件。
6. 任务推送里的 `立即发送` / `排队发送` 选择不变；本任务不改变任务推送契约。

## 8. 风险与回滚

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| 普通直聊不再能从底部临时切 immediate | 用户少一个快捷开关 | 范围按用户确认收敛；任务推送仍保留显式选择。 |
| 旧客户端残留按钮 | 用户点击旧按钮可能以为仍能切换 | handler 保留并给出明确“按钮已移除”提示，同时刷新新主键盘。 |
| 旧状态文件存在 | 可能误导后续排障 | `_get_worker_direct_send_mode()` 不再读取旧文件；文档明确“忽略不删除”。 |
| 不支持 queued 的模型 | 若默认 queued 会失败 | `_supports_queued_send_mode()` fail-closed 回退 immediate。 |

回滚方式：恢复 `bot.py::_build_worker_main_keyboard` 第三列、恢复 `_get_worker_direct_send_mode()` 的状态文件读写与 `_set_worker_direct_send_mode()`，恢复旧按钮切换逻辑，并回滚本任务测试断言；无数据库回滚。
