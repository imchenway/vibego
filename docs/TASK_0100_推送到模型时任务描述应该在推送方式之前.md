# /TASK_0100 推送到模型时，任务描述应该在推送方式之前

## 1. 背景

用户要求：

> 推送到模型时，任务描述应该在推送方式之前，也就最后一步再选择立即发送还是排队发送。

目标是调整单任务“推送到模型”交互顺序，使 **Codex** 下的发送方式选择从“补充描述前”改为“补充描述后”。

## 2. 关键证据

- 全局/仓库规约：
  - `$HOME/.config/vibego/AGENTS.md`
  - `AGENTS.md`
- 当前推送链路：
  - `bot.py`（锚点：`on_task_push_model`、`on_task_push_model_choice`、`on_task_push_model_send_mode`、`on_task_push_model_supplement`）
- 既有发送方式设计：
  - `docs/TASK_0091_在codex中可以按tab发送排队的消息我希望在vibego也可以发送这种消息.md`
- 既有 PLAN/YOLO 设计：
  - `docs/TASK_0046_推送到模型增加PLAN_YOLO模式选择.md`

## 3. vibe 结论（用户视角）

### 3.1 现状

当前 Codex 下的单任务推送顺序为：

```text
推送到模型
-> 选择 PLAN / YOLO
-> 选择 立即发送 / 排队发送
-> 输入补充任务描述
-> 执行推送
```

### 3.2 期望

用户希望改为：

```text
推送到模型
-> 选择 PLAN / YOLO
-> 输入补充任务描述
-> 最后选择 立即发送 / 排队发送
-> 执行推送
```

### 3.3 不变项

1. 非 Codex 模型仍不展示“发送方式”步骤
2. 现有 CLI / 并行 CLI 选择方式不变
3. PLAN / YOLO 选择能力不变
4. 最终推送 payload 与 session 绑定语义不变

### 4.2 实施方案

#### Codex

```text
处理方式
-> PLAN / YOLO
-> 补充任务描述
-> 发送方式（立即/排队）
-> 执行推送
```

#### 非 Codex

```text
处理方式
-> PLAN / YOLO
-> 补充任务描述
-> 执行推送
```

### 4.3 最小实现策略

1. `on_task_push_model_choice(...)`
   - 不再进入 `waiting_send_mode`
   - 统一先进入 `waiting_supplement`
2. `on_task_push_model_supplement(...)`
   - Codex 且尚未选 `send_mode` 时：
     - 先把补充文本/附件暂存到 FSM
     - 再进入 `waiting_send_mode`
   - 非 Codex 或已带 `send_mode` 时：
     - 直接执行最终推送
3. `on_task_push_model_send_mode(...)`
   - 改为读取前一步暂存的补充描述/附件
   - 选择发送方式后直接执行最终推送
4. 新增 `_execute_task_push_from_state_data(...)`
   - 统一收口当前 CLI / 选中现有并行会话 / 新建并行 CLI 三条最终执行路径

## 7. 最小实现

### 7.1 顺序调整

- `bot.py:on_task_push_model_choice`
  - 选择 `PLAN/YOLO` 后统一进入 `waiting_supplement`

### 7.2 暂存补充描述

- `bot.py:on_task_push_model_supplement`
  - Codex 且未选发送方式时：
    - 保存 `pending_push_supplement`
    - 保存 `pending_push_attachments`
    - 再提示选择发送方式

### 7.3 最终执行收口

- `bot.py:_execute_task_push_from_state_data`
  - 统一处理：
    - 当前 CLI
    - 已选择的现有并行会话
    - 新建并行 CLI
  - 统一发送：
    - `_push_task_to_model(...)`
    - `_send_model_push_preview(...)`
    - `_send_session_ack(...)`
