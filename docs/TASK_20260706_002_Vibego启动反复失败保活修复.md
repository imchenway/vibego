# TASK_20260706_002：Vibego 启动反复失败保活修复

## 1. 目标

用户反馈：Vibego 项目在 Telegram Master 面板里“反复启动，反复失败”。截图显示 `worker 进程 95660 已退出`，最近日志停在 `Telegram 连通性检查失败：在 30.0 秒内未能与 Telegram 成功握手`。

本轮目标不是伪装 Telegram 网络已恢复，而是修复程序行为：启动期 Telegram 瞬时不可达时，worker 不应直接退出并让用户重复点击启动；应保持进程存活、进入长轮询/后台重试，Master 项目列表保持 `starting`，避免回到可重复启动状态。

## 3. 根因裁决

已确认根因：worker 启动流程把 Telegram 初始 `get_me()` 连通性检查当成硬门禁。只要代理/Telegram 在启动窗口内抖动 30 秒，worker 就退出；Master 看到 pid 死亡后把状态回落为 `stopped`，于是用户继续点击启动，触发同一失败循环。

外部 Telegram/代理是否可用仍是环境变量；本轮不把网络恢复伪装成代码修复。代码修复的边界是：网络暂不可用时不再退出和重复启动，而是保活等待恢复。

## 4. 修复方案

1. `bot.py`：初始 Telegram 连通性检查失败时改为 warning，继续初始化本地服务并进入带重试的长轮询；同时启动后台任务周期性重试 `ensure_telegram_connectivity()`。
2. `bot.py`：初始握手失败时跳过启动期命令、聊天菜单和自动 `/start` 广播，避免网络故障下继续卡住；后台重试成功后再补做这些同步动作。
3. `bot.py`：`_ensure_bot_commands()`、`_ensure_worker_menu_button()` 对启动期网络异常 fail-soft；`_start_worker_polling_with_retry()` 捕获 aiogram `start_polling()` 启动前 `bot.me()` 网络异常并持续重试，不再升级成进程退出。
4. `master.py`：健康检查超时但 worker pid 与 tmux 主会话仍存活时，不再抛硬失败；状态保持 `starting` 并返回，让项目列表展示 `⏳ 启动中/停止`，阻止重复启动。
5. `master.py`：pid 存活但 tmux 缺失仍保持原 degraded/fail-closed 语义，避免把模型会话缺失伪装成可用。

## 7. 风险与回滚

- 风险：Telegram 长时间不可用时，worker 会保持 `starting`，不会立刻变成 `running`；这是刻意选择，避免半启动状态被重复启动覆盖。
- 风险：后台重试成功前，Worker 不会主动推送主键盘；恢复后会补做命令、菜单和 `/start` 等价广播。aiogram 长轮询启动前握手也可能重试一段时间，期间 Master 保持 `starting`。
- 回滚：恢复 `bot.py::main()` 为初始握手失败后 `SystemExit(1)`，移除 `_start_worker_polling_with_retry()` 的长轮询网络重试，并恢复 `master.py::run_worker()` 对健康检查超时直接抛错；回滚后会重新暴露本次反复启动失败问题。
